# 全页缓存实现隐患分析

## 发现的潜在隐患

### 1. ⚠️ 输出缓冲区嵌套问题（严重）

**位置**：`Core.php:603` 和 `Template.php:391`

**问题**：
- `Core::route()` 在第 603 行调用 `ob_start()`
- 模板系统的 `Template::ob_file()` 也调用 `ob_start()`
- 这会导致嵌套的输出缓冲区

**影响**：
- 如果控制器使用模板系统，会导致输出缓冲区嵌套
- `ob_get_clean()` 可能只获取到内层缓冲区的内容
- 可能导致 HTML 内容不完整

**建议修复**：
```php
// 检查是否已有输出缓冲区
if (ob_get_level() > 0) {
    // 已有输出缓冲区，不重复开启
} else {
    ob_start();
}
```

### 2. ⚠️ 响应头捕获时机问题（中等）

**位置**：`Core.php:616`

**问题**：
- 响应头在 `ob_get_clean()` 之后捕获
- 但 `header_xss()` 在方法开始时就调用了（第 529 行）
- 如果控制器或事件处理器在 `route_after` 事件中修改了响应头，这些修改会被捕获
- 但如果响应头在 `ob_get_clean()` 之前就已经发送，可能无法捕获

**影响**：
- 某些响应头可能无法正确恢复
- 响应头顺序可能与原始请求不一致

**建议修复**：
- 在 `ob_start()` 之前捕获初始响应头
- 在 `ob_get_clean()` 之后捕获最终响应头
- 合并两者，确保完整性

### 3. ⚠️ 异常处理不完善（中等）

**位置**：`Core.php:603-611`

**问题**：
- 如果控制器执行过程中抛出异常，`ob_get_clean()` 可能不会被调用
- 输出缓冲区可能残留，导致后续请求出现问题

**影响**：
- 异常情况下输出缓冲区未清理
- 可能导致后续请求输出混乱

**建议修复**：
```php
ob_start();
try {
    $result = call_user_func([$dispatch, $method], ...);
    // ... 其他代码
} catch (\Exception $e) {
    ob_end_clean(); // 清理输出缓冲区
    throw $e;
}
$output = ob_get_clean();
```

### 4. ⚠️ 缓存键一致性检查缺失（中等）

**位置**：`Core.php:649` 和 `CheckFullPageCache.php:88`

**问题**：
- 保存和读取时都调用 `buildUnifiedRequestCacheKey()`，但如果在两次调用之间 `WELINE_FULL_REQUEST_URI` 被修改，可能导致键不一致
- 没有验证缓存键是否匹配

**影响**：
- 可能导致缓存无法命中
- 或者读取到错误的缓存数据

**建议修复**：
- 在保存和读取时记录缓存键，用于调试
- 添加缓存键验证逻辑

### 5. ⚠️ 响应头恢复失败处理（轻微）

**位置**：`Core.php:543-552` 和 `CheckFullPageCache.php:97-106`

**问题**：
- 如果响应头已经发送，`header_remove()` 和 `header()` 会失败
- 没有错误处理

**影响**：
- 响应头可能无法正确恢复
- 但不会导致致命错误

**建议修复**：
```php
if (!headers_sent()) {
    header_remove($headerName);
    header($header, true);
}
```

### 6. ⚠️ 缓存数据完整性检查（轻微）

**位置**：`Core.php:540` 和 `CheckFullPageCache.php:94`

**问题**：
- 只检查 `fpc_html` 是否存在，不检查其他键
- 如果缓存数据不完整（缺少某些键），可能导致问题

**影响**：
- 如果缓存数据损坏，可能导致运行时错误

**建议修复**：
- 添加缓存数据完整性验证
- 如果数据不完整，清除缓存并重新生成

### 7. ⚠️ 并发写入问题（轻微）

**位置**：`Core.php:650`

**问题**：
- 多个请求同时写入同一个缓存键时，可能导致数据竞争
- 没有锁机制

**影响**：
- 高并发情况下可能出现缓存数据不一致
- 但影响较小，因为主要是读取操作

**建议修复**：
- 如果需要，可以添加文件锁或 Redis 锁

### 8. ⚠️ 事件触发时机问题（轻微）

**位置**：`CheckFullPageCache.php:70-74`

**问题**：
- 在 `App::run_before` 事件中，如果 `WELINE_IS_BACKEND` 未设置，会提前返回
- 这导致在 `App::run_before` 时基本不会命中缓存
- 主要依赖 `App::url_parsed_after` 事件

**影响**：
- 性能优化效果可能不如预期
- 但功能正常

**建议修复**：
- 可以考虑移除 `App::run_before` 事件的注册，只保留 `App::url_parsed_after`

## 优先级修复建议

1. **高优先级**：修复输出缓冲区嵌套问题（隐患 1）
2. **中优先级**：添加异常处理（隐患 3）
3. **中优先级**：优化响应头捕获时机（隐患 2）
4. **低优先级**：其他优化（隐患 4-8）
