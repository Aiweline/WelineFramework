## Goal

为 `GuoLaiRen/PageBuilder` 设计一个完整的 AI 建站智能体，从站点需求采集开始，贯穿站点信息生成、域名推荐与购买、域名状态跟踪、虚拟主题生成、页面内容生成、可视化微调和最终建站发布。

## Positioning

这是一个“计划中的单项方案”，不是 Codex 的统筹笔记本。

Codex 在这里的职责是：

- 把原始想法整理成可执行计划。
- 将计划拆成阶段、模块、依赖和风险。
- 后续围绕该计划拆分具体任务，分别记录到 `dev/ai/codex/tasks/`。

## User Journey

1. 用户描述建站需求。
2. 智能体生成并允许微调站点基础信息。
3. 智能体进入域名推荐阶段，支持反复刷新推荐、调整提示词、检查可用性。
4. 用户选定购买平台账户并完成域名购买。
5. SSE 控制台持续显示购买进度、DNS/可建站状态和阶段变化。
6. 域名等待就绪期间，智能体继续生成虚拟主题和页面内容。
7. 用户对主题组件和页面内容进行可视化微调。
8. 域名可用后，系统执行建站、绑定站点、挂载页面并提供预览入口。

## Major Phases

### Phase 1: Site Brief Generation

- 输入用户需求描述。
- 生成站点标题、主题方向、SEO 描述、核心页面方向和基础内容。
- 支持用户继续微调提示词与生成结果。

### Phase 2: Domain Recommendation And Purchase

- 选择可购买域名的平台账户。
- 根据站点信息生成域名候选。
- 支持重新推荐、手动输入域名、域名可用性检测。
- 支持直接购买域名，并把购买过程反馈到 SSE 控制台。

### Phase 3: Domain Readiness Tracking

- 域名购买成功后，自动拉取到本地系统。
- 后台继续处理 DNS 解析等准备动作。
- 每 10 秒检查一次域名当前状态。
- SSE 顶部持续展示域名所处阶段，例如已购买、待解析、可建站。

### Phase 4: Virtual Theme Generation

- 基于站点信息和 AI 生成主题整体布局 JSON。
- 主题结构应由 `header`、`content`、`footer` 等组件组合而成。
- 当前文件式模板能力需要扩展到“文件模板 + 虚拟模板”混合支持。
- 虚拟主题需要支持落库，而不仅仅是文件生成。

### Phase 5: Page Type Selection And Content Generation

- 域名对应站点需要选择页面类型。
- 首页为必选项，其他页面类型可勾选，默认可全选。
- 根据页面类型生成对应主题内容和页面内容。
- 支持基于提示词继续微调。
- 支持在组件层面直接修改配置，并结合 AI 辅助重新生成。

### Phase 6: Visual Editing And Final Publish

- 用户持续调整虚拟主题中的任意组件，直到可视化结果满足预期。
- 域名就绪后执行正式建站流程。
- 建站逻辑需复用页面构建器内部已有建站链路：
  `www` 与 `root` 域名绑定、站点创建、页面创建、页面挂载。
- 完成后显示站点地址和预览效果，且预览效果需与可视化编辑一致。

## Required Capabilities

### SSE Control Panel

- 实时显示当前阶段、子任务进度、异常和下一步建议。
- 支持不中断流程地修改提示词与生成参数。
- 支持域名购买与域名状态轮询结果的持续反馈。

### Persistent Scope Memory

- 智能体应通过 scope 标签或等价机制记住当前站点上下文。
- 同一站点会话中的提示词、生成结果、选中的域名和主题微调结果应可持续引用。

### Virtual Theme Support

- PageBuilder 需要兼容虚拟模板主题。
- 主题不能只存在文件系统，需要支持数据库落库与运行时加载。
- 文件模板与虚拟模板的统一抽象需要单独设计。

### Multilingual Support

- 多语言是明确要求，不应作为后补功能。
- 站点信息、页面内容、SEO 文案和主题配置都要考虑语言维度。

## Key Design Questions

1. 虚拟主题的数据模型如何设计，才能同时兼容文件模板和数据库模板？
2. SSE 控制台的状态机如何定义，才能覆盖域名购买、域名等待和建站发布全过程？
3. 域名购买平台账户、购买流程和结果回传如何安全接入？
4. 页面类型与主题组件的映射如何抽象，才能支持按页面类型批量生成内容？
5. 多语言内容是单次生成多语言版本，还是先生成主语言再增量翻译？

## Suggested Delivery Order

1. 先梳理状态机与整体流程图。
2. 再设计虚拟主题的数据结构与加载机制。
3. 接着定义 SSE 控制台的数据协议和事件流。
4. 然后串起站点信息生成、域名推荐、域名状态轮询。
5. 最后接入页面生成、可视化微调和正式建站。

## Suggested Task Breakdown

- 任务 1：整理 AI 建站智能体状态机。
- 任务 2：设计虚拟主题的数据模型和存储结构。
- 任务 3：设计文件模板与虚拟模板的统一加载抽象。
- 任务 4：设计 SSE 控制台事件协议与 UI 状态展示。
- 任务 5：梳理域名推荐、购买和状态轮询链路。
- 任务 6：梳理页面类型选择与页面内容生成链路。
- 任务 7：梳理建站发布复用的现有 PageBuilder 能力。
- 任务 8：设计多语言支撑方案。

## Risks

- 域名购买和 DNS 生效具有明显异步性，流程设计不好会让用户感到卡住。
- 虚拟主题如果直接套用文件模板思路，后续会很难扩展。
- 页面生成、主题生成、建站发布如果没有统一状态机，SSE 面板会难以维护。
- 多语言如果后置处理，后面会出现主题、页面和 SEO 配置无法统一的问题。
