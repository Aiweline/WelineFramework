# AI建站工作台契约流程与强约束

## 统一契约结构

每个 AI 生成契约都必须使用统一外壳，避免各阶段自由发挥字段结构。

必备字段如下：

- `contract_meta`：契约 id、版本、阶段、状态、生成 agent/adapter、创建时间。
- `permission_matrix`：当前阶段允许创建、修补、只读或禁止访问的字段范围。
- `frozen_fields`：确认后下游阶段不得修改的字段。
- `mutable_fields`：后续 Repair 阶段允许修补的字段。
- `source_contracts`：当前契约依赖的上游契约 id 列表。
- `qa_gates`：校验项的 `pending/pass/fail/warn` 状态和错误信息。

缺少 `contract_meta`、下游契约缺少 `source_contracts`、尝试修改 `frozen_fields` 都是 hard error。错误要暴露为明确的接口错误和队列日志，不能静默降级成空数据。

## Stage1：建站蓝图契约

Stage1 由 `AiSitePlanQueue` 执行，输入是用户需求、选中的技能快照、站点上下文和必要的历史兼容视图。Stage1 输出以下契约：

- `Site Brief`
- `Design Manifest`
- `Page Contract`
- `Block Plan`

Stage1 允许创建站点结构和设计方向。它可以决定页面有哪些、每个页面解决什么问题、页面内有哪些区块、整体品牌和视觉风格是什么。Stage1 输出后必须进入待确认状态，用户确认后冻结关键字段。

Stage1 确认后应冻结页面列表、页面目标、品牌方向、设计系统、区块顺序、核心内容策略和主要 SEO 方向。

## Stage2：视觉与任务契约

Stage2 由 `AiSiteTaskPlanQueue` 执行，只能消费已确认的 Stage1 契约。Stage2 输出以下契约：

- `Block Visual Contract`
- `Block Task Contract`

Stage2 负责把 Stage1 的页面和区块规划细化成可执行构建任务。它可以补充区块视觉细节、资源需求、文生图提示、组件选择、交互细节、验收标准和构建顺序。

Stage2 不允许修改 Stage1 已冻结字段。具体禁止修改页面列表、品牌方向、设计系统、区块顺序和站点核心定位。如果确实发现 Stage1 有问题，应生成修复建议或阻断提示，而不是在 Stage2 中偷偷改。

## Build：构建契约消费

Build 队列只消费已确认的 Stage2 契约，输出 `Build Render Data` 和 theme manifest。Build 只能把任务契约转成可渲染页面、主题配置、资源引用和预览结果。

Build 不允许修改 Stage1 或 Stage2 的规划契约。如果构建中发现任务不可执行，应写入构建错误和 QA 状态，由 Repair 走受控补丁流程。

## QA 与 Repair

QA 读取构建输出和所有相关契约，检查契约完整性、冻结字段一致性、页面结构、视觉质量、文案质量、SEO 基础、资源可用性和构建结果。

Repair 只能 patch `permission_matrix` 允许修改的字段。Repair Patch 必须声明来源契约、修改字段、修改原因和 QA 结果。Repair 不允许绕过用户确认修改冻结字段。

## 队列与前端边界

前端负责需求编辑、技能选择、阶段确认、状态展示和错误展示。前端不运行 AI、不组装隐式历史、不在浏览器内做构建。

队列 worker 必须能在没有浏览器页面、没有浏览器内存状态的情况下独立完成任务。开始队列前，后端必须已经写入选中技能、技能快照、契约上下文和必要元信息。

SSE 只展示队列状态、日志、阶段输出和错误，不承担轮询替代执行器，也不维护独立业务状态。
