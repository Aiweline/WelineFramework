# 计划完成后代码审计子代理（Plan Code Auditor）

本技能是一个**专门的审计子代理**，在计划完成后执行深度代码审计，对比计划与实际实现的一致性，检测遗漏和缺陷，并提供修复方案。

## 〇、何时触发

| 场景 | 触发词 |
|------|--------|
| 计划完成后审计 | 审计一下, 检查完成情况, 计划完成审计, 审计代码 |
| 对比计划与代码 | 计划和代码一致吗, 实现完了吗, 对比一下 |
| 缺陷深度检查 | 缺陷检查, 找漏洞, 遗漏检查, 深度审视 |
| 批量模块审计 | 批量审计, 全面检查, 所有模块审计 |

---

## 一、审计流程概述

```
┌──────────────────────────────────────────────────────────────────┐
│                    Plan Code Auditor 工作流程                     │
├──────────────────────────────────────────────────────────────────┤
│ 1. 加载计划文件（plan.md + task.md）                               │
│    ↓                                                              │
│ 2. 解析计划结构（功能点、任务项、验收标准）                           │
│    ↓                                                              │
│ 3. 扫描相关代码文件（根据计划涉及的模块）                            │
│    ↓                                                              │
│ 4. 逐条对比：计划功能 ↔ 实际代码                                   │
│    ↓                                                              │
│ 5. 生成审计报告（完成✓ / 部分实现⚠ / 未实现✗ / 有缺陷❌）            │
│    ↓                                                              │
│ 6. 提供修复方案（针对每个缺陷/遗漏）                                │
└──────────────────────────────────────────────────────────────────┘
```

## 二、审计步骤详解

### 2.1 步骤一：加载计划文件

**必须加载的文件：**

| 文件类型 | 路径 | 说明 |
|----------|------|------|
| 总计划 | `.cursor/plans/<名称>.plan.md` | 整体目标、范围、里程碑 |
| 模块计划 | `app/code/<Vendor>/<Module>/doc/开发/plan.md` | 该模块具体变更 |
| 任务清单 | `app/code/<Vendor>/<Module>/doc/开发/task.md` | 任务细分与完成标记 |

**加载顺序：**
1. 先读取总计划，获取涉及的模块列表
2. 遍历读取每个模块的 `plan.md` 和 `task.md`
3. 建立「功能点 → 预期实现」映射

### 2.2 步骤二：解析计划结构

从计划文件中提取以下信息：

```markdown
## 解析目标

1. **功能点列表**
   - 每个功能的描述
   - 预期行为
   - 验收标准（如有）

2. **任务项列表**
   - task.md 中的所有 `[ ]`、`[-]`、`[x]` 任务
   - 每个任务的完成状态

3. **涉及文件列表**
   - 计划中提到的文件路径
   - 预期修改的文件

4. **技术要求**
   - 数据库变更
   - API 接口
   - 前端组件
   - 配置项
```

### 2.3 步骤三：扫描相关代码文件

**扫描范围：**
- 计划中明确提到的文件
- 模块的核心文件（Controller、Service、Model）
- 相关配置文件（menu.xml、event.xml、register.php）
- 前端文件（.phtml、.css、.js）

**扫描方法：**
```bash
# 获取模块文件列表
ls app/code/<Vendor>/<Module>/

# 检查关键文件是否存在
# 读取文件内容进行分析
```

### 2.4 步骤四：逐条对比

**对比维度：**

| 维度 | 检查内容 | 判定标准 |
|------|----------|----------|
| **存在性** | 功能/文件是否存在 | 文件存在 + 相关代码存在 |
| **完整性** | 功能是否完整实现 | 所有预期行为都有对应代码 |
| **正确性** | 实现逻辑是否正确 | 逻辑符合计划描述 |
| **规范性** | 是否符合框架规范 | 符合相关技能要求 |

**对比方法：**

```markdown
## 功能点对比示例

### 功能：用户登录
- 计划要求：支持用户名/邮箱登录
- 代码检查：
  - [ ] LoginController 存在
  - [ ] 支持用户名登录逻辑
  - [ ] 支持邮箱登录逻辑
  - [ ] 登录失败错误处理
  - [ ] 登录成功跳转
- 状态：✓ 完成 / ⚠ 部分实现 / ✗ 未实现
```

## 三、审计检查清单

### 3.1 功能完整性检查

- [ ] 所有计划功能点都有对应代码实现
- [ ] 所有 task.md 中 [x] 的任务确实已实现
- [ ] 边界情况已处理
- [ ] 错误处理已实现
- [ ] 用户输入验证已实现

### 3.2 代码规范检查

- [ ] PHP 8.4 严格类型（`declare(strict_types=1)`）
- [ ] 所有方法有类型声明
- [ ] Null 安全处理
- [ ] 使用框架提供的方法（不自造）
- [ ] 符合 SOLID 原则

### 3.3 数据库操作检查

- [ ] 所有查询链式调用 `fetch()`
- [ ] 无业务层手写 SQL
- [ ] 字段升级使用 `hasField()` 检查
- [ ] 分页使用框架分页方法

### 3.4 前端规范检查

- [ ] CSS 使用主题变量（无硬编码颜色）
- [ ] JS 使用 IIFE 闭包（无全局污染）
- [ ] 无 `alert()`/`confirm()`/`prompt()`
- [ ] 使用 `BackendToast`/`BackendConfirm`
- [ ] 国际化使用 `__()`/`<lang>`

### 3.5 配置与依赖检查

- [ ] `register.php` 依赖声明完整
- [ ] 版本号已递增（若有 schema 变更）
- [ ] `menu.xml` ACL 配置正确
- [ ] `event.xml` 事件注册正确

## 五、使用示例

### 示例 1：单模块审计

```
用户：审计一下 Weline_Backend 模块的计划完成情况

AI 执行：
1. 读取 app/code/Weline/Backend/doc/开发/plan.md
2. 读取 app/code/Weline/Backend/doc/开发/task.md
3. 扫描模块代码
4. 逐条对比
5. 输出审计报告
```

### 示例 2：整体计划审计

```
用户：审计一下消息订阅通知系统的整体实现情况

AI 执行：
1. 读取 .cursor/plans/message-subscription.plan.md
2. 识别涉及的所有模块
3. 逐模块执行审计
4. 汇总生成整体报告
```

### 示例 3：缺陷修复后复审

```
用户：我修复了审计报告中的问题，再审一遍

AI 执行：
1. 加载上次审计报告（或重新审计）
2. 针对上次发现的缺陷/遗漏重新检查
3. 更新审计状态
4. 输出复审报告
```
