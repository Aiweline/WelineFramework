# 创建计划（create-plan）

创建或修改计划文件时**必须先指定目标路径**，并遵守以下协议。

---

## 〇、计划状态标记（强制 - 必须最先执行）

**所有计划文件必须在文件头标注当前状态**：

| 状态 | 标记 | Emoji | 说明 |
|------|------|-------|------|
| 未开始 | `status: pending` | 🔴 | 计划已创建，未开始实施 |
| 进行中 | `status: in_progress` | 🟡 | 正在开发实施 |
| 测试中 | `status: testing` | 🔵 | 开发完成，测试阶段 |
| 已完成 | `status: completed` | 🟢 | 测试通过，计划完成 |
| 已取消 | `status: cancelled` | ⚫ | 计划取消不再执行 |

### 总计划状态头（模板）

```markdown
# 计划名称

**状态**：🟡 进行中（status: in_progress）  
**当前阶段**：第X阶段 - XXX（YYYY-MM-DD）  
**完成度**：60%（6/10 阶段完成）  
**最后更新**：YYYY-MM-DD
```

### 模块计划状态头（模板）

```markdown
# Weline_Xxx 模块计划

**状态**：🔵 测试中（status: testing）  
**完成度**：95%  
**最后更新**：YYYY-MM-DD

> 总计划：[xxx.plan.md](../../../.cursor/plans/xxx.plan.md)
```

### 子计划进度汇总表（总计划必须包含）

总计划的「涉及模块及子计划」表**必须包含状态和完成度列**：

```markdown
| 模块 | 子计划 | 任务进度 | 状态 | 完成度 |
|------|--------|---------|------|--------|
| Weline_Backend | [plan.md](...) | [task.md](...) | 🔵 测试中 | 95% |
| Weline_Framework | [plan.md](...) | [task.md](...) | 🟢 已完成 | 100% |
| Weline_Theme | [plan.md](...) | [task.md](...) | 🟡 进行中 | 70% |
```

## 〇.5、决策自审机制（强制 - 每个决策前必须执行）

**制定计划的每一个决策前，必须进行系统性自我审视**。不要急于给出方案，先问自己以下问题：

### 自审问题清单（必问）

| # | 问题 | 目的 | 必须回答 |
|---|------|------|----------|
| 1 | **为什么这么做？** | 明确动机和理由 | ✅ |
| 2 | **这么做有什么收益？** | 量化/描述预期收益 | ✅ |
| 3 | **有没有什么缺陷/风险？** | 识别潜在问题 | ✅ |
| 4 | **有没有其他地方在用？** | 评估影响范围 | ✅ |
| 5 | **牵扯到哪些地方？** | 列出关联模块/文件/功能 | ✅ |
| 6 | **要怎么修复/处理？** | 给出具体应对方案 | ✅ |
| 7 | **有没有安全隐患？** | 安全性评估 | ✅ |
| 8 | **有没有可用的技能？** | 命中技能直接参考 | ✅ |
| 9 | **有没有更好的替代方案？** | 方案对比 | 可选 |

### 自审流程

```
决策点 → 自审7问 → 分析结论 → 制定任务（含修复方案）
```

### 自审记录格式（计划中必须体现）

每个重要决策在计划中必须包含自审记录：

```markdown
### 决策：[决策名称]

#### 自审分析

| 问题 | 分析 |
|------|------|
| **为什么这么做？** | [原因说明] |
| **收益** | [具体收益] |
| **缺陷/风险** | [已识别的问题] |
| **影响范围** | [哪些地方在用/被影响] |
| **关联模块** | [牵扯到的模块/文件列表] |
| **应对方案** | [如何修复/处理] |
| **安全隐患** | [安全评估结果] |
| **命中技能** | [可用技能列表，见下方] |

#### 命中技能（必须参考）

| 技能 | 路径 | 用途 |
|------|------|------|
| [技能名] | `dev/ai/skills/[技能名]/SKILL.md` | [该决策为何需要此技能] |

#### 结论

[最终决定及理由]
```

### 技能命中检查（必做）

制定计划时，**必须**检查是否有可用技能并在计划中列出参考：

#### 常见场景→技能映射表

| 计划涉及场景 | 必须命中的技能 |
|--------------|----------------|
| 创建模块/控制器/模型 | `module-development`, `code-generation-standards` |
| 数据库/字段/表结构 | `database-model-standards`, `module-development` |
| 前端/CSS/JS/组件 | `theme-development` ⭐ |
| 事件/观察者 | `create-event` |
| 扩展点/Extends | `create-extends`, `implement-extends` |
| Hook/视图扩展 | `create-hook` |
| 用户提示/通知/弹窗 | `friendly-notifications` ⭐ |
| 模块间通信/查询 | `unified-query-provider` |
| 国际化/多语言 | `i18n-internationalization` |
| 菜单/权限/ACL | `acl-permission-system` |
| Session/登录 | `session-development` |
| 缓存 | `cache-usage` |
| 进程/Worker/WLS | `process-management`, `weline-server` |
| 测试 | `php-unit-testing`, `frontend-automation-testing` |
| 错误修复 | `error-learning`, `error-tracking` |
| 安全/验证 | `quality-assurance` |
| 文档 | `documentation-standards` |

#### 技能检查流程

1. **分析计划涉及的场景**（如：创建模块、写前端、数据库操作等）
2. **查阅上方映射表**，找出对应技能
3. **在计划中列出命中技能**，格式如下：

```markdown
## 命中技能（开发时必须参考）

| 技能 | 路径 | 说明 |
|------|------|------|
| module-development | `dev/ai/skills/module-development/SKILL.md` | 模块创建规范 |
| database-model-standards | `dev/ai/skills/database-model-standards/SKILL.md` | 数据库字段升级 |
| theme-development | `dev/ai/skills/theme-development/SKILL.md` | 前端样式规范 |
```

4. **开发时先阅读技能再写代码**

#### 技能命中的好处

- ✅ 避免踩坑（技能记录了常见错误和最佳实践）
- ✅ 代码规范一致（遵循框架约定）
- ✅ 减少返工（一次做对）
- ✅ 知识复用（站在巨人肩膀上）

### 影响范围检查方法

在回答"有没有其他地方在用"时，**必须**执行：

1. **代码搜索**：Grep 搜索类名、方法名、常量名
2. **引用分析**：检查 `use` 语句、`extends`、`implements`
3. **配置检查**：检查 `env.php`、`event.xml`、`menu.xml` 等配置文件
4. **依赖检查**：检查 `register.php` 的 `depend` 声明

```bash
# 示例：检查某个类被哪些地方引用
rg "ClassName" --type php
rg "use.*ClassName" --type php
```

### 安全隐患检查清单

| 检查项 | 说明 |
|--------|------|
| 权限控制 | 是否需要 ACL 权限？ |
| 数据验证 | 用户输入是否验证？ |
| SQL 注入 | 是否使用参数化查询？ |
| XSS 攻击 | 输出是否转义？ |
| CSRF 保护 | 表单是否有 CSRF token？ |
| 敏感数据 | 是否涉及密码、密钥等？ |
| 日志泄露 | 日志是否会记录敏感信息？ |
| 状态泄漏 | WLS 下是否有跨请求状态问题？ |

### 任务生成要求

自审完成后，生成的任务必须包含：

1. **主功能任务**：实现核心功能
2. **修复任务**：针对自审发现的缺陷/风险
3. **关联更新任务**：更新受影响的地方
4. **安全加固任务**：修复安全隐患
5. **测试任务**：验证功能和修复效果

### 任务模板（含修复方案）

```markdown
## 命中技能（开发时必须参考）

| 技能 | 路径 | 说明 |
|------|------|------|

## 任务列表

### 核心功能
- [ ] 任务1：[功能描述]
- [ ] 任务2：[功能描述]

### 缺陷修复（自审发现）
- [ ] 修复1：[缺陷描述] → [修复方案]
- [ ] 修复2：[缺陷描述] → [修复方案]

### 关联更新（影响范围）
- [ ] 更新1：[模块/文件] - [更新内容]
- [ ] 更新2：[模块/文件] - [更新内容]
```

### 禁止事项

- ❌ **禁止**不经自审直接给出方案
- ❌ **禁止**只写功能任务不写修复任务
- ❌ **禁止**忽略影响范围分析
- ❌ **禁止**跳过安全隐患检查
- ❌ **禁止**自审分析不写入计划文档
- ❌ **禁止**不检查可用技能就制定计划
- ❌ **禁止**命中技能但不列入计划参考
- ❌ **禁止**开发时不阅读命中技能就写代码

## 一、总计划与子计划关系

- **大计划包含小计划**：大计划写总览，通过**引入子计划**的方式组织内容。
- **大计划**：在 `.cursor/plans/` 下写**总计划概述**（目标、范围、里程碑、依赖关系等），并**列出并链接各模块子计划**（子计划路径见下表）。
- **小计划**：只写**本模块涉及到的变化**，不写其他模块细节；可被总计划引用。

**编写方式**：采用「总计划概述 + 引入子计划」。总计划中用链接或路径引用各 `模块/doc/开发/plan.md`，子计划内只关注该模块的变更与任务。

---

## 二、计划文件必须写到哪里

| 类型 | 路径 | 说明 |
|------|------|------|
| **大/整体计划** | `.cursor/plans/<名称>.plan.md` | 总计划概述 + 状态汇总表 + 引入各模块子计划 |
| **模块/小计划** | `app/code/<Vendor>/<Module>/doc/开发/plan.md` | 只写该模块变化 + 状态标记，进度在此更新 |

**硬性要求**：创建计划时先确定是「大计划」还是「模块计划」，再写入对应路径，禁止写错位置。

示例：
- 整体方案、多模块协作 → `.cursor/plans/xxx.plan.md`（内含概述 + 状态 + 子计划列表）
- Weline_Cdn 模块实现 → `app/code/Weline/Cdn/doc/开发/plan.md`（仅 Cdn 相关变更与状态）

## 三、任务与进度文件

| 用途 | 路径 | 说明 |
|------|------|------|
| **进度更新** | 对应模块 `doc/开发/plan.md` | 在该 plan 文件内随时更新任务进度 |
| **模块任务细分与完成标记** | `app/code/<Vendor>/<Module>/doc/开发/task.md` | 该模块任务列表、子项、状态标记 |

**协议**：
- **新增任务**：修改对应 `plan.md`（大计划改 `.cursor/plans/` 下；模块改该模块 `doc/开发/plan.md`）。
- **进度更新**：在对应模块的 `doc/开发/plan.md` 内更新（不单独用 doc/计划.md）。
- **完成任务**：在对应模块的 `doc/开发/task.md` 中标记完成（如 `[x]`、`已完成`、`done`）。
- **模块任务细分**：写在**该模块**的 `doc/开发/task.md` 下，并认真标记每项进度（未开始/进行中/已完成）。

### 实时更新 task.md（强制）

**开发过程中必须实时更新 task.md**：

| 时机 | 必须执行的操作 |
|------|----------------|
| 开始某项任务 | 将对应任务标记为 `[-]` 进行中 |
| 完成某项任务 | 将对应任务标记为 `[x]` 已完成 |
| 发现新任务 | 添加新任务条目到 task.md |
| 任务有变化 | 更新任务描述或拆分子任务 |
| 遇到阻塞 | 在任务后添加备注说明阻塞原因 |

**禁止**：开发完所有功能后再统一更新 task.md。必须边开发边更新！

## 四、进度标记约定

在 `task.md` 中建议使用统一标记，便于检索与命中：

- `[ ]` 或 `待办` / `pending`：未开始  
- `[-]` 或 `进行中` / `in_progress`：进行中  
- `[x]` 或 `已完成` / `done` / `completed`：已完成  

可在文件头部注明「最后更新」时间或版本。

## 六.1、计划完成后代码审视（强制）

**计划完成后，必须重新按照计划审视一遍所有涉及的代码！**

### 审视检查清单

| 检查项 | 说明 | 检查方法 |
|--------|------|----------|
| **功能完整性** | 计划中所有功能点是否都已实现？ | 逐条对照计划/task.md |
| **遗漏检查** | 是否有遗漏的边界情况、错误处理？ | 审视每个方法的异常路径 |
| **逻辑正确性** | 业务逻辑是否正确？条件判断是否完备？ | 人工审视 + 单元测试 |
| **PHP 8.4 语法** | 是否符合 PHP 8.4 严格模式？ | 参考 `php84-performance` 技能 |
| **类型安全** | 参数类型、返回类型是否正确声明？ | 检查所有方法签名 |
| **Null 安全** | 是否有潜在的 null 传递给非 nullable 参数？ | 检查所有变量使用 |

### PHP 8.4 语法检查重点

```php
// ❌ 错误：可能传 null 给非 nullable 参数
function process(string $data) { ... }
process($maybeNull);  // 如果 $maybeNull 可能为 null，报错！

// ✅ 正确：显式处理 null
process($maybeNull ?? '');
// 或声明参数为 nullable
function process(?string $data) { ... }
```

### 必须检查的语法问题

1. **严格类型声明**：文件头是否有 `declare(strict_types=1);`
2. **参数类型**：所有方法参数是否有类型声明
3. **返回类型**：所有方法是否有返回类型声明
4. **Null 安全**：
   - 禁止传 null 给非 nullable 参数
   - 使用 `??` 或 `?:` 处理可能的 null
   - 使用 `?type` 声明 nullable 类型
5. **数组操作**：使用 `array_find()`、`array_any()`、`array_all()` 等 PHP 8.4 新函数
6. **资源类型**：使用 `get_resource_id()` 而非 `(int)$resource`

### 审视流程

1. **列出计划涉及的所有文件**
2. **逐文件审视**：
   - 读取文件内容
   - 对照计划检查功能完整性
   - 检查逻辑正确性
   - 检查 PHP 8.4 语法合规
3. **运行 Lint 检查**：使用 `ReadLints` 工具检查语法错误
4. **修复发现的问题**
5. **重新运行测试**：确保修复没有引入新问题

### 常见遗漏点

- [ ] 错误处理：try-catch 是否完备？
- [ ] 日志记录：关键操作是否有日志？
- [ ] 边界情况：空数组、null、空字符串等
- [ ] 并发安全：WLS 下是否有状态泄漏？
- [ ] 权限检查：ACL 是否配置正确？
- [ ] 国际化：用户可见文本是否使用 `__()`？

## 六.5、计划完成后文档更新（强制）

**计划完成后，必须按照计划内容更新各模块相关文档，保持文档最新！**

### 必须更新的文档类型

| 文档类型 | 路径 | 更新条件 |
|----------|------|----------|
| **模块 README** | `app/code/<Vendor>/<Module>/README.md` | 新增功能、API 变更、配置变更 |
| **技能文档** | `dev/ai/skills/<技能>/SKILL.md` | 涉及开发规范、流程、最佳实践变更 |
| **架构文档** | 模块 `doc/` 下的架构图、设计文档 | 架构变更、新增组件、流程变化 |
| **API 文档** | 模块 `doc/API/` 或 README | 新增接口、接口签名变更 |
| **配置文档** | 相关 env.php 说明、配置项说明 | 新增配置项、配置格式变更 |

### 文档更新检查流程

1. **列出计划涉及的所有模块**
2. **逐模块检查**：
   - 该模块是否有 README.md？内容是否需要更新？
   - 该模块 doc/ 下是否有需要更新的文档？
   - 新增的功能/接口是否需要补充文档？
3. **检查技能文档**：
   - 是否涉及开发规范变更？→ 更新相关 SKILL.md
   - 是否涉及新的最佳实践？→ 补充到对应技能
4. **更新或创建文档**：
   - 已有文档：更新相关章节，标注更新日期
   - 需要新文档：在模块 `doc/` 下创建

### 文档更新模板

```markdown
## 更新记录

| 日期 | 版本 | 变更内容 | 关联计划 |
|------|------|----------|----------|
| YYYY-MM-DD | x.x.x | 新增 XXX 功能 | [计划名](链接) |
```

### 禁止事项

- ❌ **禁止**计划完成后不更新文档
- ❌ **禁止**文档内容与实际代码不一致
- ❌ **禁止**只更新代码不更新相关文档中的示例
- ❌ **禁止**新增功能无任何文档说明

## 七、完成度计算方法

1. **按阶段计算**：已完成阶段数 / 总阶段数 × 100%
2. **按任务计算**：task.md 中 [x] 任务数 / 总任务数 × 100%
3. **建议**：task.md 任务细化到可执行粒度，便于准确计算

## 八、何时触发本技能

- 用户说：创建计划、写计划、新建 plan、任务拆分、更新计划、标记完成、更新进度
- 用户说：计划状态、更新状态、完成了、测试中、搞定了、做完了
- 用户说：更新文档、文档同步、保持文档最新
- 用户说：分析影响、评估风险、安全检查、自审、决策分析、为什么这么做
- 用户提到：plan.md、task.md、doc/开发/plan、.cursor/plans  
- 用户要求：把计划写到某处、按模块写计划、大计划放哪里
- 用户要求：分析一下再做、考虑周全、有没有其他影响、牵扯到哪里

触发后：
1. **先执行决策自审**：对每个重要决策进行7问自审
2. 确认计划类型与目标路径
3. 生成含修复方案的完整任务列表
4. 创建或修改对应文件，**并更新状态标记**

**计划完成时额外触发**：检查并更新各模块相关文档，确保文档与代码同步。
