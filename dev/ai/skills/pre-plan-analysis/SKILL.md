# 计划前分析（Pre-Plan Analysis）

本技能是 `create-plan` 技能的**前置 Hook**，在创建计划前执行缺陷检查和现状分析，为计划编写提供依据和建议。

---

## 〇、执行流程

```
┌──────────────────────────────────────────────────────────────────┐
│                     Pre-Plan Analysis 工作流程                    │
├──────────────────────────────────────────────────────────────────┤
│ 1. 明确分析范围（目标模块/功能领域）                                │
│    ↓                                                              │
│ 2. 收集信息（代码、配置、文档、历史计划）                           │
│    ↓                                                              │
│ 3. 缺陷检查（代码规范、架构、安全、性能）                           │
│    ↓                                                              │
│ 4. 现状评估（完成度、技术债、依赖关系）                             │
│    ↓                                                              │
│ 5. 生成建议（优先级排序、风险提示、实施路径）                        │
│    ↓                                                              │
│ 6. 输出分析报告 → 进入 create-plan 编写计划                        │
└──────────────────────────────────────────────────────────────────┘
```

---

## 一、触发时机与条件

### 1.1 必须触发的场景

| 场景 | 触发词示例 |
|------|-----------|
| 用户要求创建计划 | 创建计划, 写计划, 做个计划, 规划一下, 写个 plan |
| 用户要求分析现状 | 分析一下, 看看现状, 检查一下现有代码, 评估一下 |
| 用户要求缺陷检查 | 有什么问题, 找找缺陷, 有没有 bug, 代码检查 |
| 用户要求优化建议 | 给点建议, 优化建议, 改进建议, 可以怎么改进 |

### 1.2 可跳过的场景

| 场景 | 说明 |
|------|------|
| 紧急修复 | 用户明确说"紧急"、"马上"、"不用分析" |
| 简单任务 | 单文件修改、明确的 bug 修复、已有详细说明 |
| 已有分析 | 用户提供了完整的分析报告或需求文档 |

跳过时输出：「已跳过计划前分析（原因：XXX），直接进入计划编写。」

## 二、分析步骤详解

### 2.1 步骤一：明确分析范围

**输入收集：**

```markdown
## 需要确认的信息

1. **目标模块**：涉及哪些模块？（单模块/多模块/整体）
2. **功能领域**：属于什么类型？（新功能/优化/重构/修复）
3. **已知问题**：用户已知的问题或痛点？
4. **期望目标**：用户希望达成什么效果？
```

**范围确定后输出：**

```markdown
### 分析范围确认

- **涉及模块**：Weline_Xxx, Weline_Yyy
- **功能领域**：XXX 功能优化
- **分析深度**：快速扫描 / 中度分析 / 深度审计
```

### 2.2 步骤二：收集信息

**必须收集的信息：**

| 类型 | 来源 | 收集方法 |
|------|------|----------|
| **代码结构** | 模块目录 | `ls app/code/<Vendor>/<Module>/` |
| **配置文件** | register.php, menu.xml, event.xml | 读取关键配置 |
| **现有文档** | README.md, doc/开发/*.md | 读取已有说明 |
| **历史计划** | doc/开发/plan.md, task.md | 查看历史任务状态 |
| **依赖关系** | register.php dependencies | 分析模块依赖图 |

**可选收集：**

| 类型 | 条件 | 收集方法 |
|------|------|----------|
| **Git 历史** | 需要了解变更频率 | `git log --oneline -20 -- <path>` |
| **错误日志** | 有已知问题 | 检查 var/log/*.log |
| **性能数据** | 涉及性能优化 | 检查慢日志/监控数据 |

### 2.3 步骤三：缺陷检查

**检查维度与检查项：**

#### A. 代码规范检查

| 检查项 | 对应技能 | 检查方法 |
|--------|----------|----------|
| PHP 8.4 严格类型 | php84-performance | 抽查是否有 `declare(strict_types=1)` |
| 国际化 | i18n-internationalization | 抽查用户文案是否使用 `__()` |
| 主题变量 | theme-development | 抽查 CSS 是否硬编码颜色 |
| 框架方法 | framework-method-validation | 抽查是否使用框架不存在的方法 |
| 数据库操作 | database-model-standards | 抽查是否链式 `fetch()` |

#### B. 架构检查

| 检查项 | 检查方法 |
|--------|----------|
| 模块依赖合理性 | 分析 register.php 依赖是否过多/循环依赖 |
| 职责单一性 | 检查 Service/Controller 是否过于臃肿（>500行警告）|
| 代码重复 | 检查是否有明显的重复逻辑（可提取为公共 Service）|
| 分层清晰度 | 检查是否存在 Controller 直接操作数据库等违规 |

#### C. 安全检查

| 检查项 | 检查方法 |
|--------|----------|
| 输入验证 | 检查用户输入是否有验证/过滤 |
| SQL 注入风险 | 检查是否有拼接 SQL |
| XSS 风险 | 检查输出是否有转义 |
| 权限控制 | 检查敏感操作是否有 ACL 控制 |

#### D. 性能检查

| 检查项 | 检查方法 |
|--------|----------|
| N+1 查询 | 检查循环内是否有数据库查询 |
| 缓存使用 | 检查热点数据是否有缓存 |
| 批量操作 | 检查是否逐条处理可批量处理的数据 |

### 2.4 步骤四：现状评估

**评估维度：**

```markdown
## 现状评估

### 1. 完成度评估

| 功能模块 | 状态 | 完成度 | 说明 |
|----------|------|--------|------|
| 模块 A | 🟢 可用 | 90% | 基础功能完成，缺少 XXX |
| 模块 B | 🟡 部分可用 | 60% | 核心功能 OK，边缘功能缺失 |
| 模块 C | 🔴 不可用 | 20% | 仅有骨架代码 |

### 2. 技术债务

| 债务类型 | 严重度 | 位置 | 说明 |
|----------|--------|------|------|
| 硬编码配置 | 中 | Config.php:45 | 应提取为 env 配置 |
| 废弃代码 | 低 | Legacy/ 目录 | 可安全删除 |
| 临时方案 | 高 | Service:123 | TODO 注释未处理 |

### 3. 依赖风险

| 依赖 | 风险等级 | 说明 |
|------|----------|------|
| Weline_Xxx | 低 | 稳定模块 |
| 第三方 API | 高 | 无降级方案 |
| 旧版组件 | 中 | 需升级 |
```

### 2.5 步骤五：生成建议

**建议分类与格式：**

```markdown
## 优化建议

### 🔴 必须处理（阻塞性问题）

| 序号 | 问题 | 位置 | 建议方案 | 预估工作量 |
|------|------|------|----------|-----------|
| 1 | 安全漏洞 | Auth.php:67 | 添加 CSRF 验证 | 小 |
| 2 | 数据丢失风险 | Save.php:120 | 添加事务 | 中 |

### 🟡 建议处理（可改进项）

| 序号 | 问题 | 位置 | 建议方案 | 预估工作量 |
|------|------|------|----------|-----------|
| 1 | 性能可优化 | List.php:45 | 添加缓存 | 中 |
| 2 | 代码可读性 | Service.php | 拆分方法 | 小 |

### 🟢 可选优化（锦上添花）

| 序号 | 建议 | 位置 | 方案 | 工作量 |
|------|------|------|------|--------|
| 1 | 增加单元测试 | 整体 | 补充测试用例 | 大 |
| 2 | 文档完善 | README | 补充使用说明 | 小 |

### ⚠️ 风险提示

1. **XXX 可能影响 YYY**：建议先测试环境验证
2. **ZZZ 依赖第三方**：建议准备降级方案
```

## 八、常见缺陷类型速查

### 代码规范类

| 缺陷 | 表现 | 对应技能 |
|------|------|----------|
| 缺少严格类型 | 无 `declare(strict_types=1)` | php84-performance |
| 硬编码文案 | 直接输出中文字符串 | i18n-internationalization |
| 硬编码颜色 | CSS 中 `#xxx`、`rgb()` | theme-development |
| 原生弹窗 | `alert()`、`confirm()` | friendly-notifications |
| 缺少 fetch | `->where()->select()` 无 fetch | database-model-standards |

### 架构类

| 缺陷 | 表现 | 建议 |
|------|------|------|
| 文件过大 | 单文件 >500 行 | 拆分职责 |
| 循环依赖 | A 依赖 B，B 依赖 A | 提取公共模块 |
| 职责混乱 | Controller 写 SQL | 分层重构 |
| 重复代码 | 多处相同逻辑 | 提取 Service |

### 安全类

| 缺陷 | 表现 | 建议 |
|------|------|------|
| SQL 注入 | 字符串拼接 SQL | 使用参数绑定 |
| XSS | 未转义输出 | 使用 `htmlspecialchars` |
| CSRF | 无 token 验证 | 添加 CSRF 中间件 |
| 权限缺失 | 无 ACL 检查 | 添加 #[Acl] 注解 |

### 性能类

| 缺陷 | 表现 | 建议 |
|------|------|------|
| N+1 查询 | 循环内查数据库 | 批量查询后映射 |
| 无缓存 | 热点数据每次查库 | 添加缓存层 |
| 无分页 | 全量加载大数据 | 使用框架分页 |
| 同步阻塞 | 耗时操作同步执行 | 异步处理 |
