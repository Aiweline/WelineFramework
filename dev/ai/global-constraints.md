# AI 总则与全局约束

本文档是 WelineFramework 仓库内 AI 开发规则的唯一总则（Single Source of Truth）。

- 根目录 `AI-ENTRY.md`、`AI-README.md`、`AGENTS.md`、`CLAUDE.md` 只作为入口索引，不再维护规则正文。
- `dev/ai/README.md`、`dev/ai/skills/_index.md`、`dev/ai/agent/README.md` 只负责路由到本总则和对应技能。
- 新增或修改跨角色、跨技能、跨工具的 AI 规则时，优先更新本文档；专项技能只写该技能独有规则。

## 1. 沟通与决策

AI 不能机械顺从输入。每次分析、设计、实现和验收前，必须同时做产品判断与工程判断。

- 先还原目标用户、业务目标、使用场景、权限边界、异常状态和验收口径。
- 用户只描述实现方式、界面或局部动作时，先补齐真实目标，再评估方案是否值得执行。
- 如果存在更简单、更稳定、更符合框架设计的方案，必须主动提出替代方案并说明取舍。
- 如果用户坚持高风险方案，可以继续执行，但必须先明确风险、影响范围和验证方式。
- 对抗性思维必须基于代码、文档、运行状态、项目约束或明确工程经验，不得主观抬杠。

## 2. 反顺从原则

AI 的目标是交付正确、稳定、可验证的结果，不是迎合每一句输入。

- 当用户说法与代码、文档、日志、运行状态或验证证据冲突时，以可验证证据为准。
- 不把“用户说了”当作正确性证明，也不把“不确定”包装成“已验证”。
- 当需求描述与当前架构冲突时，优先维护工程正确性，再解释为什么要调整方案。

## 3. 开发执行

- 面对开发任务，先判断是否可以按模块、层次、风险边界并行拆分。
- 支持并行时，优先拆成互不冲突的子任务，提高交付速度。
- 拆分前必须评估文件冲突、接口依赖和验证顺序。
- 主智能体负责最终整合、冲突检查、验证和交付结论，不能只分派不验收。

## 4. 开发前阅读顺序

1. 先读本总则。
2. 再读 `AI-ENTRY.md` 获取仓库入口索引。
3. 优先读 `dev/ai/diagrams/00-INDEX.txt` 和相关模块文档。
4. 按需读取 `dev/ai/skills/_index.md` 命中的技能。
5. 最后才大范围读源码。

### 4.1 任务工作区与进度留痕（强制）

- 默认所有仓库内任务都必须留下可恢复记录，不能只依赖聊天上下文、临时记忆或口头汇报。
- 仅完全与仓库无关、且明确不需要后续恢复的瞬时问答或简单系统查询可以不建任务工作区；只要涉及代码、文档、排查、验证、计划、规则调整或任何可能继续的工作，一律按“必须记录”处理。
- 统一使用 `dev/ai/codex/tasks/YYYY-MM-DD/YYYY-MM-DD-HHMM-short-slug/` 作为单任务工作目录；新任务优先执行 `php dev/ai/codex/scripts/init-task.php "short title" --source="user request"` 初始化。
- 开始较大分析、实现、改造或验证前，至少补齐当前任务目录内的 `task.md` 与 `plan.md`。
- 执行过程中必须持续追加 `progress.md`，至少记录：关键决策、已读资料、已改文件、阻塞点、验证动作、下一步。
- 完成、暂停、交接或中断时，必须更新 `result.md` 与 `task.md` 状态，写明已验证证据、未完成项、剩余风险和下次恢复入口。
- 跨任务方案、Epic 拆解与长期路线图写入 `dev/ai/codex/plans/*.plan.md` 或模块 `doc/开发/plan.md`；单任务状态禁止回写共享 `ACTIVE.md` 或散落在多个共享文件中。
- 用户明确给出已有任务目录时，优先在原目录续写；不要为同一任务平行创建第二套状态记录。

## 5. 全局禁止项

- 禁止直接修改 `generated/`；必须回到源代码、模板、生成规则或构建链路解决问题。
- 禁止手工编辑 `view/tpl/**`、`app/code/*/*/view/tpl/**`、`app/design/*/*/*/view/tpl/**` 等本地化/编译/生成模板产物；必须修改对应源模板、服务、Hook、Widget、Taglib、生成规则或构建链路。若生成产物被污染，只允许删除生成产物并清缓存/重载让框架重新生成，不得在 `tpl` 内修补。
- 禁止使用 `routes.xml`；路由通过控制器目录和 `php bin/w setup:upgrade --route` 同步。
- 禁止使用原生 `alert()`、`confirm()`、`prompt()`。
- 禁止前端模块脚本直接发起简单 `fetch`、`XMLHttpRequest`、`$.ajax`、axios 等请求来绕过主题请求链路。
- 禁止硬编码用户可见文案。
- 禁止为了让任务“看起来完成”而绕过架构、权限边界、i18n、配置系统或验证流程。
- 禁止硬编码、一次性兼容补丁、隐藏开关、假数据或跳过真实调用来伪造功能通过。
- 禁止跨模块直接引用对方内部 PHP 类或内部实现；跨模块协作必须使用框架认可的契约、Hook、事件、Query Provider、配置、接口、队列或扩展点。
- 禁止用类名白名单、特例分支、路径判断、字符串修补等方式只修一个症状而不解决根因。
- 禁止在 `Setup/Upgrade.php` 做字段 CRUD；字段结构使用 Model `#[Col]` / `#[Index]` 后执行 `setup:upgrade`。
- 禁止在 `<w:*>` 或自定义 Taglib 属性中写 `<?= ?>` 或 `<?php ?>`。
- 禁止在 `.phtml` 中写 `declare(strict_types=1)`。
- 禁止在 WLS 运行时敏感路径使用 `sleep()`、`usleep()`、`die()`、`exit()`。
- WLS 持久运行模式下，长循环、长连接、批量 I/O、大 JSON 序列化、模板渲染和其他可能超过健康探测窗口的同步热路径，必须通过 `SchedulerSystem::yield()` / `yieldDelay()` 或框架封装的协作式出让 API 分段让出控制权；不得写成一次性长同步阻塞。
- WLS Master、Dispatcher、Worker、Redirect、Session、Memory 等长驻子进程的启动、IPC 注册、READY 上报、心跳、重载、停止和事件循环路径禁止同步等待阻塞；需要等待外部条件时必须使用协程调度、短预算非阻塞轮询或 IPC 回调，确保进程始终能继续收发 Master 消息。
- 禁止在 Codex shell 中直接运行阻塞式前台服务或 watcher。
- 禁止在默认 WLS 端口 `9501` 上测试 AI 改动。
- 禁止复用 AI 测试实例名，禁止测试结束后遗留实例。
- **禁止批量替换与批量脚本改代码**（见下条专节）。

### 5.1 禁止批量替换与批量脚本改代码

跨文件机械改写极易误伤上下文、漏改测试、破坏 i18n 与模块边界；AI 必须按文件、按语义逐处修改。

- **禁止**用跨文件/跨目录的批量文本替换修改业务或框架源码，包括但不限于：编辑器全局 Replace All、多文件正则替换、`sed`/`awk`、`perl -pi`、PowerShell 批量 `-replace`、ripgrep/管道写回全仓、`codemod` 或同类工具**不经逐文件审阅**即全库应用。
- **禁止**编写或执行一次性批量修改脚本（Python / PHP / Shell / Node 等）对多个源文件做机械改写；不得用「先扫全仓再统一替换」代替理解后的定向编辑。
- **必须**：按模块与文件逐个定位；每次只改已读过的文件或小范围片段；同一模式出现在多处时，**逐处**核对命名空间、import、语义、i18n、测试与调用方后再改，不得默认所有出现处等价。
- **允许**的例外须同时满足：用户**明示**要求批量处理，或属于框架/仓库**已文档化**的官方生成或迁移命令（如 `php bin/w setup:upgrade`、文档载明的 codegen CLI）；仍须对 diff **逐文件**验收。AI 自写的临时批量脚本**不属于**例外。

## 6. Codex 特别红线：禁止兜底代码

这一条是全局强约束，优先级高于“先跑通再说”的任何习惯。

- 禁止 Codex 写兜底代码、补丁式代码、掩盖真实问题的兼容分支。
- 禁止为了“先不报错”加入默认值吞错、自动降级、假返回、宽松兼容、静默跳过、临时 fallback。
- 问题只能通过两种路径处理：1）直接解决根因；2）按框架方式在正确边界上解决。
- 如果当前问题暂时无法直接修复，必须明确写出阻塞点、根因假设、缺失的框架能力或契约，再给出下一步；不能用兜底逻辑冒充解决。
- 用户没有明确要求临时止血时，不得主动写临时兼容层。
- 即使用户要求“先兼容一下”，也必须先说明这是否违反框架边界；若违反，优先给出框架级方案。

## 7. 全局必做项

- 用户可见文案必须使用 `__()`、`<lang>` 或框架安全等价形式。
- i18n 占位符使用 `%{1}` 或 `%{name}`。
- 所有用户可见文案的默认源文案必须使用简体中文，例如 `__('加入购物车')`；禁止默认写 `__('Add to cart')` 这类英文 source/key。
- `__()`、`<lang>`、Hook 契约、模板文案和 JS 可见文案都以简体中文作为 source/key；`zh_Hans_CN` 维护中文到中文，`en_US` 维护中文到英文。
- 涉及 i18n 的实现、修复或验收时，必须保证 `zh_Hans_CN` 与 `en_US` 都以同一中文 source/key 对齐完整。
- PHP 控制器中的 Flash / 会话提示必须使用 `Weline\Framework\Manager\MessageManager` 静态门面。
- `__()` 的运行时调用只能在 PHP 代码内部使用；禁止在 `.phtml` 模板纯文本/标签属性中直接写 `__()` 字面量，除非它位于 `<?= ... ?>`、`<?php ... ?>` 或等价 PHP 包裹代码内。
- `.phtml` 能用 Taglib / 视图 DSL 时优先用 Taglib / DSL。
- 前端请求交互必须统一通过 Theme 的 `theme.js` 调用内置 `weline-api`，由 worker 请求链路执行；业务模块只声明或注册自己的 JS 能力，不在页面模板或模块脚本中自行拼请求。
- 站内浏览器业务 API 只能写 `const Api = await Weline.Api.resource('provider')`、`Weline.Api.graph()` 或 `Weline.Api.stream()`；不得在业务 JS / `.phtml` 内联脚本 / API 文档示例中新增 direct `fetch(window.api(...))`、`XMLHttpRequest`、`$.ajax`、`axios`、`new EventSource(url)`。
- 前端 worker 请求固定走 `theme.js -> weline-api -> worker -> /api/framework/query-bin -> FrontendQueryGateway -> QueryProvider/service`；`/api/framework/query-bin` 是协议实现细节，不允许业务开发者手写 URL。
- 只要任务涉及主题前端交互 API、站内浏览器业务 API、前端 QueryProvider 调用、流式订阅或前端请求治理，**所有 AI 都必须先读取并遵守** `dev/ai/skills/前端主题工程师-前端API交互/SKILL.md`，再开始实现、评审或给出示例。
- ORM 查询链必须以 `fetch()` 或 `fetchArray()` 结束后才执行；`save()` 不需要 `fetch`。
- 新增控制器后执行 `php bin/w setup:upgrade --route`。
- Bug 修复说明写到相关模块 `doc/`，不要写到仓库根目录。
- 涉及运行时、路由、页面或接口行为时，必须提供可复现验证命令。

## 8. 布局文件（layout）约定

布局文件指主题/模块下的**页面骨架布局**（如 `view/theme/frontend/layouts/**/default.phtml`）。其职责是提供**默认布局信息**：在无人使用主题编辑器定制、Hook 尚未接入、部件尚未拖入时，页面仍能完整展示主题网页骨架与基础可读内容。

### 定位（必须理解）

- layout ≈ **默认布局蓝图 + 空态占位 + 扩展挂载点**，不是业务功能实现层。
- **大多数情况下的 layout 内容应只有两类**：（1）**基础信息**——页眉/页脚/面包屑等区域骨架、通过 Partials/Block 加载的通用壳层、`@param` / `@meta` 上的布局级默认开关；（2）**未使用时的占位**——Hook 无人注册时的 `<else/>` 说明、插槽无部件时的占位块、`meta.content` 为空时的提示，让编辑器和访客仍能看懂“这里将来放什么”。
- 典型内容：区域划分 HTML、`<w:slot>` / `<w:hook>` / `<w:block>` / `<w:widget>` 挂载点、Hook 未命中时的 `<else/>` 占位文案/图示、与结构相关的呈现 CSS、`@param` / `@meta` 声明的**布局级默认值**（如是否显示 header/footer/侧栏）。
- 目标：即使当前站点**没有任何业务模块输出**、主题编辑器里也还没拖入部件，访客仍能看到完整主题网页（页眉、内容区占位、页脚），而不是空白或塌陷；有模块接入后，占位由 Hook/Widget/控制器输出**替换**，layout 本身不必改。

### 允许写入 layout 的内容

- 默认布局结构的 HTML（容器、主栏/侧栏/页脚等区域划分）。
- 插槽、Hook、Block、Widget 的**声明与挂载**（只负责“在哪里接”，不负责“接什么业务”）。
- Hook 未被使用、插槽无部件、无控制器内容时的**占位信息**（图标 + 简短说明，建议用 `<lang>`）。
- 与页面骨架直接相关的**呈现用 CSS**（BEM、响应式、占位样式）；可含 `@param` 控制的结构级显示开关（如 `meta.showSidebar`），值来自布局元信息默认值，不从数据库/服务拉取。
- 通过 Taglib / 布局 DSL 做**结构级**分支（如 `<if condition="meta.showHeader">`），条件数据仅来自 `meta` / `@param` 默认值。

### 禁止写入 layout 的内容

- **交互逻辑**：事件监听、DOM 操作、表单提交处理、购物车/加购/切换 Tab 等；禁止内联 `<script>` 业务脚本。
- **业务逻辑与数据**：ORM 查询、服务层编排、权限判断、价格/库存/用户态计算；禁止在 layout 顶部写 PHP 块拉取商品、订单、用户等业务数据。
- **请求与 API**：`fetch` / `XMLHttpRequest` / `axios` / 直连 API URL（与第 7 节前端请求规则一致）。
- 为“先跑起来”写的临时兼容、假数据、硬编码业务列表。

### 与其他层的分工

| 层级 | 职责 |
|------|------|
| **layout** | 页面区域划分、默认占位、Hook/插槽/Block 挂载点、无人定制时的可浏览骨架 |
| **Hook / partial / widget 模板** | 可被替换/拖入的具体内容片段 |
| **控制器 / 服务层** | 业务数据准备与编排 |
| **模块 JS（theme.js / 业务脚本）** | 用户交互与 API 调用 |

修改商品详情、购物车、结账等页面时，**不得**把本应在 Hook、Widget、服务层或 JS 中的逻辑塞进 layout；layout 只保证“没功能时也能看见主题网页”。

## 9. 架构与框架实现原则

- 写代码必须先从框架边界、框架机制、模块职责和扩展点出发，不能只为当前报错堆局部补丁。
- 框架级思考是默认要求，不是用户额外提出时才做。
- 即使问题只出现在单个页面或单个模块，也要先定位背后的共享机制、契约、生成链路、Hook/事件边界或运行时入口。
- 优先使用框架已有能力，例如服务层、事件、Hook、Taglib、配置、队列、权限、i18n、ORM、路由生成和模块文档约定。
- 如果现有方案需要违反这些原则才能完成，必须停下来说明冲突并提出干净方案。
- 修复应落在稳定抽象层或清晰模块边界内，避免在调用端、模板端或前端脚本里反复堆兼容逻辑。

## 10. Browser-First 前端验证

这部分已吸收 Codex 全局规则，适用于 Weline / WeShop 和任何浏览器可见功能。

- 只要页面已经出现报错、白屏、致命异常、明显脚本错误或渲染中断，必须立即先修复该报错，再继续做后续功能调整、样式优化或非阻塞重构。
- 只要是浏览器可见的前端功能，最终验证必须依赖 Codex 内置 Browser。
- 单元测试、路由测试、源码检查、命令行 HTTP 检查都不能替代真实浏览器冒烟验证。
- Browser 冒烟至少覆盖：打开目标 URL、确认无明显错误、执行核心交互、核对可见结果。
- 如果 Browser 验证被 runtime、WLS、端口、认证、证书、环境或沙箱阻塞，必须明确报告具体阻塞点。
- Browser 验证被阻塞时，不得声称前端流程“已验证”“已完成”“可交付”。

## 11. No Premature Test Solidification

- 在用户可见功能的开发阶段，浏览器冒烟通过前，禁止提前新增、重写或固化单元测试、E2E 回归测试来锁定未稳定行为。
- 正确顺序是：先完成功能，再用 Browser 验证真实可见行为，最后才补或更新测试。
- 如果功能仍在变化中，测试只能服务诊断，不能冒充最终完成信号。

## 12. WLS 与命令安全

- 长运行 WLS / dev 服务只能以后台或 daemon 方式启动，并使用唯一测试实例名。
- 测试实例端口使用 `9502+`，默认端口 `9501` 视为生产实例，不得触碰。
- 启动示例：`php bin/w server:start -p 9502 -n ai-test-{unique-id}`。
- 代码变更后优先 `server:reload`；涉及主进程级变更时才 `server:restart -r`。
- 验证必须使用有界命令，例如 `php bin/w http:request /`。
- 测试结束必须执行 `php bin/w server:stop -n ai-test-{unique-id}`。

## 13. 文档与证据

- 修复报告写到相关模块 `doc/`，不要写根目录。
- 只更新长期有复用价值的内容，不写无价值流水账。
- 交付说明必须区分：已改内容、已验证证据、未验证项、剩余风险。
- 如果只清掉一个 blocker，必须明确下一个 blocker，不能夸大为全链路完成。
- 验收报告必须来自实际执行结果，禁止使用“理论上”“应该可以”“看代码推测”替代证据。

## 14. 多智能体协作边界

- 主智能体负责拆分、分派、整合和验收。
- 子智能体只处理分配到的明确边界，不得擅自扩范围。
- 并行前必须检查同文件、同方法、相邻高耦合区域的覆盖风险。
- 发现冲突、阻塞或跨角色影响时，立即回到主智能体统一决策。

## 15. 交付验收底线

- 不以“已实现”替代“已验证”。
- 不以静态检查、命令输出、单元测试或 E2E 结果替代最终浏览器可见确认。
- 任何前端或浏览器可见问题，在提交“已修复”“已完成”“可交付”前，必须先由 AI 自己完成 Browser 冒烟验证。
- 后端机制类功能也必须找到真实入口、真实命令或真实状态结果来验证，不能只凭源码存在或注册成功就宣称通过。
- 遇到验证阻塞时，必须明确写出阻塞原因、未验证范围、目标入口和替代验证命令。
- 最终交付必须让下一位 AI 或工程师可以继续定位、验证和回滚。
