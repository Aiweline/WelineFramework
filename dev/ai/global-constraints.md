# AI 总则与全局约束

本文件是 WelineFramework 仓库内 AI 开发规则的唯一总则（Single Source of Truth）。

- 根目录 `AI-ENTRY.md`、`AI-README.md`、`AGENTS.md`、`CLAUDE.md` 只作为入口索引，不再维护规则正文。
- `dev/ai/README.md`、`dev/ai/skills/_index.md`、`dev/ai/agent/README.md` 只负责路由到本总则和对应技能。
- 新增或修改跨角色、跨技能、跨工具的 AI 规则时，优先更新本文件；专项技能只写该技能独有规则。

## 1. 沟通与决策：必须具备产品经理思维和对抗性思维

AI 不能机械顺从用户输入。每次沟通、需求分析和方案执行前，必须同时做产品价值判断和现实性判断。

- 先用 Web 软件产品经理视角还原目标用户、使用场景、业务目标、用户路径、转化或效率指标、权限边界、异常状态和验收口径。
- 当需求不清楚、只描述界面或实现、不说明用户问题或成功标准时，必须先补齐需求假设、提出关键澄清问题，或给出可验证的最小方案，不得直接堆实现。
- 判断用户方案是否符合当前项目架构、已有约束、成本收益、可维护性和验证条件。
- 如果用户方案不合理、风险高、范围过大、方向错误，必须明确指出问题，并给出理由。
- 如果存在更简单、更稳定、更符合框架设计的方案，必须主动提出替代方案，并说明取舍。
- 如果用户坚持原方案，可以继续执行，但必须先记录风险、影响范围和验证方式。
- 产品经理思维和对抗性思维必须一起工作：先问“这个产品行为是否有用户价值、清晰闭环和可验收结果”，再问“这个实现是否符合工程边界、成本收益和可验证性”。
- 对抗性思维不是抬杠，必须基于代码、文档、运行状态、项目规则或明确工程经验，而不是主观否定。

## 2. 反顺从原则

AI 的目标是交付正确、稳定、可验证的结果，不是迎合用户的每一句输入。

- 当用户输入与项目真实情况冲突时，优先维护工程正确性，并用清晰理由推动更优方案。
- 当需求描述与代码、文档、运行态、测试证据冲突时，以可验证证据为准。
- 当用户只给出实现方案而未说明目标时，先还原真实目标，再评估该方案是否值得执行。
- 不把“用户说了”当作正确性证明，也不把“不确定”包装成已验证结论。

## 3. 开发执行：默认评估并行拆分

面对开发任务时，AI 不能默认单线程顺序完成。必须先判断任务是否可拆分。

- 可并行任务应按模块、层级或风险边界拆分，例如后端逻辑、前端视图、数据结构、测试验证、文档更新。
- 当环境支持子智能体、多代理或并发协作时，应优先并发分派互不冲突的子任务，以提升开发速度。
- 拆分前必须评估文件冲突、接口依赖和验证顺序，避免多个智能体覆盖同一文件或同一逻辑块。
- 不适合并行的小任务、强依赖任务或高冲突任务，应说明原因后串行处理。
- 主智能体负责最终整合、冲突检查、测试验证和交付结论，不能只分派不验收。

## 4. 开发前阅读顺序

1. 先读本总则。
2. 再读 `AI-ENTRY.md` 获取仓库入口索引。
3. 优先读 `dev/ai/diagrams/00-INDEX.txt` 和相关模块文档。
4. 按需读取 `dev/ai/skills/_index.md` 命中的技能。
5. 最后才大范围读取源码。

## 5. 全局禁止项

- 禁止直接修改 `generated/`。
- `generated/` 是框架生成目录；禁止通过修改该目录下文件来补充功能、修复逻辑或绕过源码问题。必须回到对应的框架源码、生成规则、模板、编译/构建链路或注册机制本身处理，再通过标准生成流程产出 `generated/` 内容。
- 禁止使用 `routes.xml`，路由通过控制器目录和 `php bin/w setup:upgrade --route` 同步。
- 禁止用 JavaScript 原生 `alert()`、`confirm()`、`prompt()`。
- 禁止硬编码用户可见文案。
- 禁止为了让当前任务“看起来完成”而临时绕过架构、框架约定、权限边界、i18n、配置系统或验证流程。
- 禁止用硬编码、一次性兼容补丁、隐藏开关、假数据或跳过真实调用来伪造功能通过。
- 禁止在 `Setup/Upgrade.php` 做字段 CRUD，字段结构使用 Model `#[Col]` / `#[Index]` 后执行 `setup:upgrade`。
- 禁止在 `<w:*>` 或自定义 Taglib 属性中写 `<?= ?>` 或 `<?php ?>`。
- 禁止在 `.phtml` 文件中写 `declare(strict_types=1)`。
- 禁止在 WLS 运行时敏感路径使用 `sleep()`、`usleep()`、`die()`、`exit()`。
- 禁止在 Codex shell 中运行阻塞式前台服务或 watcher，例如 `server:start --no-daemon`、前台 dev server、watch mode。
- 禁止把详细修复过程报告写到仓库根目录。
- 禁止在默认 WLS 端口 `9501` 上测试 AI 改动。
- 禁止复用测试实例名。
- 禁止测试结束后遗留 AI 测试实例。

## 6. 全局必做项

- 用户可见文案必须使用 `__()`、`<lang>` 或对应框架安全形式。
- i18n 占位符使用 `%{1}` 或 `%{name}`。
- `__()`、Hook 约定等括号内默认源文案使用简体中文；英文场景走语言包。
- 涉及 i18n 语言翻译的实现、补全、修复或验收时，必须保证 `zh_Hans_CN` 和 `en_US` 两个语言的完备性与一一对应；其他语言不纳入本仓库当前强制要求，除非需求明确指定。
- PHP 控制器里的 Flash / 会话提示必须使用 `Weline\Framework\Manager\MessageManager` 静态门面，例如 `MessageManager::warning(__('提示文案'))`。
- 禁止写成 `$this->getMessageManager()->addWarning()` / `addError()` / `addSuccess()` 等实例链式调用，除非框架官方文档对特定场景另有明确例外。
- `.phtml` 视图模板在能力允许时优先使用 Taglib / 视图 DSL，例如 `<notempty>`、`<empty>`、`<if>`、`<var>`、`<lang>`。
- ORM 查询链必须以 `fetch()` 或 `fetchArray()` 结束才会执行；`save()` 不需要 fetch。
- 表结构变更使用 Model `#[Col]` / `#[Index]` 后执行 `php bin/w setup:upgrade`。
- 新增控制器后执行 `php bin/w setup:upgrade --route`。
- Bug 修复后的说明写到相关模块 `doc/` 目录，不写到根目录。
- Bug 修复后按影响范围更新模块 README、架构文档或 API 文档。
- 涉及运行时、路由、页面或接口行为时，必须提供可复现验证命令；若要作为可交付结果提交，还必须补充对应 URL、入口或流程的浏览器冒烟证据，无法验证时明确说明原因。

## 7. 架构与框架实现原则

- 写代码必须先从架构边界、框架机制、模块职责和扩展点出发，不得只为当前报错或当前页面做局部堆补丁。
- 优先使用框架已有能力，例如服务层、事件/Hook、Taglib、配置、队列、权限、i18n、ORM、路由生成和模块文档约定。
- 如果现有方案需要违反原则才能完成，必须停下来说明冲突，提出干净方案或框架级方案，不能默认用临时硬编码推进。
- 修复应落在稳定抽象层或清晰模块边界内，避免在调用端、模板端或前端脚本里反复堆同类兼容逻辑。
- 临时兼容只允许作为明确标注的短期止血方案，并必须写明移除条件、后续框架级修复路径和验证边界。

## 8. WLS 与命令安全

- 长运行 WLS / dev 服务只能以后台或 daemon 方式启动，并使用唯一测试实例名。
- 测试实例端口使用 `9502+`，默认端口 `9501` 视为生产实例，不得触碰。
- 启动示例：`php bin/w server:start -p 9502 -n ai-test-{unique-id}`。
- 代码变更后用 `server:reload`，主进程级变更用 `server:restart -r`，只作用于测试实例。
- 验证必须使用有界命令，例如 `php bin/w http:request /` 或状态检查命令。
- 测试结束必须执行 `php bin/w server:stop -n ai-test-{unique-id}`。
- 如果确实需要观察实时日志，必须用后台隐藏 helper 加有界读取，不能阻塞主 Codex shell。

## 9. 文档与证据

- 修复报告写到相关模块 `doc/`，不要写根目录。
- 不写无价值的逐步流水账；只更新需求、架构、API、README、验证状态等可长期复用内容。
- 交付说明必须区分：已改内容、已验证证据、未验证项、剩余风险。
- 如果只清除了一个 blocker，必须说明下一个 blocker，不能夸大为全链路完成。
- 如果验证依赖分支、SHA、实例名、URL、命令或日志，必须给出具体证据。
- 验收报告必须来自实际执行结果；禁止用“应该可以”“理论上”“看代码推测”替代验证证据。

## 10. 多智能体协作边界

- 技术主管或主智能体负责拆分、派发、整合和验收。
- 子智能体只处理被分配的明确边界，不擅自扩大范围。
- 并行前必须检查同文件、同方法、相邻高耦合块的覆盖风险。
- 发现冲突时暂停相关提交，由主智能体裁决：可并行、需串行、先合并基线、或重新拆分。
- 发现问题、风险、阻塞、跨角色影响或验证失败时，必须通知 `@Weline-技术主管`。

## 11. 交付验收底线

- 不以“已实现”替代“已验证”。
- 不以静态检查、HTTP、WLS、单元测试或 E2E 结果替代最终浏览器冒烟确认。
- 任何开发类问题在提交“已修复”“已完成”或“可交付”结论前，必须先由 AI 自己使用浏览器完成冒烟验证，而不是只看代码、日志、命令输出或测试结果。
- 即使问题主要发生在后端、接口、数据、配置、脚本或队列层，也必须找到受影响的真实浏览器入口、页面、后台路径或最小用户流程做确认；找不到浏览器落点时，只能按“未完成浏览器校验 / 被阻塞”交付，不得宣称完成。
- 浏览器冒烟必须至少覆盖：打开目标 URL、确认页面无明显错误、执行核心交互、核对可见结果；失败时不得提交通过结论。
- 浏览器冒烟优先使用 `@浏览器` / browser-use 能力；若环境无法使用浏览器，必须明确写出阻塞原因、未验证范围、目标 URL/入口和可复现的替代验证命令，且该次交付不能标记为通过。
- 不因验证困难而省略风险说明。
- 不把用户无法复现的结论当作交付证据。
- 禁止猜想式验收报告；报告必须包含实际命令、浏览器步骤、URL、结果或失败证据。
- 最终交付必须让下一位 AI 或人类工程师能继续定位、验证和回滚。
