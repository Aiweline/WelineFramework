# WelineFramework 开发者入口

本文档承载面向开发者的完整介绍。根目录 `README.md` 只保留项目主页、快速启动和导航入口。

WelineFramework 是面向复杂业务系统的 PHP 8.4+ 框架。它把模块生命周期、自动路由、ORM Schema、事件/Hook、后台 ACL、主题模板、国际化、CLI 运维和 WLS 长运行服务放进同一套工程模型，让团队可以把业务能力做成可安装、可升级、可扩展、可验证的模块。

> WLS 边界：`php bin/w server:start` 启动的是 Weline 框架内置长运行服务器 WLS。它用于 HTTP Worker、Session Server、Memory Server、Maintenance Worker、Dispatcher/Gateway、热重载和运行时治理；它是框架运行时，不是通用 HTTP 调试入口。传统 FPM 部署仍然是一等部署路径。

更多产品能力、行业方案和业务场景，请访问 [www.aiweline.com](https://www.aiweline.com)。

## The Pitch

多数 PHP 项目会在业务变大后遇到同一组问题：路由散落、SQL 与模型脱节、后台权限补丁化、模板难复用、模块升级不可控、长运行进程容易串状态。

WelineFramework 的目标很直接：让业务系统从第一天就按模块边界生长。

| 你要解决的问题 | Weline 的做法 |
| --- | --- |
| 新业务不想污染主应用 | 每个模块独立注册、配置、菜单、权限、事件、Hook、模板资源和安装升级 |
| Controller 改了还要到处补路由 | Controller 由框架发现，路由统一生成；变更后执行 `php bin/w setup:upgrade --route` |
| 数据库结构和 Model 容易漂移 | 表、字段、索引通过 `#[Table]`、`#[Col]`、`#[Index]` 等属性贴近模型声明 |
| 后台、前台、模板和组件需要可扩展 | area、Theme、Block、Taglib、Widget、Hook 共同组织页面能力 |
| 需要长连接、共享 Session、维护任务和热重载 | WLS 提供长运行服务编排，FPM 部署路径同时保留 |
| 团队要可复制的运维入口 | `bin/w` 覆盖安装、升级、缓存、模块、迁移、路由、WLS、队列、邮局、SMTP 和诊断 |

## Install

### Recommended install

一键安装适合快速准备本地运行环境。脚本会克隆仓库、准备 `extend/server` 下的运行时、安装依赖并执行初始化流程。

Linux / macOS / Git Bash:

```bash
curl -fsSL https://gitee.com/aiweline/WelineFramework/raw/master/bin/bootstrap.sh | bash -s --
```

Windows PowerShell:

```powershell
$f="$env:TEMP\weline-bootstrap.ps1"; irm 'https://gitee.com/aiweline/WelineFramework/raw/master/bin/bootstrap.ps1' -OutFile $f; & $f
```

常用参数：

- `-b <branch>`: 指定分支，例如 `-b dev`
- `-y`: 自动确认安装提示
- `-f`: 强制重新安装，会清空已有数据
- `--path-only`: 只写入 PATH，不重新安装组件
- `php` / `pgsql` / `mysql`: 指定安装组件

请以当前用户执行安装命令，不要直接用 `sudo` 启动一键安装。Linux 以 root 运行时，脚本会创建并切换到 `weline` 用户执行克隆和安装。

### Clean source install

纯净安装只获取源码和 Composer 依赖，不安装 `extend/server` 下的 PHP 或项目本地数据库。

```bash
git clone https://gitee.com/aiweline/WelineFramework.git weline
cd weline
composer install
php bin/w command:upgrade
```

查看当前版本支持的安装参数模板：

```bash
php bin/w system:install:sample
```

启动 WLS 本地实例：

```bash
php bin/w server:start
```

### Requirements

- PHP `^8.4`
- Composer `^2.7`
- MySQL / MariaDB / PostgreSQL 等可用数据库环境
- Nginx / Apache 或 WLS

## Why Developers Care

### Module-native architecture

模块不是一个目录命名习惯，而是完整的交付边界。一个模块可以拥有自己的注册、依赖、配置、菜单、权限、事件、Hook、模板、静态资源、安装升级脚本和文档。模块之间通过契约、事件、Hook、Query Provider、队列或接口协作，避免直接耦合内部实现。

### Route generation by convention

Controller 由框架发现并生成路由。新增或调整 Controller 后执行：

```bash
php bin/w setup:upgrade --route
```

这让路由成为框架可验证的结果，而不是项目里散落的手写配置。

### Schema close to the model

数据表字段、索引和模型结构通过 PHP 属性声明，结构意图贴近业务模型：

```php
#[Table('demo_entity')]
class Entity extends Model
{
    #[Col(type: 'varchar', length: 255, comment: 'Name')]
    public string $name;

    #[Index(columns: ['name'])]
    public string $nameIndex;
}
```

字段或索引变更后执行：

```bash
php bin/w setup:upgrade
```

### Service layer that stays readable

Controller 负责请求边界、权限、参数和响应；复杂业务进入 Service；Model 负责数据表达和 ORM 查询。这个分工让后台、API、CLI、队列和 WLS Worker 可以复用同一套业务能力。

### Theme system for real products

前后台模板、Block、Taglib、Widget、Hook 和静态资源按 `frontend` / `backend` area 与模块边界组织。业务页面可以被模块增强，也可以被主题复写，而不是把所有页面能力压进 Controller 或全局模板。

### Runtime services, not just request-response

WLS 是框架内置长运行服务器，覆盖 Master/Worker、Dispatcher、Gateway、HTTP Redirect、Session Server、Memory Server、Maintenance Worker、IPC 控制面、热重载、证书动态加载和进程治理。它适合需要长连接、共享状态、后台维护和高频请求复用的场景。

## Contributing from Source

新贡献者先阅读 [项目文档索引](../README.md)、[架构总览](./架构总览.md) 和相关模块 `doc/README.md`。短路径如下：

1. Fork 并克隆仓库，安装 PHP `^8.4`、Composer `^2.7` 和数据库环境。
2. 执行 `composer install` 与 `php bin/w command:upgrade`。
3. 通过 `php bin/w system:install:sample` 查看安装参数模板，再执行项目安装。
4. 修改 Controller 或路由后执行 `php bin/w setup:upgrade --route`。
5. 修改 Model 字段或索引后执行 `php bin/w setup:upgrade`。
6. 使用 `php bin/w http:request /`、`php bin/w http:request admin -b` 或独立 WLS 实例验证改动。

提交前请确认没有包含密钥、token、`.env`、生产连接串或本地敏感配置。

## Runtime Commands

```bash
php bin/w                              # 查看可用命令
php bin/w command:upgrade              # 刷新命令注册
php bin/w setup:upgrade                # 升级模块、模型结构和系统配置
php bin/w setup:upgrade --route        # Controller 或路由变更后刷新路由
php bin/w server:start                 # 启动 WLS
php bin/w server:status --all          # 查看 WLS 实例和进程状态
php bin/w server:reload                # 热重载业务代码和 Worker
php bin/w http:request /               # 使用框架命令请求前台入口
php bin/w http:request admin -b        # 使用 backend 上下文请求后台入口
```

更多命令见 [命令文档](../commands/) 和对应模块文档。

## Repository Layout

```text
WelineFramework/
|-- app/
|   |-- code/             Core and business modules
|   |   `-- Weline/       Framework core modules
|   |-- design/           Frontend/backend themes, templates, static assets
|   `-- etc/              Project configuration, including env.php
|-- bin/                  bin/w CLI and install/bootstrap scripts
|-- docs/                 Project docs, API docs, deployment docs, multilingual README
|   |-- commands/         CLI command documentation
|   |-- readme/           Localized README files
|   `-- weline/           Framework-level developer and architecture docs
|-- dev/ai/               AI engineering rules, diagrams, skills, task records
|-- extend/server/        Project-local runtime components
|-- generated/            Generated artifacts; do not edit by hand
|-- pub/                  Web entry and public static assets
|-- var/                  Cache, logs, session, runtime state
`-- vendor/               Composer dependencies
```

模块内部文档保留在各模块自己的 `doc/` 目录中。处理具体模块前，优先阅读模块文档，再进入源码。
