# WelineFramework 测试指南

## 📋 概述

WelineFramework 支持多种测试方式，包括单元测试、E2E 测试等。本文档介绍如何编写和运行测试。

## 🎯 测试类型

### 1. 前端单元测试（Vitest）

**位置**: `tests/unit/`

**用途**: 测试前端 JavaScript 代码

**特性**:
- ✅ Watch 模式：文件变更自动重新运行测试
- ✅ 快速执行：基于 Vite，测试运行速度极快
- ✅ UI 界面：提供可视化测试界面
- ✅ 覆盖率报告：自动生成代码覆盖率报告

**快速开始**:

```bash
cd tests/unit
npm install
npm start  # Watch 模式，文件变更自动测试
```

**详细文档**: 参见 `tests/unit/README.md`

### 2. E2E 测试（Playwright）

**位置**: `tests/e2e/`

**用途**: 端到端测试，验证前端功能是否正常工作

**特性**:
- ✅ 模块化测试用例：测试用例放在模块目录下
- ✅ 自动收集：系统升级时自动生成模块信息
- ✅ 统一运行：收集后的测试用例统一运行

**快速开始**:

```bash
# 1. 生成 modules.json（系统升级时自动生成）
php bin/w setup:upgrade

# 2. 运行测试
cd tests/e2e
npm install
npm start  # 一条命令完成收集和运行
```

**详细文档**: 参见 `tests/e2e/README.md`

### 3. PHP 单元测试（PHPUnit）

**位置**: 模块目录下的 `test/` 目录

**用途**: 测试 PHP 后端代码

**快速开始**:

```bash
# 运行所有测试
vendor/bin/phpunit

# 运行特定模块的测试
vendor/bin/phpunit app/code/Weline/Theme/test/
```

## 📝 编写测试用例

### 前端单元测试（Vitest）

**测试文件位置**:

```
app/code/Weline/Theme/view/theme/frontend/assets/js/
├── theme.js              # 源代码
└── theme.test.js         # 测试用例（可选）

# 或统一放在
tests/unit/specs/
└── Weline/
    └── Theme/
        └── theme.test.js
```

**示例测试用例**:

```javascript
// tests/unit/specs/Weline/Theme/theme.test.js
import { describe, it, expect, beforeEach, afterEach } from 'vitest';
import { readFileSync } from 'fs';
import { resolve } from 'path';

describe('Weline Theme.js', () => {
  let themeScript;
  
  beforeEach(() => {
    // 加载主题 JS 文件
    const themePath = resolve(__dirname, '../../../../app/code/Weline/Theme/view/theme/frontend/assets/js/theme.js');
    themeScript = readFileSync(themePath, 'utf-8');
    
    // 模拟全局配置
    window.__WelineThemeConfig = {
      baseUrl: '/',
      theme: 'default',
      locale: 'zh_CN'
    };
    
    // 执行脚本
    eval(themeScript);
  });
  
  afterEach(() => {
    // 清理全局变量
    if (window.Weline) delete window.Weline;
    if (window.__WelineThemeConfig) delete window.__WelineThemeConfig;
  });
  
  it('should initialize Weline object', () => {
    expect(window.Weline).toBeDefined();
    expect(typeof window.Weline).toBe('object');
  });
  
  it('should have theme management', () => {
    if (window.Weline && window.Weline.Theme) {
      expect(window.Weline.Theme).toBeDefined();
    }
  });
});
```

### E2E 测试（Playwright）

**测试文件位置**:

```
app/code/Weline/Theme/test/e2e/frontend/
└── theme-override.spec.js
```

**示例测试用例**:

```javascript
// app/code/Weline/Theme/test/e2e/frontend/theme-override.spec.js
const { test, expect } = require('@playwright/test');

test.describe('Theme frontend override behavior', () => {
  test('WeShop_Default 主题应覆盖 Weline 基础主题的 header 模板', async ({ page }) => {
    await page.goto('/');
    
    const header = page.locator('header');
    
    // 验证使用的是 WeShop 头部结构
    await expect(header.first()).toHaveClass(/sticky/);
    
    // 验证 WeShop 头部特有的文案
    await expect(page.getByText("Shop by Category")).toBeVisible();
  });
});
```

### PHP 单元测试（PHPUnit）

**测试文件位置**:

```
app/code/Weline/Theme/test/
└── Unit/
    └── Helper/
        └── ThemePathResolverTest.php
```

**示例测试用例**:

```php
<?php
namespace Weline\Theme\Test\Unit\Helper;

use PHPUnit\Framework\TestCase;
use Weline\Theme\Helper\ThemePathResolver;

class ThemePathResolverTest extends TestCase
{
    public function testResolveThemePath()
    {
        $resolver = new ThemePathResolver();
        $path = $resolver->resolve('frontend', 'default', 'header/default.phtml');
        
        $this->assertStringContainsString('header/default.phtml', $path);
    }
}
```

## 🚀 运行测试

### 前端单元测试

```bash
cd tests/unit

# Watch 模式（推荐，文件变更自动测试）
npm start

# UI 模式（可视化界面）
npm run test:ui

# 运行一次（CI/CD）
npm run test:run

# 生成覆盖率报告
npm run test:coverage
```

### E2E 测试

```bash
cd tests/e2e

# 一键运行（自动收集和运行）
npm start

# 运行特定模块的测试
npm start -- --module=Weline_Theme

# UI 模式
npm run test:ui
```

### PHP 单元测试

```bash
# 运行所有测试
vendor/bin/phpunit

# 运行特定模块的测试
vendor/bin/phpunit app/code/Weline/Theme/test/

# 生成覆盖率报告
vendor/bin/phpunit --coverage-html coverage/
```

## 🔧 配置说明

### Vitest 配置

配置文件: `tests/unit/vitest.config.js`

主要配置项:
- `environment: 'happy-dom'` - DOM 环境
- `watch: true` - 默认启用 watch 模式
- `globals: true` - 全局测试 API
- 路径别名配置

### Playwright 配置

配置文件: `tests/e2e/playwright.config.js`

主要配置项:
- `baseURL` - 测试服务器地址
- `testDir` - 测试文件目录（动态收集）
- `projects` - 浏览器配置

## 📚 最佳实践

### 1. 测试文件命名

- 单元测试: `*.test.js` 或 `*.spec.js`
- E2E 测试: `*.spec.js`
- PHP 测试: `*Test.php`

### 2. 测试组织

- 按模块组织测试用例
- 测试用例与源代码放在一起（可选）
- 或统一放在 `tests/` 目录下

### 3. 测试编写原则

- **AAA 模式**: Arrange（准备）、Act（执行）、Assert（断言）
- **独立性**: 每个测试应该独立，不依赖其他测试
- **可读性**: 使用描述性的测试名称
- **覆盖率**: 关键功能应该有测试覆盖

### 4. Watch 模式使用

- 开发时使用 watch 模式，文件变更自动测试
- CI/CD 时使用单次运行模式

## 🐛 故障排除

### 前端单元测试

**问题**: 模块解析失败

**解决**: 检查 `vitest.config.js` 中的路径别名配置

**问题**: DOM API 不可用

**解决**: 确保配置了 `environment: 'happy-dom'`

### E2E 测试

**问题**: `modules.json` 不存在

**解决**: 运行 `php bin/w setup:upgrade` 生成

**问题**: 服务器连接失败

**解决**: 确保服务器正在运行 `php bin/w server:start`

### PHP 单元测试

**问题**: 类找不到

**解决**: 检查 autoload 配置和命名空间
