# WelineFramework 部署文档

## 🚀 一键快速启动

**最简单的方式 - 运行快速启动脚本：**

```bash
# 给脚本执行权限并运行
chmod +x quick-start.sh && ./quick-start.sh
```

脚本会自动检测您的环境并选择最佳部署方式：
- ✅ 自动检测 Docker 环境
- ✅ 自动检测 PHP 8.2+ 环境  
- ✅ 智能选择部署方式
- ✅ 自动显示访问信息

## 依赖环境

- **PHP 8.2+** - 框架运行环境
- **SQLite** - 内置本地数据库，无需额外配置【只是工具性质，如果后期需要大量使用，可以考虑切换数据库即可】
- **Composer** - PHP依赖管理工具
- **Docker** (可选) - 容器化部署，零环境配置

## 快速部署

WelineFramework 提供多种部署方式，选择最适合您的方式：

### 方式一：一键部署脚本 (推荐)

```bash
# 给脚本执行权限
chmod +x deploy.sh

# 运行一键部署脚本
./deploy.sh

# 或者指定部署模式
./deploy.sh --dev    # 开发环境
./deploy.sh --prod   # 生产环境
```

**脚本功能：**
- ✅ 自动检测并安装 PHP 8.2+
- ✅ 自动安装 Composer
- ✅ 智能权限处理
- ✅ 自动设置环境变量
- ✅ 支持多操作系统 (Ubuntu/Debian/CentOS/Arch/macOS)

### 方式二：Docker 部署 (最简单)

```bash
# 使用 Docker Compose 一键部署
docker-compose up -d

# 查看部署状态
docker-compose ps

# 查看日志
docker-compose logs -f
```

**Docker 优势：**
- ✅ 零环境配置，开箱即用
- ✅ 包含 PHP 8.2 + Nginx + SQLite
- ✅ 自动健康检查
- ✅ 数据持久化
- ✅ 支持 Redis/MySQL 扩展

### 方式三：手动部署

```bash
# 1. 安装依赖
php composer.phar install

# 2. 系统升级
php bin/w setup:upgrade

# 3. 安装定时任务
php bin/w cron:install
```

**部署完成！** 🎉

## Docker 部署详解

### Docker 环境要求

- **Docker** 20.10+
- **Docker Compose** 2.0+

### Docker 部署步骤

#### 1. 基础部署

```bash
# 克隆项目
git clone <repository-url>
cd WelineFramework

# 一键启动
docker-compose up -d

# 查看状态
docker-compose ps
```

#### 2. 获取访问密钥

```bash
# 获取管理员密钥
docker exec -it welineframework php bin/w admin:key:show

# 获取API密钥
docker exec -it welineframework php bin/w api:key:show
```

#### 3. 访问系统

```bash
# 后台管理
http://localhost/admin/{admin_key}/dashboard

# API接口
http://localhost/api/{api_key}/rest/v1/
```

#### 4. 常用 Docker 命令

```bash
# 查看日志
docker-compose logs -f

# 重启服务
docker-compose restart

# 停止服务
docker-compose down

# 重新构建
docker-compose up -d --build

# 进入容器
docker exec -it welineframework bash

# 执行框架命令
docker exec -it welineframework php bin/w cache:clear
```

### 4. 配置 Nginx (非Docker环境)

使用项目根目录下的 `nginx.conf` 配置文件，根据您的服务器环境调整配置：

```bash
# 复制配置文件到 Nginx 配置目录
cp nginx.conf /etc/nginx/sites-available/welineframework
ln -s /etc/nginx/sites-available/welineframework /etc/nginx/sites-enabled/

# 重启 Nginx
sudo systemctl restart nginx
```

### 5. 设置文件权限

```bash
# 设置必要的文件权限
chmod -R 755 pub/
chmod -R 755 var/
chmod -R 755 generated/
```

## 数据库配置

WelineFramework 默认使用 **SQLite 本地数据库**，无需额外配置：

```php
'db' => [
    'default' => 'sqlite',
    'master' => [
        'type' => 'sqlite',
        'path' => 'app/etc/db.sqlite',  // 数据库文件路径
    ],
],
```

**SQLite 优势：**
- ✅ 无需安装数据库服务器
- ✅ 零配置，开箱即用
- ✅ 文件型数据库，便于备份
- ✅ 性能优秀，适合中小型应用

**可选：切换到 MySQL**
如需使用 MySQL，修改 `app/etc/env.php` 配置即可。

### WLS 下登录后仍跳回登录页

现象：后台登录成功（POST 返回 302），紧接着访问 admin 又被 302 到登录页。

**在浏览器开发者工具 → Network 中确认：**

1. **POST 登录的 302 响应**
   - 看 **Response Headers** 是否有 `Set-Cookie: WELINE_SESSID=...`。
   - 若有诊断头 `X-WLS-Redirect-Cookies`：为 **0** 表示本次 302 未带 Cookie（Session/HeaderCollector 未写入）；≥1 表示已带 Cookie。
   - 若 **没有** Set-Cookie：可能是 Nginx 去掉了该头，需在 Nginx 中保留并透传 `Set-Cookie`（不要 `proxy_hide_header Set-Cookie`）。

2. **随后 GET admin 的请求**
   - 看 **Request Headers** 是否带有 `Cookie: WELINE_SESSID=...`。
   - 若 POST 302 有 Set-Cookie 但 GET 没有 Cookie：多为浏览器未接受 Cookie（检查 Cookie 的 Domain/Path/Secure/SameSite 与当前访问的 URL 是否一致，例如 HTTPS 需配 Secure）。

3. **若 Cookie 已带上仍跳回登录**（你当前就是这种情况）
   - 说明服务端认为未登录：**Session 未在多个 Worker 之间共享**。例如 POST 登录在 Worker 10444 写入 session，GET admin 被 10445 处理，10445 读不到该 session 就会重定向到登录。
   - **必须启动 Session Server**，让所有 Worker 通过 `WlsSharedStorage` 连接同一 Session 服务（默认端口 19970），session 才能跨 Worker 共享。
   - **检查**：在项目根执行 `ls var/session/wls.discovery`（或 Windows 下 `dir var\session\wls.discovery`）。若文件不存在，说明 Session Server 未运行。
   - **解决**：使用会启动完整 WLS 栈的命令（含 Session Server），例如：
     - `php bin/w server:start`（多 Worker 时会自动启动 Session Server）；
     - 或 `php bin/w cursor:supervisor:start`。
   - 启动后确认 `var/session/wls.discovery` 已生成，再执行 `php bin/w server:reload` 让现有 Worker 重新解析存储类型并连接 Session Server。
   - **若线上已经能看到 SessionServer:19970 的日志**（说明 Session Server 在跑），但登录仍跳回：多半是当前这批 Worker 在 Session Server 就绪前就启动了，还在用进程内/文件存储。在项目根执行 **`php bin/w server:reload`**，让 Worker 重启并重新连上 Session Server，再试登录。
   - 未启动 Session Server 时会降级为文件存储，多 Worker 下无法可靠共享 session，会出现“带 Cookie 仍跳回登录”。
   - **为何 Keep-Alive 解决不了？** 登录成功返回 302 后，浏览器会**新发一条 GET 请求**去访问 admin；这条请求可能走新连接或被调度到**另一个 Worker**。Keep-Alive 只能让同一条连接上的多次请求打到同一 Worker，无法保证“下一次请求”仍落在写 Session 的那个 Worker。只有 **Session 存在共享存储（Session Server）** 里，任意 Worker 都能读到，登录态才稳定。

4. **超管权限**
   - 若已登录但被当“无权限”重定向，见 [后台用户与超管权限检查](backend-user-role-check.md)。可用 `php bin/w user:assign-super-admin --username=admin` 为指定用户分配超管角色。

5. **重定向循环导致封禁**
   - 若未修复 Session 共享，会出现 admin → login → admin → … 的**无限重定向**，同一 IP 在短时间内产生大量请求，触发 WLS 的 **rate_limit**（请求频率过高）和 **SSL 握手失败**计数，导致该 IP 被临时封禁。
   - **根本解决**：按上面第 3 步修好 Session（Session Server + `server:reload`），重定向循环消失，请求量恢复正常，不再误封。
   - **临时恢复**：执行 `php bin/w server:security:unblock --clear-all` 或 `--ip=你的IP` 解封；必要时在 `server.attack_detector` 配置中把该 IP 加入 `ip_whitelist`。框架已把默认 rate_limit 从 100/60s 提高到 200/60s，减轻重定向循环时的误封概率。
