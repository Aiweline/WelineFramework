## 依赖注入问题

### 问题：无法实例化 `Weline\Framework\App\Env` - Access to non-public constructor

**错误信息**：
```
无法实例化该类：Weline\Framework\App\Env，错误：Access to non-public constructor of class Weline\Framework\App\Env

Call to undefined function Weline\Framework\UnitTest\Console\PhpUnit\popen() at 
app/code/Weline/Framework/Manager/ObjectManager.php:467
```

**触发场景**：
运行 `php bin/w setup:upgrade` 时报错

**根本原因**：
`Weline\Framework\App\Env` 是一个**单例类**，其构造函数是**私有的**（private 或 protected），不能通过依赖注入（DI）容器实例化。框架的 `ObjectManager` 尝试通过构造函数注入创建实例时会失败。

**错误代码示例**（修复前）：
```php
// ❌ 错误：尝试通过构造函数注入单例类
class SecretStoreService
{
    private Env $env;

    public function __construct(Env $env)  // ❌ 错误：Env 不能被注入
    {
        $this->env = $env;
        $this->initMasterKey();
    }

    private function initMasterKey(): void
    {
        $config = $this->env->getConfig();  // 使用注入的实例
        // ...
    }
}
```

**正确代码示例**（修复后）：
```php
// ✅ 正确：使用 getInstance() 静态方法获取单例
class SecretStoreService
{
    // 移除 $env 属性声明

    public function __construct()  // ✅ 正确：不注入 Env
    {
        $this->initMasterKey();
    }

    private function initMasterKey(): void
    {
        $config = Env::getInstance()->getConfig();  // ✅ 使用静态方法
        // ...
    }
}
```

**修复步骤**：
1. 移除类中的 `private Env $env;` 属性声明
2. 从构造函数参数中移除 `Env $env`
3. 将所有 `$this->env->` 调用改为 `Env::getInstance()->`
4. 运行 `php bin/w setup:upgrade` 验证修复

**验证方法**：
- [ ] ✅ 运行 `php bin/w setup:upgrade` 无错误
- [ ] ✅ 检查日志 `var/log/error.log` 无相关错误
- [ ] ✅ 相关功能测试通过

**相关文件**：
- `app/code/Weline/Ai/Service/SecretStoreService.php`（本次修复）
- 任何尝试注入 `Env` 的服务类

**防范措施**：
1. **识别单例类**：检查类是否有私有构造函数或 `getInstance()` 静态方法
2. **禁止注入单例**：单例类必须使用 `ClassName::getInstance()` 获取实例
3. **常见单例类**：
   - `Weline\Framework\App\Env` - 环境配置
   - `Weline\Framework\ObjectManager` - 对象管理器
   - `Weline\Framework\App\System\Config` - 系统配置（待确认）

**扩展阅读**：
- WelineFramework DI 容器文档
- Singleton Pattern 最佳实践

## 语法错误

### 问题：PHP 解析错误 - unexpected identifier "dev"

**错误信息**：
```
Parse error: syntax error, unexpected identifier "dev", expecting "function" in 
C:\Users\Administrator\Desktop\Weline\WelineFramework\app\code\Weline\Ai\Model\AiMobileNotification.php on line 156
```

**触发场景**：
运行 `php bin/w setup:upgrade` 时报错

**根本原因**：
在 PHPDoc 注释结束标记 `*/` 后，误添加了多余的文本 `dev`，导致 PHP 解析器认为这是一个标识符，但期望的是 `function` 关键字。

**错误代码示例**（修复前）：
```php
/**
 * 获取通知数据
 * 
 * @return array
 */dev  // ❌ 错误：注释结束后多余的文本
public function getNotificationData(): array
{
    // ...
}
```

**正确代码示例**（修复后）：
```php
/**
 * 获取通知数据
 * 
 * @return array
 */
public function getNotificationData(): array  // ✅ 正确：移除多余文本
{
    // ...
}
```

**修复步骤**：
1. 定位错误行（根据错误信息中的行号）
2. 检查该行附近的 PHPDoc 注释
3. 移除注释结束标记 `*/` 后的任何文本
4. 保存文件并运行 `php bin/w setup:upgrade` 验证

**验证方法**：
- [ ] ✅ 运行 `php bin/w setup:upgrade` 无语法错误
- [ ] ✅ PHP 语法检查通过：`php -l <file_path>`
- [ ] ✅ 模型升级成功显示 "已更新"

**相关文件**：
- `app/code/Weline/Ai/Model/AiMobileNotification.php`（本次修复，第 156 行）

**防范措施**：
1. **使用 IDE/编辑器**：现代 IDE（如 PHPStorm、VS Code）会高亮显示语法错误
2. **启用 Linter**：配置 PHP CS Fixer 或 PHPStan 自动检测语法问题
3. **提交前检查**：使用 `php -l` 命令检查语法
4. **Git Pre-commit Hook**：配置 pre-commit 钩子自动运行语法检查

**常见类似错误**：
- PHPDoc 注释未正确闭合
- 注释中使用了特殊字符未转义
- 多行注释嵌套（`/* /* */ */`）

## 模型设计问题

### 问题：模型类名与表名不匹配导致 ORM 无法找到表

**错误信息**：
```
table ai has no column named [field_name]
```
或者模型收集/查询时返回空结果。

**触发场景**：
- 创建了模型类 `AiModel`，但数据库表名是 `ai`
- ORM 期望表名是 `ai_model`（根据类名自动推导）

**根本原因**：
WelineFramework 的 ORM 从**模型类名自动推导表名**，遵循以下规则：
- `Queue` → `queue`（类名的小写形式）
- `AiModel` → `ai_model`（驼峰转下划线）
- 如果实际表名与推导的表名不匹配，ORM 无法找到正确的表

**错误做法**：
```php
// ❌ 错误：尝试通过声明属性覆盖表名
class AiModel extends Model
{
    protected string $_table = 'ai';  // ORM 可能无法正确识别此覆盖
    protected string $_id_field_name = 'id';
    
    public function _init(): void
    {
        $this->useMainDbMaster();  // 不需要的方法
    }
}
```

**正确做法（参考 `Weline_Queue`）**：

**方案 A：重命名模型类以匹配表名**
```php
// ✅ 正确：类名与表名匹配
class Ai extends Model  // 类名 Ai → 表名 ai
{
    public array $_unit_primary_keys = ['id'];
    public array $_index_sort_keys = ['id', 'supplier', 'model_code'];
    
    // 不需要 _init() 方法
    // 不需要声明 $_table 和 $_id_field_name
    
    public function install(ModelSetup $setup, Context $context): void
    {
        if ($setup->tableExist() === false) {
            $setup->createTable('AI模型表')  // 描述性文字，不是实际表名
                ->addColumn(self::fields_ID, TableInterface::column_type_INTEGER, null, 'primary key auto_increment', 'ID')
                // ... 其他字段
                ->create();
        }
    }
}
```

**方案 B：保持类名但表名遵循框架约定**
```php
// 如果类名是 AiModel，则表名应该是 ai_model
// 在数据库中创建/重命名表为 ai_model
```

**修复步骤**：
1. **确认表名**：检查数据库中的实际表名
2. **对标参考模块**：查看 `Weline_Queue` 等成熟模块的命名约定
3. **选择方案**：
   - 方案 A：重命名模型类（需要更新所有引用，工作量大）
   - 方案 B：重命名数据库表（相对简单，但需要数据迁移）
4. **清理不必要的代码**：
   - 移除 `_init()` 方法（大多数模型不需要）
   - 移除 `$_table` 和 `$_id_field_name` 属性声明（除非必要）
5. **验证修复**：运行 `php bin/w setup:upgrade` 和模型收集命令

**验证方法**：
- [ ] ✅ 模型类名与表名匹配框架约定
- [ ] ✅ 运行 `php bin/w ai:model:collect` 成功收集模型
- [ ] ✅ 查询模型数据能正常返回结果
- [ ] ✅ 模型 CRUD 操作正常工作

**相关文件**：
- `app/code/Weline/Ai/Model/AiModel.php`（问题源）
- `app/code/Weline/Queue/Model/Queue.php`（参考模块）

**防范措施**：
1. **对标现有模块**：开发新模块前，参考 `Weline_Queue` 等成熟模块的实现模式（Constitution XI.A）
2. **遵循命名约定**：确保模型类名与表名遵循框架的自动推导规则
3. **避免自创模式**：不要凭想象或外部框架经验实现，必须有框架内实际案例支撑
4. **最小化覆盖**：除非绝对必要，不要声明 `$_table` 等属性去覆盖框架默认行为
5. **文档化决策**：如果必须使用非常规命名，在代码注释和 README 中说明原因

**扩展阅读**：
- Constitution XI.A: 对标现有模块学习规范
