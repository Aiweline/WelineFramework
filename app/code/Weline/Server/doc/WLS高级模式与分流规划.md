# WLS 高级模式与分流规划

本文档为 **规划文档**，描述 WLS（Weline Server）在分流、发布模式、Worker 标记与可配置负载均衡上的目标设计，以及推荐的部署模式与后续实现计划。

---

## 2. 推荐发布模式（目标）

规划中的"最推荐发布模式"与可选模式如下，便于在文档和配置中统一推荐。

| 模式 | 名称 | 适用场景 | 说明 |
|------|------|----------|------|
| **标准模式（推荐）** | Dispatcher + 多 Worker | 生产/预发，单口对外、多进程扛量 | 单端口（80/443）→ Dispatcher → 多 Worker 内网端口，均衡策略与权重可配置 |
| **单 Worker 直连** | 无 Dispatcher | 开发/调试、低流量 | `-c 1` 或 `--no-dispatcher`，单进程直连端口，无分流 |
| **多端口直连** | 无 Dispatcher，多 Worker | 需 Nginx 等自建 upstream | `-c 4 --no-dispatcher`，每 Worker 一端口，由 Nginx 做均衡 |
| **高级模式（规划）** | 带 Worker 标记 + 可配置分流 | 灰度、金丝雀、测试池 | Worker 带 tag（如 stable/canary/test），按策略/权重/请求头等分流 |
| **Git Tag 渐进式发布（最高级）** | 版本标记 + API 动态规则 | 生产大版本升级、A/B 测试 | Worker 带 Git 版本标记，Dispatcher API 动态调比例，支持渐进式发布与秒级回滚 |

文档中应明确：**默认推荐为「标准模式」**（多 Worker + Dispatcher），并说明何时选用单 Worker 或多端口直连。

## 3. Worker 标记与流量分组（规划）

### 3.1 设计目标

- 为每个 Worker（或每组 Worker）赋予**标记**（tag），例如：`stable`、`canary`、`test`，或**版本号**（如 `v2.1.0`）。
- 均衡层可根据**标记 + 配置**决定将请求发往哪些 Worker，实现：
  - 金丝雀/灰度：仅部分流量到 `canary` 或新版本 Worker。
  - 测试 Worker 池：仅带特定请求头或 Cookie 的请求到 `test` Worker。
  - 默认全量：仅 `stable` 或未标记 Worker 参与默认均衡。

### 3.2 Worker 标记方式（候选）

| 方式 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| **启动参数** | `server:start --tag=v2.1.0` 或 env 为每个 Worker 指定 tag | 简单、与进程一一对应、可对应 Git Tag | 需扩展 Start 与实例信息 |
| **端口段映射** | 约定某段端口为 canary，某段为 test | 不改 Worker 进程参数 | 配置与端口规划耦合 |
| **实例级 + 端口列表** | 同一实例内多组端口，每组一个 tag | 灵活，易与现有 Dispatcher 端口列表兼容 | 需在 Dispatcher 侧解析"端口 → tag" |

**推荐方向**：支持 `server:start --tag=xxx` 启动参数，Worker 启动时带版本标识（可对应 Git Tag），Dispatcher 维护"端口 → tag"映射。

### 3.3 流量规则（规划）

- **默认规则**：仅向 `stable`（或 default / 未标记）Worker 分发；策略与权重仅作用于该集合。
- **金丝雀/版本规则**：按比例（如 5%）或按请求头（如 `X-Version: v2.1.0`）将请求分到指定版本 Worker。
- **测试规则**：仅当请求带特定头（如 `X-Test-Worker: 1`）或 Cookie 时，将请求分到 `test` Worker，且可不计入主池权重。

实现时可在 LoadBalancer 或 Dispatcher 上层增加"规则引擎"（先匹配 test/canary/版本，再走通用均衡）。

## 4. 可配置流量分发（规划）

### 4.1 均衡策略与权重（从 env 读取）

目标：**均衡策略与权重由配置决定，而非写死。**

- **策略**：与 `LoadBalancer::STRATEGY_*` 对齐，在 env 中提供配置项，例如：
  - `server.load_balance.strategy`：`round_robin` | `weighted_round_robin` | `least_connections`
- **权重**：在 `server.load_balance.weights` 中按端口或按 Worker 索引配置，例如：
  - `weights: { "10443": 2, "10444": 1, "10445": 1 }` 或 `weights: [2, 1, 1]`
- **超时等**：可选 `connect_timeout`、`read_timeout` 等，与现有 LoadBalancer 参数对齐。

Start 在启动 Dispatcher 时从 env 读取上述配置，通过**新参数或配置文件**传给 Dispatcher（避免仅依赖命令行长度受限的参数字符串）。

### 4.2 Dispatcher API 动态规则（规划）

目标：**支持 HTTP API 动态调整分流规则，无需重启 Dispatcher。**

- `POST /api/dispatcher/rule`：设置 tag 的流量比例，如 `{ "tag": "v2.1.0", "ratio": 0.05 }`
- `GET /api/dispatcher/stats`：查询各 tag 的流量统计
- `DELETE /api/dispatcher/rule/{tag}`：移除某 tag 的规则

## 5. 高级发布模式（规划）

### 5.1 模式一览

| 模式 | 说明 | 依赖能力 |
|------|------|----------|
| **标准均衡** | 单口入口，多 Worker 轮询/加权/最少连接 | 已有；需策略/权重可配置 |
| **金丝雀发布** | 少量流量到新版本（canary Worker） | Worker 标记 + 比例或请求头分流 |
| **测试 Worker 池** | 仅测试流量到带 test 标记的 Worker | Worker 标记 + 请求头/Cookie 规则 |
| **蓝绿式** | 两套 Worker 池，通过配置或开关切换主池 | 可视为 stable/canary 的 0% vs 100% 特例 |
| **Git Tag 渐进式发布** | 按 Git 版本标记 Worker，API 动态调比例 | Worker 版本标记 + Dispatcher API |

### 5.2 最高级发布模式（目标形态）—— Git Tag 渐进式发布

"最高级"形态可归纳为：**Git Tag 版本标记 + Worker 版本分组 + API 动态规则 + 渐进式切换**。

#### 核心流程

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        Git Tag 渐进式发布流程                                │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   ① 发布新版本                                                              │
│      git tag v2.1.0 && git push --tags                                     │
│      └─ 触发部署，生成 v2.1.0 代码目录或容器                                  │
│                              ↓                                              │
│   ② 启动带版本标记的 Worker                                                  │
│      php bin/w server:start --tag=v2.1.0 --count=1                         │
│      └─ 新 Worker 加入 Dispatcher，tag=v2.1.0，权重=0（待激活）               │
│                              ↓                                              │
│   ③ 后台/API 配置引流规则                                                    │
│      POST /api/dispatcher/rule { "tag": "v2.1.0", "ratio": 0.05 }          │
│      └─ Dispatcher 热更新：5% 流量 → v2.1.0 Worker                          │
│                              ↓                                              │
│   ④ 监控与验证                                                              │
│      └─ 观察 v2.1.0 的错误率、延迟、业务指标                                  │
│      └─ 若异常：ratio=0 或回滚；若正常：继续                                  │
│                              ↓                                              │
│   ⑤ 逐步放量                                                                │
│      ratio: 5% → 20% → 50% → 80% → 100%                                    │
│      └─ 每步验证无误后再增加                                                 │
│                              ↓                                              │
│   ⑥ 全量切换 & 下线旧版本                                                    │
│      └─ v2.1.0 达到 100%，旧版本 Worker（如 v2.0.0）停止并移除                │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

#### 关键能力

| 能力 | 说明 |
|------|------|
| **Worker 带版本标记** | 启动时通过 `--tag=v2.1.0` 或 env 配置为 Worker 赋予版本标识，Dispatcher 可识别 |
| **Dispatcher API 热更新规则** | 提供 HTTP API（如 `POST /api/dispatcher/rule`），动态调整各 tag 的流量比例，无需重启 |
| **比例 + 请求头双模式** | 支持按比例（ratio）或按请求头（如 `X-Version: v2.1.0`）精准引流 |
| **多版本共存** | 同一 Dispatcher 下可同时运行 v2.0.0、v2.1.0、v2.2.0-beta 等多个版本的 Worker |
| **可观测** | 按 tag 统计 QPS、错误率、平均延迟，便于灰度决策与快速回滚 |

#### 与常规模式的区别

| 常规 Dispatcher 模式 | Git Tag 渐进式发布模式 |
|---------------------|----------------------|
| 所有 Worker 运行相同代码版本 | 多版本 Worker 共存 |
| 部署 = 全量重启 | 部署 = 新增 Worker，逐步切流量 |
| 出问题需回滚整个发布 | 出问题只需 ratio=0，秒级回滚 |
| 无版本维度统计 | 按版本统计，便于对比 |

#### 适用场景

- **生产环境大版本升级**：先小流量验证，降低全量发布风险。
- **A/B 测试**：不同版本跑不同功能，按比例分流。
- **多租户差异化**：不同租户可通过请求头指向不同版本 Worker。
- **快速回滚**：新版本出问题，API 一键将其 ratio 设为 0，流量立即回到旧版本。

## 6. 实现阶段建议

### 阶段 1：可配置均衡（无标记）

- 在 `env.php` / `env.sample.php` 中增加 `server.load_balance`（strategy、weights、超时等）。
- `Start.php` 读取该配置，通过**配置文件或扩展参数**传给 Dispatcher；Dispatcher 入口解析并传入 `LoadBalancer`。
- 保证 `round_robin`、`weighted_round_robin`、`least_connections` 均从配置生效，并更新《WLS模式部署指南》说明推荐发布模式与配置示例。

### 阶段 2：Worker 版本标记

- 支持 `server:start --tag=v2.1.0` 为 Worker 赋予版本标识（可与 Git Tag 对应）。
- Start 将 `worker_ports` 与 `worker_tags` 一并写入实例文件或传给 Dispatcher；Dispatcher/LoadBalancer 维护"端口 → tag"映射。
- 默认行为：未指定 tag 的 Worker 视为 `stable`（或 `default`），参与默认流量池。

### 阶段 3：Dispatcher API 动态规则

- 实现 Dispatcher 内部 API 端点（如 `POST /api/dispatcher/rule`），支持动态设置 tag 的流量比例。
- 规则热更新：Dispatcher 接收 API 后立即生效，无需重启。
- 支持按请求头精准引流：如 `X-Version: v2.1.0` 则走指定版本 Worker。

### 阶段 4：可观测与统计

- 在 Dispatcher 或独立统计模块中增加按 tag 的计数（QPS、错误率、平均延迟）。
- 便于灰度决策：新版本指标异常时快速回滚。
- 可选：提供 `GET /api/dispatcher/stats` 查询各版本流量与状态。

### 阶段 5：文档与运维

- 在 `doc/` 下保留本文档为规划总览，并新增或更新：
  - **推荐发布模式**：明确标准模式为默认推荐，何时使用单 Worker / 多端口直连。
  - **高级模式使用说明**：Worker 标记、env 配置项、API 调用示例、请求头约定。
  - **Git Tag 渐进式发布指南**：完整流程与最佳实践。
- 在《WLS模式部署指南》中增加"分流与负载均衡"小节，指向本规划与 LoadBalancer 策略说明。

## 7. 配置示例（目标形态）

以下为规划中的配置形态，**非当前实现**，仅作设计参考。

```php
// app/etc/env.php 示例（规划）

'server' => [
    'host' => '127.0.0.1',
    'port' => 443,
    'worker_count' => 4,
    'mode' => 'io',
    'https' => true,

    // 负载均衡（高级模式）
    'load_balance' => [
        'strategy' => 'weighted_round_robin',  // round_robin | weighted_round_robin | least_connections
        'weights' => [10443 => 2, 10444 => 1, 10445 => 1, 10446 => 1],
        'connect_timeout' => 1,
        'read_timeout' => 30,
    ],

    // Worker 标记（可选，用于金丝雀/测试池/版本分流）
    'worker_tags' => [
        10443 => 'v2.0.0',     // 旧版本
        10444 => 'v2.0.0',     // 旧版本
        10445 => 'v2.1.0',     // 新版本（金丝雀）
        10446 => 'test',       // 仅测试流量
    ],

    // 分流规则（可选，也可通过 API 动态设置）
    'traffic_rules' => [
        'v2.1.0' => [
            'ratio' => 0.05,              // 5% 流量到新版本
            'header' => 'X-Version',      // 或按请求头精准引流
        ],
        'test' => [
            'header' => 'X-Test-Worker',  // 带此头则走 test Worker
        ],
    ],
],
```

### 9.2 后台管理界面（规划）

#### 两种访问方式

| 方式 | 路由 | 访问限制 | 说明 |
|------|------|----------|------|
| **WLS 内置后台** | `/_wls/` | 仅本地（127.0.0.1） | 轻量、快速，不依赖框架，账号配置在 env |
| **Weline 后台嵌入** | `/admin/server/version` | 走框架 ACL 权限 | 远程可访问，需登录后台，权限控制 |

#### WLS 内置后台（本地访问）

- **路由**：`https://127.0.0.1:443/_wls/`
- **访问限制**：仅允许 `127.0.0.1` 或 `::1` 访问，其他 IP 返回 403
- **账号配置**：在 `env.php` 中配置

```php
// env.php
'server' => [
    // ... 其他配置 ...
    
    // WLS 内置管理后台（仅本地访问）
    'admin' => [
        'enabled' => true,
        'username' => 'wls_admin',
        'password' => 'your_secure_password',  // 建议使用强密码
        // 'password_hash' => '...',           // 或使用 password_hash() 生成的哈希值
    ],
],
```

#### Weline 后台嵌入（远程访问）

- **路由**：`/admin/server/version`
- **权限**：`Weline_Server::version_manage`
- **特点**：
  - 走框架 ACL 权限系统，需后台登录
  - 支持远程访问（公网管理）
  - 与 Server 模块其他功能（SSL 证书、服务器管理）统一入口

#### 安全设计

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           访问控制设计                                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  本地开发/运维：                                                             │
│     ssh 到服务器 → 访问 https://127.0.0.1:443/_wls/ → 内置后台              │

│     /_wls/* 请求 → Dispatcher 内部处理（不转发给 Worker）                    │
```

### 9.3 预览模式设计

预览模式允许**仅标记用户**访问新版本 Worker，用于内部测试。

#### 标记用户的方式

| 方式 | 说明 | 示例 |
|------|------|------|
| **Cookie** | 用户浏览器带特定 Cookie | `wls_preview=v2.1.0` |
| **请求头** | API 请求带特定 Header | `X-WLS-Preview: v2.1.0` |
| **IP 白名单** | 指定 IP 段可访问预览 | `192.168.1.0/24` |
| **用户 ID** | 登录用户的 ID 在白名单中 | `user_ids: [1, 2, 3]` |

#### 预览入口生成

```bash
# 部署预览版本
php bin/w server:version:preview v2.1.0

# 输出：
# ✓ 预览 Worker 已启动
# 预览链接：https://example.com/?wls_preview=v2.1.0
# 预览 Cookie：wls_preview=v2.1.0; Path=/; HttpOnly
# 
# 在后台「版本管理」可配置预览用户白名单
```

#### 预览配置示例

```php
// env.php 或后台配置
'server' => [
    'preview' => [
        'v2.1.0' => [
            'cookie' => 'wls_preview',           // Cookie 名
            'header' => 'X-WLS-Preview',         // Header 名

    ],
```

### 9.4 Master/Dispatcher 控制协议

后台与命令行通过 Master/Dispatcher 进程控制版本分流。

#### 架构说明

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           内置后台请求流向                                    │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  普通业务请求：                                                              │
│     Client → Dispatcher:443 → LoadBalancer → Worker:10443 → 业务响应        │
│                                                                             │
│  管理后台请求（/_wls/*）：                                                    │
│     Client → Dispatcher:443 → 检测 /_wls/ 前缀                              │
│                            → 检测来源 IP（仅 127.0.0.1）                     │
│                            → 检测 Basic Auth（env 配置的账号）               │
│                            → Dispatcher 内部处理（不转发 Worker）            │
│                            → 返回管理界面/API 响应                           │
│                                                                             │
│  优势：                                                                      │
│     ✓ 管理接口与业务隔离，Worker 全挂也能管理                                 │
│     ✓ Dispatcher 直接处理，延迟极低                                          │
│     ✓ 仅本地访问，安全                                                       │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

#### 通信方式

| 方式 | 说明 | 适用场景 |
|------|------|----------|
| **Dispatcher 内置 HTTP** | `/_wls/*` 路由，Dispatcher 直接处理 | 内置后台，最推荐 |
| **Unix Socket** | Master 监听本地 Socket | Linux/Mac，CLI 通信 |
| **TCP 端口** | Master 监听管理端口 | Windows，或远程管理 |
| **文件信号** | 写入配置文件，Master 轮询读取 | 简单场景，兼容性好 |

#### 内置后台 API（Dispatcher 处理）

```
# 仅 127.0.0.1 可访问，需 Basic Auth

# 版本管理
GET    /_wls/api/version/list              # 列出所有版本
POST   /_wls/api/version/deploy            # 部署新版本
POST   /_wls/api/version/ratio             # 调整流量占比
POST   /_wls/api/version/preview           # 设置预览规则
POST   /_wls/api/version/promote           # 提升为主版本
POST   /_wls/api/version/rollback          # 回滚
DELETE /_wls/api/version/{tag}             # 停止版本

# 状态监控
GET    /_wls/api/status                    # 整体状态
GET    /_wls/api/stats                     # 流量统计
GET    /_wls/api/workers                   # Worker 列表
```

## 10. 实现优先级建议

| 优先级 | 功能 | 说明 |
|--------|------|------|
| P0 | Worker 版本标记 + 基础分流 | `--tag` 参数、Dispatcher 按 tag 选 Worker |
| P1 | CLI 命令（deploy/ratio/list） | 命令行控制版本部署与流量 |
| P1 | Master 控制协议 | Master 接收指令、热更新分流规则 |
| P2 | WLS 内置后台（本地） | Dispatcher 处理 `/_wls/*`，仅本地访问，env 配置账号 |
| P2 | Weline 后台嵌入（远程） | `/admin/server/version`，走 ACL 权限，远程管理 |
| P2 | 预览模式 | Cookie/Header/IP 标记用户访问预览 Worker |
| P3 | 按版本统计 | QPS、错误率、延迟，便于灰度决策 |
| P3 | 自动化发布 | Git Hook 触发部署、CI/CD 集成 |

### 内置后台与框架后台的分工

| 场景 | 推荐方式 | 原因 |
|------|----------|------|
| 本地开发调试 | WLS 内置后台 `/_wls/` | 轻量快速，无需启动框架 |
| 服务器运维（SSH） | WLS 内置后台 `/_wls/` | 即使框架挂了也能管理 |
| 远程日常管理 | Weline 后台 `/admin/server/version` | 权限控制、操作日志、多用户 |
| 紧急故障处理 | CLI 命令 | 最快速、最可靠 |

## 11. 架构级建议

以下为架构层面的设计与实现建议，便于后续开发时统一原则、少走弯路。

### 11.1 状态与配置：单一事实源（Single Source of Truth）

| 建议 | 说明 |
|------|------|
| **流量规则由 Master 持有** | 比例、预览规则、版本→端口映射等，以 Master 内存/持久化为准；Dispatcher 从 Master 拉取或由 Master 推送，不单独维护一份。 |
| **启动参数与运行时解耦** | `server:start --tag=xxx` 只负责「把 Worker 拉起来并注册到 Master」；当前流量占比、是否预览等由 Master 的规则决定，避免「启动时写死、后续改不动」。 |
| **持久化可选、可恢复** | 规则可落盘（如 `var/server/rules.json`），Master 重启后能恢复上次的 ratio/预览配置；CLI/后台改规则时同时更新内存 + 持久化。 |

**原则**：谁「做决策」谁「持有一份权威状态」；其他人只读或通过 API 更新，避免多处写导致不一致。

### 11.2 职责边界：Dispatcher 与 Master

| 组件 | 建议职责 | 不建议 |
|------|----------|--------|
| **Dispatcher** | 按「当前规则」选 Worker、转发请求、统计计数；必要时拉取/接收规则更新。 | 不负责「部署/停止 Worker」「修改规则」；不持久化规则。 |
| **Master** | 进程生命周期（启动/停止 Worker）、规则管理（比例、预览）、健康检查、汇总统计、对外提供控制 API。 | 不解析 HTTP 请求体、不直接做流量转发。 |
| **CLI / 后台** | 调用 Master API 或写约定文件，触发「部署/改比例/回滚」等动作。 | 不直接操作 Worker 进程（由 Master 统一 fork/kill）。 |

**原则**：Dispatcher = 数据面（转发 + 统计）；Master = 控制面（生命周期 + 规则）；管理层（CLI/后台）= 调用控制面。

### 11.3 故障与恢复

| 场景 | 建议行为 |
|------|----------|
| **Master 挂掉** | Worker 和 Dispatcher 继续按「挂掉前的规则」转发；恢复时：先启动 Master，Master 从持久化加载规则，再通知 Dispatcher 或等其拉取，不要求先停 Worker。 |
| **Dispatcher 挂掉** | 入口不可用，需外部（如 Nginx）或运维重启 Dispatcher；Master 不自动拉起重启 Dispatcher（或可选「看门狗」）。 |
| **部分 Worker 挂掉** | Master 健康检查发现后重启该 Worker；Dispatcher 转发失败时按现有 failover 换下一个 Worker；同一版本内多 Worker 互为备份。 |
| **规则持久化损坏** | 降级为「仅 stable、按端口轮询」或从备份/默认配置恢复；启动时校验规则文件，异常时打日志并回退到安全默认值。 |

**原则**：控制面（Master）故障时，数据面尽量保持「最后已知良好状态」；数据面故障时，控制面仍可操作（如改比例、停版本），便于恢复。

### 11.4 安全

| 建议 | 说明 |
|------|------|
| **内置后台仅本机** | `/_wls/` 仅允许 127.0.0.1/::1；若从其他 IP 访问，直接 403，不暴露登录页。 |
| **Master 管理通道隔离** | Master 的 Socket/管理端口只监听 127.0.0.1，避免从外网直连控制面。 |
| **敏感配置不落日志** | env 中的 admin 密码、API 密钥等不要打印到普通日志；调试用脱敏或仅 debug 级别。 |
| **后台/API 鉴权统一** | 内置后台用 env 的账号 + Basic Auth；Weline 后台走框架 ACL；Master API 若对外（仅本机），可用本地 token 或 Unix Socket 鉴权。 |

### 11.5 可观测性

| 建议 | 说明 |
|------|------|
| **按版本 + 按端口** | 统计 QPS、错误数、延迟时，同时带上 version tag 和 port，便于区分「哪个版本、哪台 Worker」有问题。 |
| **滑动窗口** | 用 1 分钟/5 分钟滑动窗口做实时指标，避免只存总量、难以看趋势。 |
| **关键操作打点** | 部署版本、改比例、回滚、停止版本等，打一条结构化日志（时间、操作、参数、操作者若可知），便于审计与排障。 |
| **健康接口** | 提供 `/_wls/api/health` 或 `GET /master/health`，仅返回 200/503，便于负载均衡或监控探测。 |

### 11.6 会话与状态一致性（跨版本）

| 建议 | 说明 |
|------|------|
| **Session 存储外置** | 若用 Redis/DB 存 Session，同一用户在不同版本 Worker 间切换时，Session 一致；若用文件/内存，换 Worker 可能丢 Session，建议 Session 不依赖单机。 |
| **灰度时的兼容性** | 新版本与旧版本在协议、Cookie、关键接口上保持兼容，避免 5% 流量到新版本时因接口不兼容大量报错。 |
| **有状态迁移** | 若有「粘滞」需求（同一用户固定到某 Worker），可在 Dispatcher 按 Cookie/Header 做一致性哈希，而不是纯随机比例。 |

### 11.7 配置与多入口一致性

| 建议 | 说明 |
|------|------|
| **env 与运行时分离** | env 描述「默认/初始」配置；运行时比例、预览规则以 Master 为准，避免改 env 不生效或误以为改 env 即改流量。 |
| **CLI 与后台同源** | CLI 和后台都调同一套 Master API，保证「改比例」等行为一致；避免 CLI 写文件、后台调 API 两套路径逻辑不同。 |
| **版本列表唯一** | 「当前有哪些版本、各多少 Worker、端口」以 Master 为准；CLI/后台只做展示与触发，不维护独立列表。 |

### 11.8 扩展与多实例（可选）

| 建议 | 说明 |
|------|------|
| **单机优先** | 先做好单机多版本、单 Dispatcher + 单 Master，再考虑多实例（多台机或多实例名）。 |
| **多实例时** | 每实例独立 Master + Dispatcher + Worker 组；版本发布可做成「按实例分批」（先实例 A 灰度，再实例 B），避免一开始就跨机协调。 |
| **共享存储** | 若多实例要「同一份规则」，再考虑共享存储（如 Redis）或领导选举；首版不必做，单实例文件即可。 |

---

以上建议与当前规划文档中的流程图、命令设计、API 设计一致，实现时可按优先级采纳（如先单事实源 + 职责边界 + 故障恢复，再可观测与安全加固）。

## 12. 总结

通过实现 **Worker 版本标记 + Master 控制 + CLI/后台管理**，WLS 可原生支持：

| 能力 | 说明 |
|------|------|
| **Git Tag 版本发布** | `server:version:deploy v2.1.0 --count=2`，一键部署指定版本到 N 个 Worker |
| **渐进式流量切换** | CLI 或后台调整占比，5% → 20% → 100%，Master 热更新分流规则 |
| **预览模式** | 发布预览 Worker，仅标记用户（Cookie/Header/IP）可访问，内部先行验证 |
| **后台可视化管理** | 版本列表、流量滑块、实时监控，无需命令行即可操作 |
| **秒级回滚** | 新版本出问题，`ratio=0` 或 `rollback`，流量立即回到旧版本 |
| **多版本共存** | 同时运行 v2.0.0、v2.1.0、v2.2.0-rc，便于 A/B 测试、多租户差异化 |

### 典型使用流程

```
开发完成 → git tag v2.1.0 → 命令部署预览 → 内部验证
    ↓
后台调整 5% 流量 → 监控无异常 → 逐步放量 20% → 50% → 100%
    ↓
全量切换 → 停止旧版本 Worker → 发布完成
```

### 与业界方案对比

| 方案 | 复杂度 | 适用场景 |
|------|--------|----------|
| K8s + Istio | 高 | 微服务、大型集群 |
| Nginx + 手动配置 | 中 | 需手动改配置、重载 |
| **WLS Git Tag 发布** | **低** | **PHP 单体应用，轻量、原生支持** |

这套能力是 WLS 的「**最高级发布模式**」目标形态，后续开发时按优先级逐步实现。
