# Weline_Payment 支付总抽象层需求

> 本文是 `Weline_Payment` 新支付架构的需求与实施计划源文档。后续新增讨论优先追加到本文，不再在会话中反复重写完整计划。

## 1. 目标

- `Weline_Payment` 是唯一支付内核，负责支付方式发现、配置、可用性、支付对象、结账会话、支付意图、分摊、交易、退款、账本、回调、锁、幂等、继续支付、自动关闭、支付路由、跳转/扫码恢复、保存支付工具、支付条款同意、异常入账、线下支付、3DS/SCA、国际化、风控、分账、对账、财务锁账、数据归档和支付统计驾驶舱。
- `WeShop_Payment`、`Weline_Order`、商城订单、应用商城、A2A 等只接入 `Weline_Payment`，不再自建支付核心。
- 不兼容旧支付接口，现有支付方式需要全部迁移到新 Provider 接口。
- 支付开发者提供一个 Provider 接口实现类、一个 checkout phtml、一个 backend 配置 phtml；特殊流程可自建控制器，但最终状态必须回写 `Weline_Payment`。
- 支付对象不是订单专属。`Weline_Payment` 提供默认 `payment_default`；正式业务建议各自实现 PayableResolver，例如 `weline_order`、`weshop_order`、`app_market_order`、`a2a_trade_order`、`a2a_escrow_case`。

## 3. 核心接口与扩展点

- Provider 扩展点：`extends/module/Weline_Payment/PaymentProvider`，由 `Weline_Payment` 扫描并按 `method_code` 注册。
- Payable 扩展点：`extends/module/Weline_Payment/PayableResolver`，声明 `payable_type`、金额快照、状态映射、资源锁释放、权益发放/撤销、可取消/可退款/可恢复规则、业务类型 tags。
- RiskRule 扩展点：`extends/module/Weline_Payment/RiskRule`，用于国家、币种、客户组、订单风险、活动风控、支付频控、KYC 状态、业务类型限制等统一判断。
- 配置模板扩展点：支付模块通过 `extends/module/Weline_SystemConfig/Config/{area}/{code}.phtml` 提供后台配置模板；scope 选择、继承显示、普通 key/value 保存、校验和缓存失效由 `Weline_SystemConfig` 负责。
- Provider 使用接口类，不使用抽象类；必须实现支付、授权、捕获、取消、退款、查询、回调校验、能力声明、货币/国家/语言支持、跳转/扫码恢复、配置元数据、展示元数据、错误归一化。
- Provider 能力是硬上限；backend 配置只能收窄能力，不能扩大 Provider 未声明支持的币种、国家、退款、分期、授权、部分捕获、3DS/SCA、线下确认、保存支付工具等能力。
- Provider 必须声明 `provider_api_version` 和 `webhook_schema_version`，历史回调按记录版本解析。
- Provider 统一使用 `Weline_Payment` 提供的 HTTP 策略：timeout、retry、circuit breaker、rate limit、熔断恢复、低速连接、错误映射、日志脱敏。

### 3.1 ProviderInterface 必实现函数

每个支付方式只需要实现一个核心 Provider 类，继承接口类，不继承抽象类。`Weline_Payment` 通过扩展点扫描 Provider 后自动完成注册、配置渲染、checkout 可用性计算、支付调用、退款调用、回调处理和后台测试。

Provider 必须实现以下函数或等价接口：

| 函数 | 责任 |
| --- | --- |
| `getCode()` | 返回稳定 `method_code`，必须和 checkout phtml、配置 phtml、EAV、后台配置一致。 |
| `getProviderCode()` | 返回 Provider code，用于多通道、多 merchant account、路由和统计。 |
| `getCapabilities()` | 声明支持国家、币种、默认币种、语言、最小/最大金额、授权/捕获/退款/分期/3DS/保存支付工具等硬能力。 |
| `getDisplayMetadata()` | 返回 logo、icon、标题、描述、前台展示文案、不可用说明模板。 |
| `getConfigSchema()` | 返回从 `Weline_SystemConfig` 配置模板 `<w:config:field>` / `<w:config:adapter>` 解析出的 schema，允许 Provider 补充动态校验。 |
| `getDynamicFormSchema(AvailabilityRequest $request)` | 返回 checkout 需要的本地字段，例如税号、银行、UPI、手机号、票据字段。 |
| `checkAvailability(AvailabilityRequest $request)` | 返回是否可用、禁用原因、排序权重、需要条款、surcharge 和动态字段。 |
| `createPayment(PaymentRequest $request)` | 创建支付，返回跳转、二维码、票据、Provider reference 或已完成结果。 |
| `resumePayment(ResumeRequest $request)` | 继续未完成支付，恢复跳转、二维码、票据或返回已终态。 |
| `authorize(AuthorizeRequest $request)` | 授权支付；不支持时必须明确返回 unsupported，不允许静默改写为 capture。 |
| `capture(CaptureRequest $request)` | 捕获授权，支持全额、部分、多次部分时必须声明能力和余额。 |
| `void(VoidRequest $request)` | 取消授权或未捕获交易，失败时给出是否可 retry。 |
| `refund(RefundRequest $request)` | 原路退款，支持全额、部分、多次部分、手续费退款规则和替代退款原因。 |
| `query(QueryRequest $request)` | 查询 Provider 侧支付、退款、授权、争议或结算状态，用于补偿。 |
| `verifyCallback(CallbackRequest $request)` | 校验签名、时间窗、secret 版本、endpoint、重放和来源。 |
| `parseCallback(CallbackRequest $request)` | 归一化 Provider 事件，返回目标 intent/transaction/refund/dispute 和建议状态转移。 |
| `testConnection(TestConnectionRequest $request)` | backend 连接测试，输出脱敏结果和可行动错误。 |
| `normalizeError(Throwable|array $error)` | 统一错误 code、message、retryable、user_visible、provider_error_code。 |

### 3.2 PayableResolverInterface 必实现函数

PayableResolver 用于把支付内核和业务对象解耦。`Weline_Payment` 不直接修改订单、应用市场订单、A2A 托管单或其他业务对象状态，只读取快照、执行支付状态机、写支付账本，并通过 Resolver 或事件通知业务模块处理业务状态。

PayableResolver 必须实现以下函数或等价接口：

| 函数 | 责任 |
| --- | --- |
| `getPayableType()` | 返回稳定 `payable_type`，例如 `weline_order`、`weshop_order`、`app_market_order`。 |
| `resolve(string $payableId)` | 解析业务对象，不存在、无权限、已归档时必须返回明确错误。 |
| `snapshot(PayableContext $context)` | 返回金额、items、税费、运费、折扣、国家、语言、时区、owner、payer、版本。 |
| `canPay(PayableSnapshot $snapshot, Actor $actor)` | 判断当前 actor 是否可支付。默认由 Weline_Payment 配置放行，Resolver 可进一步收窄。 |
| `canCancel(PayableSnapshot $snapshot)` | 判断支付窗口过期或人工关闭时业务对象是否可取消。 |
| `canRefund(RefundRequest $request)` | 判断业务层是否允许退款、按 item 退款、运费/税费/手续费退款。 |
| `onPaid(PaymentIntent $intent)` | 支付完成后执行业务确认、发货前置、权益发放或等待风控。 |
| `onPartiallyPaid(PaymentIntent $intent)` | 部分支付、定金、账期付款进入业务侧中间状态。 |
| `onRefunded(RefundResult $result)` | 退款后撤销权益、回滚库存/服务、触发红冲或售后状态。 |
| `onExpired(PaymentIntent $intent)` | 支付过期后释放业务资源、取消订单或保持可重建支付。 |
| `onRiskReview(PaymentIntent $intent)` | 风控或异常入账时冻结业务发放，等待人工裁决。 |
| `releaseResources(PaymentIntent $intent, string $reason)` | 关闭 intent 时释放库存、优惠、资产预占和临时输入。 |
| `getPayerPolicy(PayableSnapshot $snapshot)` | 返回允许支付的 actor 类型、客户组、企业账户、A2A 参与方策略。 |
| `getBusinessTags(PayableSnapshot $snapshot)` | 返回订阅、预售、虚拟商品、跨境、高风险品类等路由 tags。 |
| `getLineItems(PayableSnapshot $snapshot)` | 提供退款、对账、税费和 receipt 需要的业务明细。 |

### 3.3 模块所有权边界

- `Weline_Payment` 拥有 payment method、checkout session、intent、allocation、transaction、refund、ledger、callback、settlement、dispute、dashboard、配置发布和事件可靠性。
- Payable 所属模块拥有业务对象状态、库存/权益、发票、售后、发货、应用授权、A2A 托管释放和业务取消。
- Provider 模块只能通过 ProviderInterface 返回结果或通过事件参与处理，不能直接改 `payment_intent`、业务订单状态或 ledger。
- 业务模块不得绕过 `Weline_Payment` 直接调用 Provider 完成支付或退款；特殊 controller 只能作为 Provider 授权页、回调页或辅助动作入口，最终仍回到 Payment 状态机。

### 3.4 SystemConfig 配置模板契约

支付配置不再定义旧配置范围接口、支付自有 scope selector 或支付自有继承链。所有支付模块只提供 `Weline_SystemConfig` 配置模板，路径固定为：

```text
app/code/{Vendor}/{Module}/extends/module/Weline_SystemConfig/Config/{area}/{code}.phtml
```

模板通过 `@meta.*`、`@config.*`、`<w:config:group>`、`<w:config:field>` 和 `<w:config:adapter>` 声明后台配置页。`Weline_SystemConfig` 负责从 Extends 注册表读取模板、解析字段、渲染配置树、处理 scope 选择、继承来源、普通 key/value 保存、校验、审计和缓存失效；支付模块模板不实现保存接口，不直接写 `system_config`，也不处理 scope 写入。

普通 key/value 的字段身份由 SystemConfig 维护为 `module + area + key + scope + locale`。`Weline_Payment` 只接收 SystemConfig 已解析出的单一 `scope` 字符串，不接受旧拆分 scope 字段，也不在 Payment 内部把缺失配置改用默认 scope。调用方拿不到合法 scope、source 或 effective config 时必须阻断，而不是在 Payment 层补隐式配置来源。

支付模块需要遵守以下边界：

| 项 | 责任 |
| --- | --- |
| 普通配置字段 | 用 `<w:config:field>` 声明，保存到 `system_config`，由 SystemConfig 按当前 scope 和 locale 管理。 |
| 支付复杂对象 | 用 `<w:config:adapter>` 暴露摘要和管理入口；真实数据由对应 adapter/service 按精确 `scope + environment` 管理。 |
| Provider 专属配置 | Provider 模块提供 `{method_code}.phtml`，`code` 必须和 Provider `getCode()`、checkout phtml、EAV 属性一致。 |
| checkout 模板 | Provider 模块提供 checkout phtml，按 `method_code` 绑定，不能承担后台配置保存职责。 |
| 特殊动作 | 连接测试、OAuth 授权页、Webhook 管理等特殊逻辑可以由 Provider 自己 controller 或 adapter 提供，最终配置发布和可用性仍回到 Payment 状态机。 |

支付复杂对象至少包含：

| 对象 | 管理方式 |
| --- | --- |
| 支付方式启用、排序、国家、货币、金额上下限 | SystemConfig 普通字段管理，`PaymentScopeConfigService` 只读取 effective runtime config。 |
| 连接测试结果、发布快照、Provider binding、capability snapshot | 跟随 method_code + provider_code + scope + environment 由 adapter/service 保存。 |
| merchant account / webhook endpoint binding | adapter 提供摘要、编辑入口和发布校验；Payment 不做隐式继承。 |
| 路由、风控、network rule | adapter 保存复杂对象，普通开关仍交给 SystemConfig key/value。 |
| effective config snapshot | intent 创建时固化 SystemConfig 来源摘要 + Payment adapter 对象版本。 |

配置模板示例：

```phtml
<?php
/**
 * @meta.title Payment core
 * @config.area backend
 * @config.sort 40
 * @config.acl Weline_Payment::payment_method
 */
?>
<w:config:group key="payment/general" label="Payment General" sort="10">
    <w:config:field key="payment/general/checkout_enabled" type="select" value-type="bool" scope="global,website,store" default="1" />
</w:config:group>
<w:config:adapter code="payment.methods" label="Payment Methods" manage-url="payment/backend/method/index" />
```

### 4.1 数据库约束与索引

- 核心 code 字段必须有唯一约束；软删除或归档表需要通过 `environment + code + archived_at` 保证历史可追溯。
- 普通 key/value 配置唯一键归属 `Weline_SystemConfig`，字段身份是 `module + area + key + scope + locale`，Payment 不复制保存。
- `payment_method_config` 等支付复杂对象唯一键使用 `method_code + scope + environment`；禁止继续把旧拆分 scope 字段作为新设计主键或兼容入参。
- Payment 复杂配置对象按精确 `scope + environment + method_code` 读取；普通 key/value 配置的继承链、来源摘要和差异显示由 `Weline_SystemConfig` 负责，Payment 不写自己的隐式配置来源逻辑。
- 幂等表唯一键至少包含 `environment + merchant_account + method_code + operation + payable_type + payable_id + idempotency_key`。
- active intent 唯一约束按 `environment + payable_type + payable_id + active_flag` 控制；合单/拆单场景必须走 `payment_group`，不能绕过唯一 active intent。
- attempt 需要索引 `intent_code`、`attempt_code`、`method_code`、`provider_code`、`merchant_account`、`status`、`created_at`。
- transaction 需要索引 `intent_code`、`attempt_code`、`method_code`、`provider_reference`、`merchant_account`、`status`、`created_at`。
- Provider reference 唯一键至少包含 `environment + provider_code + merchant_account + provider_reference`；不同 Provider 可重复 reference，但同 Provider/商户/环境下不得重复。
- callback event 需要索引 `provider_event_id`、`webhook_endpoint_code`、`transaction_code`、`intent_code`、`received_at`、`processed_at`。
- ledger 必须不可变，按 `payable_type + payable_id`、`intent_code`、`transaction_code`、`refund_code`、`ledger_type`、`created_at` 建索引。
- analytics snapshot 按 `bucket_time + scope + method_code + provider_code + currency_code + country_code` 建聚合索引；scope 维度必须支持父 scope 聚合和子 scope drill down。
- 归档分区键默认使用 `created_at` 和 `environment`；涉及财务锁账的数据归档前必须校验 finance period 状态。
- 发布配置前执行 schema/config lint，阻断默认货币不在支持货币、Provider 不支持退款但配置开启、强制条款缺翻译、国家为空但启用等冲突。

当前锁与幂等最小服务契约：

- `PaymentLockService::acquire()` 按 payable、method、operation、lock_key 和环境/商户上下文获取锁，返回持有者 token；同 token 可重入，不同 token 在未过期时阻止执行。
- `PaymentLockService::release()` 只允许当前 owner token 释放；`releaseExpired()` 批量释放过期 active 锁。
- `PaymentIdempotencyService::begin()` 按 idempotency key、payable、method、operation 和请求指纹判断是否执行、返回既有结果、阻止进行中重复请求或报告失败记录。
- `PaymentIdempotencyService::complete()` 保存成功结果，后续同 key 同指纹请求 replay；`fail()` 保存失败原因和失败 payload，后续同 key 同指纹请求不重复执行。
- 该服务层只提供锁和幂等状态能力，不接管 `PaymentService` 主编排；状态机、Provider 调用和 Payable 回调必须由后续主流程显式接入。

### 4.2 Ledger 入账规则

Ledger 是支付内核的最终审计账，必须不可变、可追溯、可重放校验。任何退款、撤销、争议、人工修正、跨期调整都只能写反向账或调整账，不能修改旧账。

Ledger 至少包含以下类型：

| 类型 | 场景 |
| --- | --- |
| `authorization_hold` | 授权占用额度但未捕获。 |
| `authorization_void` | 授权取消或过期释放。 |
| `capture` | 授权捕获入账。 |
| `payment` | 即时支付入账。 |
| `asset_reserve` | 信用、积分、W币预占。 |
| `asset_commit` | 资产支付或抵扣正式确认。 |
| `asset_release` | 支付失败、取消、过期、退款时释放资产预占。 |
| `discount` | 活动折扣、Provider 折扣、平台补贴。 |
| `surcharge` | 客户侧手续费或便利费。 |
| `provider_fee` | Provider 手续费。 |
| `platform_fee` | 平台手续费或通道服务费。 |
| `tax_on_fee` | 手续费税费。 |
| `refund` | 原路退款。 |
| `refund_reversal` | 退款失败或撤销后的反向处理。 |
| `alternative_refund` | 人工退款、余额退款、线下退款、补偿券等替代退款。 |
| `chargeback` | 拒付或争议损失。 |
| `dispute_hold` | 争议期间冻结。 |
| `dispute_release` | 争议胜诉或释放冻结。 |
| `settlement` | Provider 结算入账。 |
| `payout` | 向商户或主体打款。 |
| `rounding_adjustment` | 汇率、币种精度、分摊尾差。 |
| `manual_correction` | 人工修正，必须有审批、原因、附件和反向关系。 |
| `period_adjustment` | 财务锁账后跨期调整。 |

Ledger 规则：

- 每条 ledger 必须记录 `ledger_code`、`ledger_type`、`direction`、`amount_minor`、`currency_code`、`precision`、`payable_type`、`payable_id`、`intent_code`、`transaction_code`、`refund_code`、`allocation_code`、`source_code`、`created_by`、`reason_code`。
- 同一 intent 的 allocation 总额必须等于 Payable 快照应付金额加 surcharge 减折扣；不同币种只能通过 `payment_fx_quote` 建立汇率关系，不能直接混算。
- 可配置是否采用严格复式记账；即使第一期不实现完整总账，也必须保留 debit/credit/account 预留字段，避免后续财务模块重构。
- 任何 `paid`、`refunded`、`settled`、`payout`、`chargeback` 状态判断必须以 ledger 和 transaction/refund 双方一致为准；Provider 回调只能提供状态证据，不能直接替代账本。
- 部分退款按原 allocation 快照比例反向入账；按 item 退款时优先使用 item allocation，无法精确匹配时写 `rounding_adjustment`。
- 已锁财务期间禁止改旧 ledger；只能写 `period_adjustment` 到开放期间，并关联原 ledger_code。

## 5. Checkout 与支付方式可用性

- `checkout_session` 承载临时结账状态：地址、配送、税费、优惠、资产预占候选、支付方式候选、支付条款接受状态、payer token、guest token、resume token、session version、过期时间。
- `payment_intent` 只承载已冻结金额快照后的支付生命周期；checkout session 变更金额、配送、税费、优惠或强制条款版本时关闭旧 intent 并重建。
- Payable 金额、税费、运费、优惠、汇率、条款、scope effective config 或 item 明细变化后，旧 intent 不能继续支付；必须关闭、标记 superseded 或要求用户重新确认后创建新 intent。
- 每个支付方式必须配置支持国家、支持货币、默认货币；默认货币必须在支持货币范围内。
- checkout 支付方式可用性由 `Weline_Payment` 统一计算：Payable 金额与币种、用户国家、支付方式国家配置、支付方式货币配置、Provider 能力、merchant account 状态、RiskRule、业务类型 tags、金额上下限、时间窗口、条款同意状态共同决定。
- 用户国家匹配来源优先级：配送国家 -> 账单国家 -> 客户资料国家 -> checkout session 国家 -> GeoIP -> website 默认国家；最终国家写入 checkout session 快照。
- 默认 checkout 只加载并展示当前可用支付方式。
- “更多支付方式”按钮点击后追加加载不可用支付方式，以灰色禁用态展示。
- 禁用支付方式不能选择，必须返回 `disabled_reason_code` 和 i18n 文案；桌面 hover 显示原因，移动端点击信息图标或禁用项显示原因。
- 禁用状态必须满足可访问性：ARIA disabled/describedby、键盘焦点可达、屏幕阅读器可读原因，不只依赖 hover。
- 合规禁止、强风控隐藏、后台配置为不公开的支付方式不进入前台“更多”列表；backend/debug 模式可查看隐藏原因。
- 如果 Payable 支付币种不被支付方式支持，只有支付方式配置允许 `currency_conversion` 且存在有效 `fx_quote` 时，才可用该支付方式默认货币创建 intent；否则禁用。
- 支付方式可用性结果支持短 TTL 缓存，并能被配置变更、Provider 健康、KYC 状态、汇率报价、地址/国家变化、活动变化、条款版本变化实时失效。
- checkout 刷新、浏览器返回、多标签同时支付必须通过 session version、前端 submit lock、后端幂等和状态恢复接口处理。
- 跳转类/扫码类支付必须支持 return_url、cancel_url、failure_url、app deep link、QR polling、voucher/QR 过期刷新、用户关闭页面后的状态恢复。
- 支付结果页只能展示 `processing`、`waiting_confirmation`、`failed`、`requires_action` 等服务端状态；浏览器 return 不等于 paid，最终状态必须以后端 callback/query 为准。

### 5.1 支付方式动态表单

- Provider 可声明动态表单 schema，`Weline_Payment` 统一渲染、校验、脱敏、快照和 i18n。
- 动态字段示例：CPF/CNPJ、DNI、银行代码、UPI VPA、手机号、邮箱、银行分行号、票据编号、企业税号。
- 动态表单字段必须声明 code、type、label、required、validation、masking、scope、保存策略、是否进入 Provider 请求、是否进入审计快照。
- 动态字段默认不进入日志；需要展示到客服时间线或后台详情时必须按敏感字段分级脱敏。
- 动态表单 schema 进入 Provider contract test，避免 checkout 模板为单个 Provider 写定制逻辑。

### 5.2 客户侧手续费与附加费

- 支持按国家、支付方式、客户类型、币种、金额区间配置客户侧手续费、便利费或 surcharge。
- 客户侧手续费必须进入 checkout_session 和 intent 金额快照，参与税费、退款、对账和客服时间线。
- 部分国家或支付网络禁止或限制 surcharge；区域合规元数据命中禁止规则时必须自动禁用该配置。
- surcharge 退款规则独立配置：全退、按比例退、不退、人工审核。
- 前台必须明确展示手续费名称、金额、币种、计算方式和条款依据。

### 5.3 Checkout 前端数据契约

Checkout 前台不得直接调用 Provider，也不得在 `.phtml` 或 JS 中写 direct fetch。支付方式列表、创建 intent、继续支付、条款同意、动态字段校验都必须通过 Weline 前端 API 链路：`Weline.Api.resource()` / `Weline.Api.graph()` / `Weline.Api.stream()` -> worker -> QueryProvider/service。

`payment/v1/available-methods` 或等价 QueryProvider 返回的支付方式项至少包含：

| 字段 | 说明 |
| --- | --- |
| `method_code` | 支付方式稳定 code，必须和 Provider、checkout phtml、配置 phtml 一致。 |
| `provider_code` | 实际路由 Provider，未决策时可为空。 |
| `title` / `description` | i18n 后的前台展示文案。 |
| `display_assets` | logo、icon、暗色模式资产、品牌色、排序分组。 |
| `available` | 是否可选择。 |
| `disabled_reason_code` | 不可用原因 code，例如 `currency_unsupported`、`country_unsupported`、`terms_required`、`provider_unhealthy`。 |
| `disabled_reason_text` | i18n 后的不可用原因，hover/tap/a11y 使用。 |
| `sort_weight` | 后端已计算排序权重，前端只做稳定排序展示。 |
| `supported_currencies` | 当前配置收窄后的支持币种。 |
| `default_currency` | 支付方式默认币种，不支持 Payable 币种且未启用换汇时必须禁用。 |
| `supported_countries` | 当前配置收窄后的支持国家。 |
| `payment_currency_code` | 本次 intent 可用支付币种。 |
| `display_currency_code` | 用户看到的展示币种。 |
| `requires_terms` | 是否需要勾选支付条款。 |
| `terms` | 条款 code、版本、标题、链接、强制同意类型。 |
| `dynamic_form_schema` | Provider 声明的本地字段 schema。 |
| `surcharge` | 手续费金额、币种、名称、退款规则、合规提示。 |
| `capabilities` | 本次可见能力，例如 refund、installment、authorize_capture、saved_instrument。 |
| `resume` | 是否存在未完成 intent、恢复入口、过期时间。 |
| `ui_template_code` | checkout phtml 模板 code，默认等于 `method_code`，允许 Provider 显式覆盖。 |

创建支付或继续支付时，前端必须提交 `checkout_session_code`、`payable_type`、`payable_id`、`method_code`、`terms_acceptance`、`dynamic_form_value`、`idempotency_key` 和前端 session version。后端必须重新校验金额、国家、币种、条款、动态字段和支付方式可用性，不能信任前端缓存。

## 6. 支付条款与同意

- 支付条款由 `Weline_Payment` 统一管理，支持按 scope、国家、语言、支付方式、Payable 类型、业务类型、订阅/自动扣款/分期/线下支付场景配置。
- 条款类型至少包含：通用支付条款、退款条款、分期条款、订阅/自动扣款授权、保存支付工具授权、线下支付到账规则、账期/信用账户规则、隐私与数据处理、第三方 Provider 跳转说明。
- 每个条款有 code、版本、语言、适用范围、生效时间、失效时间、是否强制、是否需要重新同意、展示模板、摘要和完整内容。
- checkout 创建 intent 前必须记录 `payment_terms_acceptance`：terms_code、version、locale、accepted_at、payer、ip、user_agent、checkout_session_code、intent_code。
- 条款变更后可配置是否只影响新 intent，或要求未完成 checkout session 重新同意。
- 自动扣款、保存支付工具、订阅、分期等强授权场景必须单独记录明确同意，不允许只复用通用支付条款。
- 条款同意记录进入审计和争议证据，归档后仍可追溯。

## 7. 支付方式展示资产

- 支付方式展示资产由 `Weline_Payment` 管理，不允许 checkout 模板硬写。
- 每个支付方式支持 logo/icon、暗色/亮色图标、禁用态图标、品牌名、本地化 display name、短说明、长说明、合规提示、Provider 跳转提示。
- 展示资产支持 scope 和 locale 覆盖；解析来源顺序必须显式记录为 method、provider、default，不能静默改写支付语义。
- 资产变更进入配置发布流程，旧 intent 和历史时间线保留当时展示名/提示快照。

## 8. PaymentIntent 生命周期

intent 状态至少包含：

- `draft`
- `pending`
- `zero_amount_ready`
- `redirect_pending`
- `qr_pending`
- `voucher_pending`
- `authentication_required`
- `authentication_processing`
- `authentication_failed`
- `authentication_abandoned`
- `requires_action`
- `processing`
- `partially_paid`
- `authorized`
- `captured`
- `paid`
- `retryable_failed`
- `failed`
- `underpaid_review`
- `overpaid_review`
- `duplicate_payment_review`
- `expired`
- `cancelled`
- `superseded`
- `closed`
- `refunding`
- `partially_refunded`
- `refunded`
- `disputed`
- `late_success_review`
- `review_required`

核心规则：

- 零金额支付：优惠/积分/W币/信用全额抵扣时创建 `zero_amount_intent`，不调用外部 Provider，直接写 allocation、ledger，并通知 PayableResolver。
- 支付失败原因归类：用户取消、余额不足、风控拒绝、认证失败、网络失败、Provider 故障、配置错误、币种不支持、国家不支持、KYC 受限、金额超限、条款未同意、授权已过期。
- 继续支付：`resumePayment(payable_type, payable_id)` 查找 active intent；未过期、金额快照未变、条款仍有效、支付方式仍可用则恢复原 intent，否则关闭旧 intent 后创建新 intent。
- 用户重新选择支付方式：原 intent 或 attempt 不能被静默改写；必须进入 `failed`、`cancelled`、`superseded`、`processing` 或 `review_required` 中的明确状态，再创建新的 intent/attempt。
- 晚到成功：旧 intent 被新尝试替代后收到成功回调，默认进入 `late_success_review`，不得直接把 Payable 改成 paid；后续由配置决定原路退款、暂挂、人工确认或关联到已支付 Payable。
- 自动关闭：intent 保存 `expires_at`、`resume_expires_at`、`auto_close_at`、`provider_expires_at`、`authorization_expires_at`、`closed_at`、`close_reason`。
- 授权支付支持授权有效期、自动 void、重新授权、捕获截止提醒。
- 默认支付窗口 30 分钟；Payable 类型、支付方式、Provider 可配置不同 TTL；Provider 返回更短有效期时以 Provider 为准。
- `Weline_Payment` 新增 cron/queue 到期扫描任务，关闭过期 checkout session、intent、QR/voucher、授权，释放资产预占、优惠占用、支付锁、临时 checkout 输入，并通知 PayableResolver。
- 同一 Payable 默认只允许一个 active intent；多个 Payable 合并支付或一个 Payable 拆分履约通过 `payment_group` 建模。
- 支付尝试次数和频控按 Payable、payment link、guest token、IP、客户账号、method_code 维度限制，防止刷 Provider API 和滥用链接。

### 8.1 PaymentAttempt

`payment_attempt` 记录用户或系统对某个 intent 的每一次明确支付尝试。它不是账本，不代表资金入账；资金动作仍以 transaction/refund/ledger 为准。

Attempt 必须记录：

- `attempt_code`
- `intent_code`
- `payable_type`
- `payable_id`
- `method_code`
- `provider_code`
- `merchant_account`
- `scope`
- `payment_currency_code`
- `amount_minor`
- `status`
- `failure_reason_code`
- `user_confirmed`
- `idempotency_key`
- `provider_reference`
- `created_at`
- `closed_at`
- `superseded_by_attempt_code`

Attempt 状态至少包含：

- `created`
- `provider_pending`
- `requires_action`
- `processing`
- `succeeded`
- `failed`
- `cancelled`
- `abandoned`
- `superseded`
- `late_success_review`

Attempt 规则：

- 一个 intent 可以有多个 attempt，但同一时间只能有一个 active attempt。
- 用户重新选择支付方式时，旧 active attempt 必须先变为 `failed`、`cancelled`、`abandoned`、`superseded` 或 `processing`，然后创建新 attempt。
- attempt 一旦拿到 Provider reference，不允许改 method_code、provider_code、merchant_account、amount 或 currency。
- transaction 必须关联 attempt_code；没有 attempt 的 transaction 只能来自 zero amount、人工收款或迁移场景，并必须注明 source。
- attempt 频控按 Payable、payer、guest token、IP、设备、method_code、provider_code 组合计算。

### 8.2 状态机转移规则

状态流转必须通过 `Weline_Payment` 状态机执行，并在需要时写入 ledger、timeline、audit 和 outbox。

| 来源状态 | 目标状态 | 触发方 | 要求 |
| --- | --- | --- | --- |
| `draft` | `pending` | checkout/API | 金额、条款、可用性、幂等校验通过 |
| `pending` | `redirect_pending`/`qr_pending`/`voucher_pending` | Provider | Provider 返回跳转、二维码或票据动作 |
| `pending` | `zero_amount_ready` | Payment | allocation 全额覆盖，无需外部 Provider |
| `pending`/`requires_action` | `authentication_required` | Provider | 需要 3DS/SCA 或强认证 |
| `authentication_required` | `authentication_processing` | Provider/前台返回 | 带签名 state，等待 Provider 确认 |
| `authentication_processing` | `authorized`/`paid`/`failed` | webhook/query | 以服务端 callback/query 为准 |
| `pending`/`processing` | `partially_paid` | webhook/query/manual | 实收小于应收且允许部分支付 |
| `pending`/`processing` | `paid` | webhook/query/manual | 实收满足应收，allocation commit |
| `authorized` | `captured`/`paid` | capture | 授权未过期，capture 成功 |
| `authorized` | `cancelled` | void/auto void | 授权过期或主动取消 |
| `paid` | `refunding` | refund request | 退款审批或自动退款创建 |
| `refunding` | `partially_refunded`/`refunded`/`review_required` | refund callback/query | 按退款结果裁决 |
| `pending`/`processing` | `expired`/`closed` | auto close | 支付窗口过期，释放预占资源 |
| `pending`/`retryable_failed`/`failed` | `superseded` | user selects another method | 用户明确重新选择支付方式，旧 intent 不再作为当前可支付对象 |
| `superseded`/`cancelled`/`failed` | `late_success_review` | webhook/query | 旧支付晚到成功，需要人工或配置裁决 |
| 任意非终态 | `review_required`/`risk_review_required` | RiskRule/manual/system | 异常入账、风控、乱序回调、状态冲突 |

禁止从 `refunded`、`closed`、`retired` 历史记录直接回到可支付状态；需要重建 intent 或写反向账本。

### 8.3 接口 DTO 契约

Provider 与核心服务之间禁止长期使用无约束 array 传递核心数据，必须定义 DTO 或等价结构：

- `MoneyData`：amount_minor、currency_code、precision。
- `PayableSnapshot`：payable_type、payable_id、payable_code、version、owner、items、amounts、country、locale、timezone。
- `AvailabilityRequest` / `AvailabilityResult`：支付方式可用性输入、可用状态、禁用原因、排序权重、展示资产。
- `DynamicFormSchema` / `DynamicFormValue`：动态表单字段定义、校验和脱敏规则。
- `PaymentRequest` / `PaymentResult`：支付创建、跳转、二维码、票据、Provider reference、下一步动作。
- `PaymentAttemptData`：attempt_code、intent_code、method_code、provider_code、merchant_account、amount、currency、status、failure_reason、user_confirmed。
- `AuthorizeRequest` / `CaptureRequest` / `VoidRequest`：授权、捕获、取消授权。
- `RefundRequest` / `RefundResult`：退款金额、allocation、原因、Provider reference、替代退款要求。
- `CallbackRequest` / `CallbackResult`：headers、raw payload、签名、event id、解析状态、目标 transaction/intent。
- `TermsAcceptanceData`：条款版本、payer、IP、user agent、session、intent。

DTO 必须可序列化到 timeline/audit/debug 快照，并明确哪些字段可展示、可导出、需要加密或脱敏。

## 9. 保存支付工具

- 支持保存支付工具：Provider token、mandate、银行卡 token、钱包 token、银行账户 token，不存储原始卡号/CVV。
- 用户可绑定、解绑、设为默认、查看过期状态、撤销授权。
- 保存支付工具必须记录同意条款版本、授权范围、payer、merchant account、Provider、有效期、撤销时间。
- off-session 支付必须校验授权范围、条款版本、mandate 状态、Provider 能力和地区规则。
- 过期卡、撤销 mandate、Provider token 失效时自动标记不可用，并在 checkout 中给出原因。

## 10. 路由、准入与排序

- 同一 `method_code` 可绑定多个 Provider、merchant account、acquirer 或支付通道。
- 支持按国家、币种、金额、客户组、成功率、费率、Provider 健康、风控、KYC、商户账号、结算主体做支付路由。
- 路由决策写入 intent 快照：provider、merchant account、acquirer、路由规则版本、命中原因。
- 谁可以支付某个 Payable 由 `Weline_Payment` 在支付方式级别配置；默认全部 actor 可以尝试支付，除非支付方式配置显式收窄。
- 支付方式配置支持 actor：游客、登录客户、管理员代收、第三方代付、企业账户、API 客户、A2A 参与方、指定客户组、指定商户账号。
- 前台继续支付必须走 `payment/v1/resume` 或签名 resume token，禁止直接用 `order_id` 或 `payable_id` 绕过支付权限。
- 支付方式排序默认使用 `sort_order`；可配置策略按用户国家、币种、成功率、成本、最近使用、业务推荐、A/B 实验覆盖排序。
- 支付方式配置可限制业务类型：虚拟商品、实物商品、订阅、预售、跨境、高风险品类、B2B 账期、部分发货、A2A 托管等。
- `Weline_Order`、`WeShop_Order`、应用商城、A2A 都各自实现 PayableResolver；`payment_default` 只用于简单一次性支付。
- 订单关闭后晚到成功回调默认进入 `paid_after_cancelled_requires_review`，可配置恢复订单、自动退款或人工收入确认，但不能静默改成已支付。

## 11. Provider 入驻、Webhook 与区域合规

- Merchant account 支持 OAuth 授权、子商户入驻、KYC 文件上传、KYC 状态同步、webhook 自动注册、账号能力刷新。
- Webhook endpoint 按 environment、merchant account、method_code 生成，记录 Provider 侧注册状态、secret 版本、endpoint 健康、旧 endpoint 退役时间。
- 支持 webhook secret 轮换和双 secret 过渡期。
- 每个真实 Provider 需要测试矩阵：成功、失败、3DS、用户取消、超时、退款失败、争议、重复回调、乱序回调、签名失败、API 限流。
- 支持 statement descriptor：银行卡账单显示名、收款主体名、门店名、订单号备注，并写入 intent 快照。
- 区域合规通过 Provider capability + country compliance metadata 承载，例如 PSD2/SCA、印度 e-mandate、巴西 Pix、欧洲 VAT、现金券/银行转账过期规则。
- 数据驻留策略覆盖 raw payload、PII、发票、支付凭证、KYC 文件，支持区域化存储、脱敏、保留期和跨境限制。
- 支付方式下线后必须明确：历史 refund 是否允许、webhook 是否继续接收、客服时间线是否可查、后台是否只读、配置是否可导出归档。

## 12. 支付方式生命周期与配置发布

- 支付方式生命周期：`draft`、`testing`、`live`、`maintenance`、`deprecated`、`retired`。
- `maintenance` 隐藏或禁用新支付但保留查询、回调、退款。
- `deprecated` 禁止新 intent 但允许历史退款。
- `retired` 只保留只读历史和必要回调兜底。
- 配置发布流程：草稿、连接测试、审批、发布、回滚、配置 diff、发布人、发布时间、发布原因。
- 密钥、支持币种、支持国家、路由、退款策略、TTL、actor access policy、支付条款、展示资产的变更必须可追溯。
- 配置保存不等于生效；只有发布后的配置版本可用于新 checkout session 和新 intent。

## 13. 资产、折扣与退款

- 信用体系、积分体系、W币分别作为独立支付资产模块接入；默认不启用、不参与支付、不参与折扣、不启用兑换比例。
- 兑换比例可预置默认值，但必须开启后才可抵扣、全额支付或部分抵扣。
- 同一个资产在同一 Payable 上只能选择 `discount` 或 `payment` 一种角色，禁止双重抵扣。
- 资产使用必须走 `reserve -> commit -> release`；支付失败、intent 关闭、订单取消、退款都要回滚或反向入账。
- 默认监管边界：信用、积分、W币仅用于平台内抵扣/支付，不默认支持提现、转让、现金等价兑换、跨主体清算。
- 金额舍入尾差必须入账；按配置由最后一条 allocation、商家或平台承担，并写入 ledger。
- 支付方式活动折扣独立建模，记录 funding source：商家承担、Provider 承担、平台承担、共同承担；退款按原分摊快照反向处理。
- 支持全额退款、部分退款、多次部分退款、按金额退款、按 item 退款、运费/税费/手续费退款、资产原路退回、混合支付按原分摊退款。
- 原路退款失败进入 `alternative_refund_required`，支持人工退款、余额退款、线下退款、补偿券，但必须有审批、账本、附件证据和关闭原因。
- 欠付默认进入 `underpaid_review`；超付默认进入 `overpaid_review`；重复支付默认进入 `duplicate_payment_review`，可配置自动退款或挂账。
- 交易和结算记录：`gross_amount`、`provider_fee`、`platform_fee`、`tax_on_fee`、`net_amount`、`fee_currency`、`fee_refundable`、`fee_refund_amount`。

## 14. 线下支付、链接、3DS 与财务

- 线下/人工支付作为 Provider 类型统一接入，包括银行转账、COD、人工转账、B2B 账期、支票、现金收款。
- 支持凭证上传、到账确认、后台审核、审核驳回、人工标记已收款、撤销人工确认；人工修正必须支持申请、审核、执行、反向账本、审计原因、附件证据，高风险金额可配置双人审批。
- 支付凭证文件治理：文件类型限制、大小限制、病毒扫描、访问权限、保留期、脱敏、删除策略。
- 新增 `payment_link`：支持客服收款、B2B、第三方代付、应用市场付款；可分享、可过期、可限制 payer、可撤销、可审计、可绑定 Payable 或 payment_group。
- 支持 3DS/SCA、挑战页、免验证豁免、责任转移、验证失败和验证放弃；Provider 归一化认证结果。
- 支付成功不等于开票；`Weline_Payment` 只发出支付/结算/退款事件，PayableResolver 决定发票、收据、红冲、权益发放、权益撤销、税务申报。
- `payment_fx_quote` 管理报价创建、有效期、过期重报、汇率来源、支付币种、结算币种、显示币种和汇兑损益归属。
- 对账差异类型包含少结、多结、手续费不一致、币种不一致、重复结算、未结算、未知入账、Provider 状态不一致，并有处理状态和关闭原因。
- 财务期间支持锁账；已锁期间的支付、退款、人工修正不能改旧记录，只能写跨期调整和反向账本。
- 多商户/多业务 scope 必须明确 settlement owner、scope、legal entity、invoice issuer、merchant account 和 payout account。
- 线下支付、银行转账、账期、提现、结算使用 business calendar，支持工作日、节假日、国家/地区日历，不只依赖简单 TTL。
- merchant account 可记录余额、日限额、单笔限额、结算冻结、退款余额不足等限制，并进入支付/退款可用性判断。

## 15. 后台配置模板与 EAV

- `Weline_Payment` 通用 key/value 配置通过 `Weline_SystemConfig` 模板声明：启用、排序、标题、描述、图标、生命周期、环境、actor access policy、支持国家、支持货币、默认货币、金额上下限、时间窗口、支付有效期、恢复支付有效期、自动关闭策略、退款策略、部分支付、组合支付、定金尾款、授权捕获、回调重试、排序策略、可用性缓存 TTL、是否在“更多支付方式”展示不可用项。
- 支付复杂对象通过 `<w:config:adapter>` 进入 Payment 后台能力：支付方式启用发布、provider binding、merchant account、webhook endpoint、路由策略、风险规则、连接测试结果和发布快照。
- backend 配置模板只允许放在 `extends/module/Weline_SystemConfig/Config/{area}/{code}.phtml`，使用 `@meta.*`、`@config.*`、`<w:config:group>`、`<w:config:field>`、`<w:config:adapter>` 声明；普通字段保存、scope 切换、继承来源、校验和缓存失效由 `Weline_SystemConfig` 完成。
- checkout phtml 仍按 `method_code` 绑定；Provider backend config phtml 的 `{code}` 也必须匹配 `method_code`，但不承担 Payment 自己的保存、scope 解析或隐式配置来源职责。
- 配置支持 schema version、迁移、密钥加密、密钥脱敏、空值不覆盖、连接测试、密钥轮换、webhook secret 过渡期；其中普通字段版本与来源由 SystemConfig 记录，支付复杂对象版本由 adapter/service 记录。
- 新增 `Weline\Payment\Helper\PaymentMethodAttributeHelper` 和后续 AttributeBag，用于按 `method_code` 读取/写入普通支付方式 EAV 属性；第三方属性 code 必须带 Provider 前缀，常用类型使用 EAV 类型 code（例如 `input_string`、`input_int`、`input_bool`），不能使用数据库字段类型名代替。
- 支付方式 EAV 只放普通扩展元数据；密钥、token、证书、webhook secret 等敏感配置必须仍由 `Weline_SystemConfig` 加密字段保存。

### 15.1 Scope 范围化配置

支付配置必须支持 scope 化，但 scope 选择、继承链、来源说明、diff、权限和普通 key/value 保存都归属 `Weline_SystemConfig`。`Weline_Payment` 不暴露 `getConfigScopes`、`resolveConfigScope`、`getConfigScopeChain` 等自有 scope API，也不维护 Payment 专用隐式配置来源。

普通 key/value：

- 字段由 `<w:config:field>` 声明，SystemConfig 按 `module + area + key + scope + locale` 保存。
- `scope="global,website,store"`、`locale=true`、`global_only=true`、`value-type="encrypted"` 等字段能力由 SystemConfig 解释。
- 后台配置中心负责展示 source scope、继承来源、当前覆盖值、最终生效值、字段 diff、审批、保存版本和缓存失效。
- Payment 创建 checkout session、intent 或后台 adapter 动作时，只消费 SystemConfig 已解析的 `scope`、source 摘要和 effective config；没有合法解析结果时阻断。

模板示例：

```phtml
<w:config:field key="payment/method/fake_card/live_secret_key" type="secret" value-type="encrypted" scope="global,website,store" required="true" label="Live Secret Key" />
<w:config:field key="payment/method/fake_card/sort_order" type="number" value-type="int" scope="global,website,store" default="0" label="排序" />
<w:config:field key="payment/method/fake_card/provider_api_version" type="text" value-type="string" scope="global" label="Provider API Version" />
<w:config:adapter code="payment.methods" label="Payment Methods" manage-url="payment/backend/method/index" />
```

PaymentScopeConfigService / adapter：

- `PaymentScopeConfigService` 只解析单一 scope 字符串并读取 SystemConfig effective runtime config，不保存普通 key/value，也不暴露旧 profile 保存接口。
- adapter/service 入参只接受单一 `scope`、`environment`、`method_code` 和业务对象标识；不接受旧拆分 scope 字段。
- merchant binding、连接测试结果、发布快照、路由/风控复杂对象按精确 scope 读写；未命中即未配置，不读取父 scope。
- 连接测试、config lint、发布、启用支付方式只针对当前精确 scope + environment。
- intent 创建时保存 `effective_config_snapshot_code`，包含 SystemConfig 来源摘要、Payment adapter 对象版本、Provider capability snapshot、merchant account、密钥版本引用。
- 历史退款、争议、对账、回调解析优先使用 intent 的 effective config snapshot；当前配置只影响新 checkout 和新 intent。

迁移边界：

- 不兼容旧拆分 scope 字段，不做运行期双读、双写或隐式配置来源。
- 旧数据若存在，必须由一次性迁移任务转换为 SystemConfig scope 字符串和 Payment adapter 精确 scope 对象后才能进入新支付配置。
- 导入导出由 SystemConfig 处理普通字段；Payment adapter 只导出复杂对象摘要，敏感字段必须脱敏或重新填写。

### 15.2 配置缓存与失效矩阵

支付可用性结果可以短 TTL 缓存，但必须支持事件驱动失效。配置发布、Provider 健康变化、merchant account 状态变化、条款版本变化、汇率报价变化、国家/币种规则变化、EAV 属性变化，都不能依赖自然过期。

| 变更来源 | 必须失效 |
| --- | --- |
| SystemConfig 普通字段保存、发布或回滚 | checkout available methods、backend method detail、dashboard filter cache、config lint result。 |
| Payment adapter 复杂对象发布或回滚 | method config、merchant binding、route decision cache、checkout availability。 |
| Provider capability 变化 | method capability、currency/country availability、config lint result、route decision cache。 |
| Merchant account KYC/冻结/余额/限额变化 | checkout availability、refund availability、routing cache、provider health panel。 |
| 支持国家/货币/默认货币变化 | checkout method list、more methods list、disabled reason、currency conversion cache。 |
| 支付条款版本或翻译变化 | checkout terms、intent create validation、terms acceptance requirement。 |
| 展示资产变化 | checkout method cards、backend method preview、dashboard legend。 |
| EAV 属性变化 | Provider dynamic config、availability、checkout dynamic form、risk rule input。 |
| FX quote 创建/过期/刷新 | checkout amount display、currency conversion availability、intent creation guard。 |
| RiskRule 配置变化 | checkout availability、intent create guard、manual review routing。 |
| Provider health/circuit breaker 变化 | checkout availability、routing cache、dashboard health metrics。 |
| SystemConfig scope 注入、停用或继承关系变化 | SystemConfig 配置中心、effective config 来源摘要、checkout availability、adapter 入口上下文。 |

缓存 key 至少包含 environment、SystemConfig 已解析 scope、method_code、provider_code、merchant_account、country_code、currency_code、payable_type、business_tags、customer_group、actor_type、SystemConfig source/version、Payment adapter object version。缓存命中返回时必须附带 `scope`、SystemConfig 来源摘要、adapter version 和 `computed_at`，便于排查。

状态流转只能由 `Weline_Payment` 状态机执行，禁止 Provider 或订单模块直接改核心支付状态。

支持未完成支付提醒、即将过期提醒、过期后重新支付提醒；通知必须有发送记录、幂等、退订/免打扰、语言模板版本。

### 16.1 Outbox/Inbox 可靠投递

- 必须进入 outbox 的事件：intent paid、intent failed、refund succeeded、refund failed、asset committed、asset released、payable fulfilled/revoked、risk review required、dispute opened、settlement reconciled。
- outbox 事件必须包含 event_code、event_type、aggregate_type、aggregate_code、sequence、payload_hash、status、retry_count、next_retry_at、dead_letter_reason。
- inbox 消费必须以 consumer_code + event_code 建唯一幂等键，消费者重复执行不得重复发货、重复退款、重复资产入账。
- 同一 aggregate 内事件必须保持顺序；跨 aggregate 不要求全局顺序。
- 失败重试采用指数退避；超过配置次数进入 dead letter，后台可查看、重放或关闭。
- 同步 dispatch 只允许作为本地即时提示，不能替代 outbox 持久事件。

### 16.2 通知、收据与模板边界

- `Weline_Payment` 负责支付事件通知的触发、幂等、语言选择、模板版本快照、发送记录和客服时间线，不直接承担业务发货、发票开具或售后通知。
- PayableResolver 负责告诉支付内核该 Payable 是否需要支付收据、退款收据、部分退款通知、失败提醒、未完成支付提醒、争议提醒，以及通知接收人范围。
- receipt 只代表支付凭证，不等同发票。发票、红冲、税务申报由 Payable 所属业务模块处理。
- 通知模板按 locale、country、scope、method_code、payable_type、event_type 匹配；模板版本必须进入 notification log 和 receipt 快照。
- 通知发送必须记录 `notification_code`、event_code、payable、intent、refund、recipient、channel、locale、template_version、status、retry_count、sent_at、failed_reason。
- 重发通知必须幂等；人工重发需要 ACL、原因、操作人和审计记录。
- 部分退款、多次部分退款、替代退款、退款失败和退款撤销必须使用不同事件类型和模板，不允许只发“退款成功”一种泛化通知。
- 邮件、短信、站内信、Webhook、A2A 回执等通道可以由其他模块实现，但必须通过 `Weline_Payment` 的通知事件和 outbox/inbox 协议接入。

### 16.3 Payable 业务补偿边界

支付内核和业务履约必须分离。Provider 资金结果一旦确认，不能因为业务模块失败而改写支付事实。

- 支付已 `paid` 后，PayableResolver `onPaid()` 发货、权益发放、应用授权失败时，payment 保持 `paid`，业务模块进入补偿队列或 `fulfillment_failed` 等业务状态。
- 退款已 `refunded` 后，PayableResolver `onRefunded()` 撤销权益、回滚库存、红冲失败时，refund 保持成功，业务模块进入撤销补偿状态。
- `onPaid()`、`onRefunded()`、`onExpired()`、`onRiskReview()` 必须幂等；重复消费 outbox 不得重复发货、重复授权、重复撤销。
- 业务补偿失败必须写 timeline、audit、notification log，并提供后台重试或人工关闭入口。
- Payment 只发布支付事实事件和补偿请求，不直接修改业务模块内部状态字段。

## 17. 后台驾驶舱与客服时间线

- 后台新增支付统计数据可视化驾驶舱，由 `Weline_Payment` 提供统一数据口径。
- 支持时间范围、scope、merchant account、provider、method_code、国家、币种、客户组、Payable 类型、环境、路由通道筛选。
- 核心指标：支付总额、净收款、订单/Payable 数、支付成功率、失败率、授权率、捕获率、退款率、争议率、继续支付成功率、自动关闭数量、平均支付完成时间。
- 图表维度：按支付方式、Provider、国家、币种、merchant account、失败原因、退款原因、通道路由、资产抵扣、活动折扣、线下支付、payment link 展示。
- 运维指标：回调延迟、回调失败、重试次数、死信数量、Provider 健康、支付方式不可用原因、KYC 异常、汇率报价失败、对账差异。
- 财务指标：gross、fee、tax_on_fee、net、settlement、payout、reconciliation difference、跨期调整。
- 支持从图表 drill down 到 intent、transaction、refund、ledger、callback event、settlement 和 dispute。
- 统计数据使用 `payment_analytics_snapshot` 或等价聚合表，按分钟/小时/天聚合；原始流水仍以 ledger/transaction 为准。
- 新增客服支付时间线，按 Payable/intent 展示用户看到的支付方式、展示资产快照、条款同意、不可用原因、支付尝试、失败原因、回调记录、退款状态、通知发送和可执行动作。
- 驾驶舱支持 CSV 导出，但受 `analytics_export` ACL 控制；导出数据必须脱敏敏感字段。
- 指标口径必须文档化，避免后台金额、账本金额、对账金额三套口径不一致。

## 18. 国际化与全球支付

- 支付上下文包含：`scope`、`system_config_source_summary`、`locale`、`language_code`、`country_code`、`region_code`、`currency_code`、`display_currency_code`、`payment_currency_code`、`settlement_currency_code`、`timezone`、`provider_locale`。
- 支持 Provider 货币范围、国家范围、语言范围、最小/最大金额、支付时效、结算币种和本地支付方式发现。
- 金额、日期、时区、国家/地区、错误提示、backend 配置 label/help/options、checkout 可见文案、支付条款、驾驶舱指标名称全部走 i18n。
- 支持 RTL checkout 布局、地区必填字段、当地合规提示、隐私同意文案、发票/税号字段、收据/邮件通知本地化。
- 缺失翻译、Provider locale mapping 缺失、币种精度缺失、条款翻译缺失、sandbox/live 混用进入健康检查和告警。
- 所有时间落库使用 UTC，前台和后台按用户/商家/Payable 所属时区展示。

## 19. 安全、合规与运维

- merchant account 状态进入支付可用性判断：KYC 未完成、账号冻结、payout 禁用、国家/币种受限、风控冻结、余额不足、限额不足时，相关支付方式不可用。
- 锁粒度包含：payable lock、checkout session lock、intent lock、transaction lock、refund lock、asset source lock、callback event lock、merchant account lock、finance period lock。
- 回调保存 raw payload、headers、signature、provider_event_id、provider_api_version、webhook_schema_version、received_at、processed_at、result、retry_count；raw payload 加密并设置保留期。
- 支持签名校验、事件去重、重放保护、乱序保护、死信、人工重放、自动重试、Provider 状态查询补偿。
- 禁止处理原始银行卡号/CVV；只支持 Provider token、mandate、subscription token、off-session token。
- PCI 范围默认限制为 hosted checkout、hosted fields、Provider iframe、token/mandate；checkout 页必须配置 CSP、iframe allowlist、日志脱敏，并记录 SAQ 范围。
- 支持 sandbox/live 环境隔离、scope 隔离、merchant account 隔离、租户隔离、密钥权限隔离、审计日志。
- 隐私与保留期覆盖 raw payload、checkout 临时输入、锁、幂等记录、审计、死信、对账文件、驾驶舱聚合数据、支付凭证、KYC 文件、条款同意证据、PII 脱敏/匿名化。

## 20. 事件可靠性与补偿

- 关键事件使用 outbox/inbox 机制，支付成功、退款成功、资产 commit、权益发放等必须可重试、可去重、可追踪。
- 同步 dispatch 只作为本地即时通知，不能作为唯一可靠投递机制。
- Webhook 只读查询补偿策略：pending 多久查一次、最多查几次、何时转 review、何时关闭，由通用配置和 Provider 能力共同决定。
- Provider 模块卸载或 retired 后保存 provider snapshot；历史退款、回调、客服时间线、对账仍可读可处理。
- 跳转支付 return/cancel/failure URL 必须带签名 state，防 open redirect、CSRF、伪造 return、重复 return；最终状态以服务端 callback/query 为准，不以浏览器返回结果为准。
- 风控人工审核状态：`risk_review_required`。支付可能已授权但不能发货/发放权益，人工审核后 capture、void、refund 或 release fulfillment。
- refund、void、reversal、chargeback、ACH return/银行退票必须是不同状态和账本类型，不能混用 refund。
- 争议处理记录 evidence_due_at、respond_by、auto_accept_loss_at、提交证据版本、Provider 回执。
- 敏感字段分级覆盖 KYC 文件、支付凭证、条款同意证据、raw payload、银行账户尾号、设备指纹，分别定义展示脱敏、导出权限和保留期。
- 配置发布前必须执行 config lint，例如默认货币不在支持货币、Provider 不支持退款但后台开启、条款缺翻译但强制展示、国家为空但启用等。

### 20.1 定时任务与队列任务

`Weline_Payment` 必须提供默认任务编排，Provider 模块只处理自己的外部 API 细节，不重复实现通用生命周期。

### 21.1 PaymentDemo 交付物

`Weline_PaymentDemo` 用来证明第三方支付模块接入成本足够低，必须作为 Phase 1 验收样例。

- 一个 Demo Provider 核心类，实现 ProviderInterface 全部函数，覆盖支付、授权、捕获、void、退款、查询、回调校验、错误归一化和连接测试。

- 一组后台测试连接、配置 lint、可用性说明、禁用原因展示和客服时间线样例。

### 21.2 切换与迁移边界

不兼容旧支付架构不代表可以无计划切换。迁移必须先冻结旧入口写入，保留历史只读和必要退款能力，再把新支付对象接管到 `Weline_Payment`。

- 切换前导出旧支付方式配置、交易、退款、订单支付状态、Provider reference、回调 endpoint、未完成支付单。
- 旧配置映射到新 `payment_method_config_release`，旧交易映射到 transaction/ledger，旧订单支付状态映射到 PayableResolver 快照。
- 未完成支付必须分类：可继续支付、已过期关闭、Provider processing 补偿查询、需要人工 review。
- 旧 Provider 必须改造成新 ProviderInterface；未完成改造的支付方式不得在 checkout 展示。
- 旧后台入口迁移到新 backend 路由；保留只读跳转期时必须明确退役日期和权限。
- 切换期间禁止同一 Payable 同时被旧支付和新支付创建 active intent。
- 回滚只能回滚未发布配置和前台展示；已经产生的新 intent、transaction、ledger 不回写旧系统。

## 22. 验收与测试

- 后台路由测试：`payment/backend/config`、`method`、`method/edit`、`terms`、`payment-instrument`、`webhook-endpoint`、`dashboard`、`timeline` 等入口可访问，ACL 生效，菜单落到 `Weline_Backend::payment_group`。
- 配置管理测试：SystemConfig 普通字段保存/发布/回滚、Payment adapter 复杂对象草稿/连接测试/审批/发布/回滚、旧 intent 使用旧配置快照。
- SystemConfig scope 测试：普通 key/value 的 scope 列表、继承来源、source summary、diff、字段保存和权限由 `Weline_SystemConfig` 验证，Payment 不提供自有 scope API。
- Payment adapter 精确 scope 测试：支付方式配置、merchant binding、连接测试结果、发布快照按精确 scope + environment 读写；未命中不读取父 scope，不改用其他配置来源。
- scope 解析测试：checkout 前端传入 scope 不可信，Payment 只消费 SystemConfig 已解析 scope/source；resume/refund/query 使用 intent 快照 scope。
- effective config 测试：intent 保存 `effective_config_snapshot_code`，包含 SystemConfig 来源摘要、Payment adapter 对象版本、Provider capability snapshot 和 merchant account。
- scope merchant binding 测试：精确 scope 绑定 merchant account、legal entity、payout account、webhook endpoint 后，checkout、退款、对账和驾驶舱都按绑定结果工作。
- scope 导入导出测试：普通字段由 SystemConfig 处理；Payment adapter 只导出复杂对象摘要；旧拆分 scope 字段不做运行期兼容，只允许一次性迁移。
- 配置 lint 测试：默认货币、支持国家、Provider 能力、强制条款翻译、退款能力、surcharge 合规冲突必须被阻断或警告。
- 配置缓存测试：SystemConfig 普通字段发布、Payment adapter 复杂对象发布、Provider 健康、merchant account、国家/币种、条款、EAV、FX quote、RiskRule 变化后，checkout 可用性缓存必须按矩阵失效。
- 条款测试：按国家/语言/支付方式展示条款，强制同意后才能创建 intent，条款版本变更触发重新同意，自动扣款/保存支付工具单独授权。
- Provider 接口测试：ProviderInterface 必实现函数完整，unsupported 能力返回明确错误，不能静默改写支付语义。
- PayableResolver 测试：支付成功、部分支付、退款、过期关闭、风控审核只通过 Resolver 或事件影响业务模块，Payment 不直接改业务状态。
- DTO 契约测试：Provider 输入/输出 DTO 可序列化、可校验、敏感字段可脱敏，禁止核心链路依赖无约束 array。
- 幂等测试：相同 idempotency key + 相同请求指纹返回同一结果；相同 key 但金额、币种、Payable、method、operation、payer 或 request body hash 不一致时返回 `payment_idempotency_conflict`。
- 状态机测试：合法状态可转移，非法状态被拒绝，状态变更写入 ledger、timeline、audit 和 outbox。
- Ledger 测试：支付、授权、捕获、void、资产 reserve/commit/release、部分退款、替代退款、手续费、争议、跨期调整都能写入正确 ledger_type，金额平衡。
- 退款顺序测试：多次部分退款、按 item 退款、运营指定来源退款、资产原路退回、手续费/税费退款、Provider 余额不足和替代退款都按配置与快照执行。
- 内核测试：checkout session、intent 创建、resume、过期关闭、重复点击、多标签页、金额变化重建、Provider 有效期覆盖默认 TTL、动态表单、零金额支付。
- payment attempt 测试：每次支付尝试生成 attempt_code，transaction 关联 attempt；用户重新选择支付方式时旧 attempt 进入明确状态，新 attempt 不能静默复用旧 Provider reference。
- superseded/晚到成功测试：旧 intent/attempt 被新支付尝试替代后收到成功回调，进入 `late_success_review` 或 suspense account，不自动改 Payable 成功。
- 金额变更测试：Payable 金额、税费、运费、优惠、汇率、条款、SystemConfig effective source 变化后，旧 intent 不能继续支付，必须关闭、superseded 或要求重新确认。
- Checkout payload 测试：可用支付方式返回 scope、system_config_source_summary、method_code、展示资产、禁用原因、条款、动态字段、surcharge、默认币种、支持国家/币种、ui_template_code，前端不得信任缓存跳过后端校验。
- 跳转/扫码测试：return/cancel/failure URL、deep link、QR polling、voucher 过期刷新、用户关闭页面后恢复。
- 支付方式展示测试：默认只展示可用方式；点击更多后懒加载不可用方式；不可用方式灰色禁用；hover/tap/a11y 显示原因；货币不支持不能选择。
- surcharge 测试：允许地区正确计费，禁止地区自动禁用，退款规则按配置执行，前台展示清晰。
- 支付方式可用性测试：默认货币、支持货币、支持国家、用户国家匹配、业务类型限制、支付币种不支持、KYC 冻结、Provider 故障、缓存失效、生命周期状态。
- 保存支付工具测试：绑定、解绑、默认工具、过期、撤销授权、off-session 同意范围。
- Provider 契约测试：能力声明、配置 schema、动态表单 schema、可用性、支付、授权、捕获、回调、退款、错误归一化必须通过统一 contract test。
- Provider 历史能力测试：Provider capability 变更、禁用、retired 或模块卸载后，历史 intent/refund/dispute/query 仍按创建时快照和最小 handler 可处理。
- PaymentDemo 验收：Demo Provider、checkout phtml、backend config phtml、回调入口、EAV 示例、连接测试、配置 lint、禁用原因和客服时间线样例完整可运行。
- Provider 测试矩阵：成功、失败、3DS、用户取消、超时、退款失败、争议、重复回调、乱序回调、签名失败、API 限流。
- 失败与重试测试：用户取消、余额不足、风控拒绝、认证失败、Provider 超时、配置错误、网络不确定分别进入正确 failure_reason、状态、用户提示、重试策略和补偿查询。
- 重新选择支付方式测试：当前支付方式失败或不可用时，只能由用户明确重新选择支付方式并创建新的支付尝试；系统不得自动切换 Provider 或支付方式。
- 支付网络规则测试：退款窗口、争议窗口、授权有效期、mandate、descriptor、surcharge、ACH/SEPA 退票、Pix/UPI 过期等规则进入可用性和 intent 快照。
- 支付工具生命周期测试：卡更新、token 替换、mandate 失效、用户撤销、Provider 撤销、merchant account 切换都能正确阻断或恢复 off-session 支付。
- 统计驾驶舱测试：指标口径、筛选、图表、drill down、导出 ACL、脱敏、聚合数据与 ledger/transaction 对账。
- 客服时间线测试：支付尝试、展示资产快照、条款同意、禁用原因、失败原因、回调、退款、通知、人工动作完整追踪。
- 业务补偿测试：支付成功后 PayableResolver `onPaid()` 失败不回滚 paid；退款成功后 `onRefunded()` 失败不改退款结果，业务补偿可重试且幂等。
- 资产测试：三种资产默认关闭、兑换比例开启后可用、折扣/支付角色互斥、混合支付、失败释放、部分退款原路退回、舍入尾差入账。
- 财务测试：对账差异、跨期调整、锁账后反向账本、结算主体和发票主体一致性、business calendar。
- 全球合规测试：受制裁国家、受限行业、KYC/KYB、AML/PEP、IP/设备风险、黑名单命中时，支付方式不可用或进入 review，原因可审计。
- 支付链接安全测试：token 泄露、转发、payer 绑定、一次性使用、多次使用、过期访问、管理员撤销、金额变化和设备/IP 限制都可按配置处理。
- 汇率测试：display FX、payment FX、settlement FX、payout FX 分开记录，汇率差异和汇兑损益入账。
- 批量操作测试：批量退款、批量关闭、批量 webhook 重放、批量导出必须 dry-run、审批、分片、限流、审计，并能恢复失败子任务。
- 多租户隔离测试：scope、配置、密钥、webhook secret、payment instrument、报表、对账、payout、客服时间线不能跨 merchant account、legal entity 泄露。
- 暂挂账户测试：超付、未知入账、重复支付、晚到成功、无 Payable 入账进入 suspense account，并能匹配、退款、确认收入、核销或关闭。
- 对账导入测试：CSV/Excel/SFTP/API 导入按 file_hash 去重，字段映射版本、编码、时区、金额精度、部分失败行和 checksum 可审计。
- 法务保留测试：隐私删除、匿名化、脱敏保留、ledger 强制保留、legal hold、争议/税务保留期冲突按策略执行。
- 后台审批测试：密钥、live 发布、退款、国家/币种、最大折扣比例、surcharge、自动捕获、webhook、payment link 的 ACL 和审批流生效。
- 通知收据测试：支付成功、部分退款、多次部分退款、替代退款、退款失败、争议通知使用不同事件和模板版本，重发幂等且写 notification log。
- 运维测试：cron 自动关闭、queue 死信、webhook 重放、通知幂等、归档恢复、审计追踪、隐私保留期清理。
- 灾备恢复测试：回调丢失、队列积压、重复通知、ledger 校验失败、对账误导入、归档误配置、webhook secret 错误、Provider 大面积超时都有 dry-run 和 recovery run 记录。
- 事件可靠性测试：outbox 顺序、inbox 幂等、重试、死信、后台重放、消费者重复执行不产生重复副作用。
- 迁移测试：旧 `Weline_Payment`、`WeShop_Payment` Provider 全部不再作为核心入口；现有支付方式在新接口下完成支付、回调、退款、查询、配置渲染。
- 迁移映射测试：旧 `PaymentTransaction`、旧 `WeShop_Payment` 配置、旧订单支付状态可以映射到新 intent、transaction、ledger、timeline 或只读归档。
- 迁移切换测试：旧入口冻结、未完成支付分类接管、同一 Payable 不存在新旧双 active intent，已生成新 ledger 不回写旧系统。

## 23. 默认假设

- 默认不兼容旧支付接口，旧数据只做一次性迁移或只读归档。
- 后台配置管理主入口默认是 `payment/backend/config` 和 `payment/backend/method`；现有 `payment/backend/method`、`payment/backend/transaction` 保留并扩展。
- 默认全局 scope 由 `Weline_SystemConfig` 提供；不再把站点/店铺写死为支付配置维度，站点、店铺、应用商城、A2A 等都通过 SystemConfig scope 注入机制提供自己的 scope。
- 普通 key/value 的继承和来源只由 SystemConfig 解释；Payment 复杂对象只按精确 scope 读取，未配置时阻断或返回不可用原因。
- 默认支付窗口 30 分钟；Provider 返回更短有效期时以 Provider 为准。
- 默认全部 actor 可以尝试支付某个 Payable；支付方式配置可按 actor、客户组、商户账号、国家、币种等收窄。
- 每个支付方式必须配置默认货币；默认货币必须属于支持货币。
- checkout 默认只展示可用支付方式；点击“更多支付方式”后再加载不可用支付方式并禁用展示原因。
- 通用支付条款默认启用；自动扣款、保存支付工具、订阅、分期、线下支付需要单独明确同意。
- 未支付订单类 Payable 到期默认可自动取消；部分支付、授权中、Provider processing 默认不自动取消。
- 同一 Payable 默认只允许一个 active intent；合单/拆单使用 payment_group。
- 一个 intent 可以有多个 attempt，但同一时间只允许一个 active attempt；用户重新选择支付方式必须创建新的明确 attempt。
- 同一资产在同一 Payable 上不能同时作为折扣和支付方式。
- 欠付、超付、重复支付默认进入 review，不自动改 Payable 成功状态。
- 配置保存默认不立即生效，必须发布后影响新 checkout session。
- 所有用户可见文案默认以中文 source/key 进入 i18n，并提供 `zh_Hans_CN` 与 `en_US`。

## 24. 待继续定界的问题

以下问题不阻塞 Phase 0 冻结主接口，但需要在进入开发前继续定界，避免全球支付场景后期返工：

- 订阅/周期扣款边界：`Weline_Payment` 执行 off-session 支付和 mandate 校验，但订阅计划、账期生成、失败重试策略应由订阅或业务模块拥有，需要明确接口。
- 支付方式活动折扣叠加顺序：购物车折扣、支付方式折扣、Provider 补贴、信用/积分/W币、优惠券、税前税后折扣的计算优先级还需要形成矩阵。
- surcharge、Provider fee、平台手续费、折扣补贴在不同国家的税务处理需要区域规则表，不能只按统一金额字段处理。
- guest、企业账户、第三方代付、A2A 参与方的 payer 身份证明、支付工具所有权、可退款对象和通知对象还需要更细 actor model。
- 风控决策点需要区分 pre-auth、post-auth、pre-capture、post-payment、pre-fulfillment；不同阶段可执行动作不同。
- dispute 证据收集需要 PayableResolver 提供商品、物流、服务交付、条款同意、沟通记录、IP/设备等证据接口。
- Provider hosted fields、iframe、redirect、SDK 模式的 CSP、PCI SAQ 范围和前端安全责任需要按 Provider capability 细分。
- 退款 SLA、退款审批阈值、自动重试次数、退款失败升级路径、余额不足时的替代退款策略需要配置模板化。
- 多主体账务需要 chart of accounts 或等价科目映射，否则 ledger 到财务报表、结算、payout 的映射会不稳定。
- sandbox 到 live 的配置复制、密钥不可导出、配置脱敏导入导出、多环境审批链需要进一步写清。
- Provider 开发者体验还需要脚手架或生成器，例如生成 Provider 类、checkout phtml、backend config phtml、回调 controller 和 contract test stub。
- 观测性需要 SLO 和告警阈值：支付成功率、Provider 超时率、回调延迟、死信数量、自动关闭异常、对账差异金额。
- 迁移完成后需要校验旧系统与新系统的笔数、金额、退款、未完成支付、Provider reference、ledger 总额 checksum，不能只看功能可用。
- 幂等不仅要保存 key，还要保存请求指纹；相同 key 但请求金额、币种、Payable、method、operation、payer 或 body hash 不一致时必须拒绝复用旧结果。
- 退款来源顺序还需要配置矩阵：按原 allocation 比例退、按 item 退、优先退资产、优先退现金、运营指定来源退、手续费和税费是否退，都需要形成可审计规则。
- Resolver 回调幂等需要更硬：`onPaid()`、`onRefunded()`、`onExpired()` 如果执行一半失败，Payment 重试时业务模块必须避免重复发货、重复发权益、重复撤销。
- Provider 能力快照需要进入 intent：支持退款、部分退款、授权捕获、3DS、保存支付工具、支持币种国家等能力变化后，历史支付仍有明确处理依据。
- 全球支付合规需要显式规则来源：制裁国家、受限行业、KYC/KYB、AML/PEP、黑名单、设备/IP 风险、拒付风险评分不应只写成泛化 RiskRule。
- dispute 开启后的资金冻结边界需要继续定界：是否冻结 payout、是否允许继续退款、是否冻结 Payable 权益、证据提交失败是否自动认损。
- Provider 卸载、禁用、retired 后的历史处理需要最小 handler 方案：历史回调、退款、查询、争议、对账不能因模块关闭而完全不可处理。
- payment link 安全还要细化：token 泄露、转发限制、payer 绑定、一次性/多次使用、金额变更、过期访问、管理员撤销、设备/IP 限制都需要配置。
- Provider 或 merchant account 退款余额不足时，需要明确退款状态、拆分退款、重试、替代退款、人工审批和用户通知。
- 汇率需要分层：checkout 展示汇率、支付创建汇率、Provider 实际结算汇率、payout 汇率可能不同，必须分别记录和入账。
- 后台人工动作审批流还需要统一策略：人工确认收款、人工退款、人工修正、关闭 dispute、关闭 dead letter、跨期调整应支持阈值和双人复核。
- Provider SDK 和外部依赖治理需要接入规范：SDK 版本、依赖升级、超时、异常映射、日志脱敏、sandbox/live 差异和安全审查。
- 报表金额口径需要锁定唯一来源：gross、net、fee、tax、refund、chargeback、settlement、payout 应明确来自 ledger、transaction 还是 settlement。
- 支付方式排序和 A/B 实验需要审计：当时为何展示这个顺序、命中哪个实验、是否受运营推荐/成功率/成本影响，都要写入 checkout 快照。
- Provider 接入测试矩阵需要标准化：最小金额、最大金额、不支持币种、不支持国家、重复回调、乱序回调、退款失败、签名失败、超时重试都必须覆盖。

## 25. 运营、法务与异常资金细化逻辑

### 25.1 支付失败原因分类与动作

失败原因必须同时服务 checkout 用户提示、客服排查、状态机、重试策略、Provider 统计和驾驶舱指标。Provider 原始错误必须归一化为稳定 `failure_reason_code`。

| 分类 | 示例 code | 默认动作 |
| --- | --- | --- |
| 用户主动取消 | `user_cancelled`、`browser_closed` | intent 保持可恢复；用户可继续原方式或明确选择其他支付方式。 |
| 用户输入失败 | `dynamic_form_invalid`、`card_declined_input`、`billing_address_invalid` | 不调用或终止 Provider，提示用户修正输入。 |
| 余额/额度不足 | `insufficient_funds`、`limit_exceeded` | 用户可明确选择其他支付方式；同方式重试需用户确认。 |
| 认证失败 | `authentication_failed`、`3ds_failed`、`sca_abandoned` | 根据 Provider 能力允许重新认证；用户也可明确选择其他支付方式。 |
| 风控拒绝 | `risk_rejected`、`aml_blocked`、`sanctioned_country` | 禁止自动重试，进入 failed 或 review，记录审计原因。 |
| Provider 临时故障 | `provider_timeout`、`provider_unavailable`、`rate_limited` | 进入 retryable_failed 或 processing，触发补偿查询。 |
| 网络/回调不确定 | `network_unknown`、`callback_missing`、`query_required` | 不允许立即创建新支付，先查询 Provider 防重复扣款。 |
| 配置错误 | `config_missing`、`currency_not_configured`、`merchant_account_invalid` | 禁用支付方式，告警后台，不提示用户重试同方式。 |
| 规则不支持 | `currency_unsupported`、`country_unsupported`、`amount_too_low`、`amount_too_high` | checkout 禁用该方式并显示原因。 |
| Provider 明确拒绝 | `provider_declined`、`bank_declined`、`network_declined` | 用户可明确选择其他支付方式；同方式是否重试由 decline code 决定。 |

失败记录必须包含 Provider 原始 code、归一化 code、是否用户可见、是否客服可见、是否可重试、是否允许用户重新选择支付方式、是否需要查询补偿、是否进入风控/人工审核。

### 25.2 重试策略

重试不是简单再次调用 Provider，必须区分用户重试、系统补偿、Provider 回调重放和后台人工重跑。

- `retryable_failed`：允许用户明确重新选择支付方式；是否允许同方式重试由 Provider decline code 和风控规则决定。
- `processing` 或 `query_required`：禁止用户立即重新支付同一 Payable；先执行 Provider 查询，避免双扣。
- `authentication_required` / `authentication_failed`：允许重新认证时复用原 intent；超过次数后关闭或要求用户明确重新选择支付方式。
- `provider_timeout`：创建支付请求超时时，如果 Provider 未返回 reference，必须按幂等 key 查询或等待回调后裁决。
- `rate_limited`：不允许前台高频重试；进入指数退避并影响 Provider health 可用性判断。
- `config_error`：禁止前台重试；后台修复配置并发布后新 checkout 才可用。
- 每个 intent 记录 retry_count、last_retry_at、next_retry_at、retry_reason、max_retry_count、retry_owner。
- 重试必须使用同一 idempotency scope 或新 scope 的明确原因；不能因为重试生成无法追踪的重复 transaction。

### 25.3 重新选择支付方式

支付失败或不可用时，`Weline_Payment` 不能自动切换 Provider，也不能自动切换支付方式。系统只允许重新计算可用支付方式并展示给用户，由用户明确选择新的支付方式。

- 创建 intent 前可以做正常支付路由，选择满足配置、scope、merchant account、币种、国家、Provider health 的通道；这是创建前决策，不是失败后的自动切换。
- intent 创建后，method_code、provider_code、merchant account、金额、币种、条款、动态表单快照固定；不能在同一个 intent 内换 Provider 或换支付方式。
- 用户重新选择支付方式时，必须关闭或保留原 intent 为失败、取消或待查询状态，并创建新的 intent 或新的明确 attempt。
- 重新选择前必须重新计算 surcharge、FX quote、terms、dynamic form、支持国家/币种、Provider capability 和 scope effective config。
- timeline 必须记录原支付方式、失败原因、用户重新选择的新支付方式、新 intent/attempt、金额是否变化、条款是否变化。
- 风控拒绝、AML/制裁、KYC 失败不能引导重新选择支付方式；只能禁用或进入 review。
- 旧 attempt 若处于 `processing` 或 Provider 状态不确定，必须先查询补偿；在确认不会重复扣款前，不允许创建会立即扣款的新 attempt，除非用户明确接受重复支付风险且配置允许。
- 旧 attempt 晚到成功时，进入 `late_success_review` 或 suspense account；默认动作是原路退款或人工 review，不自动合并到新 intent。

### 25.4 支付网络规则

Provider capability 之下还要有支付网络规则，承载 Visa/Mastercard、Pix、UPI、SEPA、ACH、PayPal、Klarna、Afterpay 等网络差异。

- 网络规则至少包含退款窗口、争议窗口、授权有效期、capture 截止、mandate 要求、descriptor 限制、surcharge 限制、最小/最大金额、工作日规则、回调延迟、退票/ACH return 规则。
- Provider 支持某币种不代表某网络在当前国家可用；checkout 可用性必须同时匹配 Provider capability、网络规则和后台配置。
- 网络规则命中后必须进入 intent 快照，避免后续网络规则变化影响历史退款和争议处理。
- 本地支付方式有票据或 QR 过期时间时，Provider 返回值优先于后台默认 TTL。

### 25.5 保存支付工具生命周期

保存支付工具不只是绑定和解绑，还要覆盖 token、mandate 和卡生命周期。

- 支付工具状态：`active`、`expiring`、`expired`、`revoked_by_user`、`revoked_by_provider`、`replaced`、`migration_required`、`suspended`。
- 卡组织自动更新卡信息时，必须记录旧尾号/新尾号脱敏快照、Provider event、更新时间和用户通知策略。
- Provider token 迁移或 merchant account 切换时，旧 token 不得直接复用到新主体，必须走 Provider 支持的迁移流程或要求用户重新授权。
- mandate 失效、条款版本升级、授权范围变化时，off-session 支付必须停止并要求重新授权。
- 历史 off-session 支付按创建时 mandate 快照处理；撤销后的新扣款禁止执行。

### 25.6 支付凭证与文件安全

支付凭证、KYC 文件、争议证据、退款附件必须进入统一文件治理，不允许作为普通公开附件处理。

- 文件必须记录 file_code、owner、payable、intent/refund/dispute、purpose、sensitivity_level、retention_policy、uploaded_by、virus_scan_status、access_policy。
- 病毒扫描失败的文件不能被 Provider 提交、客服下载或用于审核，只能删除或重新上传。
- 后台预览必须脱敏或加水印；下载需要 ACL、原因、审计、过期链接和访问日志。
- 文件保留期按国家、业务类型、争议窗口、税务/法务要求计算；到期后按策略删除、脱敏或法务保留。

### 25.7 批量操作治理

批量退款、批量关闭 intent、批量重放 webhook、批量导出账单、批量归档都属于高风险操作。

- 批量操作必须先 dry-run，输出影响 Payable 数、intent 数、金额、币种、支付方式、失败原因和风险提示。
- 超过阈值的批量操作需要审批；高金额或跨财务期间的操作需要双人复核。
- 批量执行必须分片、限流、可暂停、可恢复、可取消，单条失败不能导致已成功项回滚不清。
- 每条子任务都要有 idempotency key、结果、失败原因、重试次数和审计记录。
- 批量导出必须受 ACL、脱敏、导出水印、过期链接和下载审计控制。

### 25.8 多租户和多商户隔离

多租户/多商户隔离必须覆盖配置、密钥、webhook、支付工具、报表、对账、payout 和日志。

- 所有核心表查询默认带 environment、scope、merchant_account、legal_entity 或 tenant scope。
- 密钥、webhook secret、Provider token、mandate、payment instrument 不能跨 merchant account 复用。
- dashboard、导出、客服时间线和对账文件必须按 ACL 限制 scope；没有跨租户权限时不能通过 code 猜测访问。
- webhook endpoint 必须绑定 environment + merchant_account + method_code；收到事件后校验 endpoint、secret 版本和 merchant account。
- payout account、settlement owner、invoice issuer、legal entity 必须进入 intent/settlement 快照。

### 25.9 异常资金暂挂账户

超付、未知入账、重复支付、晚到成功、无 Payable 入账、Provider 状态不一致都不能直接改成 paid 或静默丢弃，必须进入 suspense account。

- suspense 状态：`open`、`matched`、`refunding`、`refunded`、`recognized_income`、`written_off`、`closed`。
- suspense 记录必须包含金额、币种、Provider reference、merchant account、可能的 Payable 候选、来源事件、匹配置信度、处理建议。
- 可执行动作：匹配到 Payable、原路退款、替代退款、确认为收入、转人工调查、关闭为无效。
- 每个动作必须写 ledger、audit、timeline；涉及收入确认或核销时必须受财务期间和审批控制。

### 25.10 对账文件导入安全

对账来源可能是 CSV、Excel、SFTP、API 拉取或 Provider 后台导出，必须先建导入批次再入库。

- 导入批次记录 source、file_hash、schema_version、encoding、timezone、currency_precision、row_count、amount_checksum、imported_by、status。
- 重复文件通过 file_hash + provider + merchant_account + period 阻断；允许重导时必须生成新 batch 并关联原 batch。
- 字段映射版本必须可追溯；Provider 格式变化时不能直接覆盖旧 mapping。
- 部分导入失败必须保留成功/失败行明细，支持修正后续导，不允许半导入后无审计。
- 金额精度、负数、时区、结算日、手续费、税费、退款、拒付必须进入校验规则。

### 25.11 数据删除与法务保留

隐私删除请求和支付审计保留会冲突，需要分层处理。

- 可删除：checkout 临时输入、非必要设备信息、过期未使用 session、可匿名化 PII。
- 可脱敏保留：raw payload 中的 PII、动态表单、支付凭证预览、客服备注。
- 必须保留：ledger、transaction、refund、settlement、payout、税务和争议所需证据，但展示时脱敏。
- legal hold 命中争议、税务审计、监管调查、财务锁账时，保留优先于删除；必须记录 hold reason、owner、expires_at、解除审批。
- 用户导出数据必须排除其他租户、其他 payer、Provider 密钥、内部风控规则和不可披露审计信息。

### 25.12 开发者错误防护

支付模块开发者必须被接口和工具保护，不能靠人工约定避免事故。

- Provider 返回未声明能力的结果、非法状态、错误币种、缺少 Provider reference、缺少 idempotency key 时，内核必须拒绝并记录 contract violation。
- Provider 声明支持退款但 `refund()` 未实现或总是 unsupported，config lint 和 contract test 必须失败。
- Provider 漏写动态字段脱敏、错误码归一化、回调签名校验、query 补偿能力时，不允许进入 live。
- 核心状态机是唯一状态入口；Provider 不能直接写 paid/refunded。
- contract test stub 必须覆盖成功、失败、超时、重复回调、乱序回调、签名失败、退款失败、unsupported capability 和非法响应。

### 25.13 后台配置审批责任链

不同配置风险不同，不能只有一个“保存配置”权限。

- 密钥修改、live 发布、启用退款、开启新国家/币种、修改最大折扣比例、开启 surcharge、开启自动捕获、修改 webhook endpoint、开启 payment link，都需要独立 ACL。
- 高风险配置支持审批流：提交、测试连接、diff、审批、发布、回滚。
- 审批记录必须包含变更前后 diff、影响范围、发布人、审批人、原因、时间、连接测试结果和 config lint 结果。
- live 环境密钥保存时只允许写入和轮换，不允许后台明文查看或导出。

### 25.14 灾备和恢复演练

支付恢复工具必须提前设计，不能等事故发生后临时写脚本。

- 场景：Provider 回调丢失、队列积压、重复通知、ledger 校验失败、对账文件误导入、归档误配置、webhook secret 错误、Provider 大面积超时。
- 恢复工具必须支持 dry-run、影响评估、分批执行、幂等、审计和回滚策略说明。
- `payment_recovery_run` 记录场景、输入条件、影响范围、执行人、审批人、执行结果、失败项、后续动作。
- 定期演练输出恢复耗时、失败率、数据差异、未恢复项和改进项。

### 27.0 Payment worker A 持久化模型补齐记录

本轮仅补齐 `PaymentCheckoutSession`、`PaymentRefund`、`PaymentLedger`、`PaymentIdempotency`、`PaymentLock`、`PaymentAllocation` 六个持久化模型的必要字段、状态常量、索引和最小 helper 方法。`GAP-PAY-004`、`GAP-PAY-005`、`GAP-PAY-009` 中的模型缺口已推进到 ORM 声明层，但服务编排、状态机写入、ledger 入账、锁/idempotency 服务、`setup:upgrade` schema 同步和 fake Browser 验收仍属于后续任务。

### 27.1 明确排除的设计

- 失败后不做 Provider 自动切换，不做支付方式自动切换；只能重新计算可用方式并由用户明确选择。
- intent 创建后不在同一 intent 内替换 `method_code`、`provider_code`、merchant account、金额、币种、条款或动态表单快照。
- scope 配置编辑冲突不做自动合并、隐式覆盖或冲突自动处理；冲突必须显式阻断、提示重新读取和重新提交。
- 旧支付迁移不做双写兜底；切换窗口内必须冻结旧写入并通过迁移映射、只读归档和新内核接管完成。

### 28.1 fake 模式浏览器验收脚本

fake 模式验收必须在独立 WLS 实例上执行，禁止使用默认 `9501`。

1. 准备 fake 数据和 Demo/Fake Provider，发布 sandbox `default.default.default` scope 配置。
2. 启动实例：`php bin/w server:start -p 9506 -n ai-test-payment-fake`。
3. HTTP 冒烟：`php bin/w http:request /checkout` 和 `php bin/w http:request /payment/backend/method`。
4. 用 Codex Browser 打开 `http://127.0.0.1:9506/checkout`，完成商品下单、支付方式选择、动态表单、条款同意、fake 支付成功。
5. 在同一实例验证失败、取消、Provider processing、重新选择支付方式、部分退款和后台 timeline。
6. 停止实例：`php bin/w server:stop -n ai-test-payment-fake`。
