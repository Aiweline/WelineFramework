# Weline 信任通信与互通授权模块产品开发方案

## 1. 产品定位

本方案面向 WelineFramework 核心能力，目标是在框架层提供统一的信任通信、账号互通和授权登录能力。它服务于 `app`、`skill`、`bbs.a2a`、应用商城、论坛等互信域名，让这些站点可以通过用户授权完成一键登录、账号注册、账号关联、Token 调用和授权撤销。

本方案留在本地仓库执行，属于框架核心支持能力，不直接承载论坛发帖、板块、积分、W 币消费等社区业务。社区模块只消费本模块提供的身份、授权、账号关联和安全能力。

## 2. 业务目标

- 用户只需一个 Weline 账号，即可登录论坛、应用商城、Skill 平台和其它可信子站。
- 论坛账号可以通过授权直接注册或关联应用商城账号，应用商城账号也可以反向授权关联论坛账号。
- 各可信应用不共享密码，不跨域传递明文凭据，只使用短期授权码、签名 Token 和受控用户资料。
- 用户可以在个人中心查看、管理、撤销全部授权应用和账号绑定。
- 高风险授权必须支持两步验证、权限范围提示、审计日志和可撤销能力。

## 3. 模块边界

### 3.1 建议归属

优先在现有 `Weline_Multipass` 基础上升级，形成“信任通信与互通授权”能力。若需要更清晰命名，可新增核心模块 `Weline_IdentityBridge`，但第一阶段建议先复用已有模块，避免重复建设。

### 3.2 复用现有能力

- `Weline_Multipass`：已有跨站 Token 登录基础，继续承载可信站点配置、站点密钥和服务端快速登录通道。
- `Weline_Api`：已有第三方应用、授权码、Installation、Scope、Token 模型，可作为标准授权流底座。
- `Weline_Customer`：作为前台账号、注册、登录、挑战页和个人中心基础。
- `Weline_TwoFactorAuth`：作为敏感授权、账号绑定、Token 管理的二次验证能力。
- `Weline_AppStore`：作为应用商城账号绑定、下载、授权安装等消费方。

### 3.3 不在本模块实现

- 不实现论坛板块、主题帖、回复和版主管理。
- 不实现 W 币业务账本和商城抵扣规则。
- 不实现插件审核和下载中心业务。
- 不为任意外部域名开放互信，只支持后台配置的可信应用和可信域名。

## 4. 核心用户角色

- 普通用户：授权登录、创建关联账号、管理授权、开启两步验证。
- 开发者用户：授权应用商城、上传插件、管理 API Token、绑定 Skill 平台。
- 应用管理员：注册可信应用、配置回调域名、配置 Scope、查看审计日志。
- 安全管理员：撤销异常授权、禁用应用、查看风险登录和 Token 使用记录。

## 5. 核心场景

### 5.1 论坛账号授权注册应用商城账号

1. 用户在应用商城点击“使用论坛账号登录”。
2. 应用商城跳转到统一授权页，携带 `client_id`、`redirect_uri`、`scope`、`state`。
3. 用户未登录时先完成论坛或 Weline 账号登录。
4. 授权页展示应用名称、可信域名、请求权限、账号资料和风险提示。
5. 用户确认授权；若请求敏感权限，触发两步验证。
6. 授权服务发放短期 `authorization_code` 并跳回应用商城。
7. 应用商城使用 `client_id`、`client_secret`、`authorization_code` 换取 Token。
8. 应用商城调用 `userinfo` 获取基础资料；如果本地无账号，则创建本地商城账号。
9. 写入账号关联记录，用户进入应用商城个人中心。

### 5.2 应用商城反向关联论坛账号

1. 已登录商城用户进入“账号关联”。
2. 点击“关联论坛账号”。
3. 跳转论坛授权页，完成登录和授权。
4. 系统检查邮箱、手机号、外部用户 ID 是否冲突。
5. 无冲突则绑定；有冲突则要求用户登录原账号确认，不允许静默合并。

### 5.3 可信应用一键登录

1. 用户访问 `skill`、`app`、`bbs.a2a` 等可信应用。
2. 应用发现未登录，发起标准授权跳转。
3. 用户授权后应用创建本地会话。
4. 应用只保存本地会话和外部身份绑定，不保存 Weline 密码。

### 5.4 用户撤销授权

1. 用户进入个人中心“安全与授权”。
2. 查看已授权应用、权限、最近使用时间和登录设备。
3. 点击撤销授权。
4. 系统撤销 Refresh Token、Access Token、Installation 状态。
5. 下游应用下一次调用接口时收到授权失效响应。

## 6. 权限 Scope 设计

第一阶段建议提供以下 Scope：

| Scope | 说明 | 风险级别 |
| --- | --- | --- |
| `profile.basic` | 读取昵称、头像、用户 ID | 低 |
| `profile.email` | 读取邮箱 | 中 |
| `profile.security` | 读取 2FA 状态，不读取密钥 | 中 |
| `appstore.account` | 创建或关联应用商城账号 | 中 |
| `community.account` | 创建或关联论坛账号 | 中 |
| `plugin.upload` | 代表用户提交插件包 | 高 |
| `skill.install` | 代表用户安装 Skill | 高 |
| `wallet.read` | 读取 W 币余额 | 中 |
| `wallet.redeem` | 发起 W 币抵扣 | 高 |

高风险 Scope 必须触发二次确认；涉及 `wallet.redeem`、`plugin.upload`、`skill.install` 时建议要求已开启两步验证。

## 7. 数据模型建议

### 7.1 可信应用表

保存应用名称、应用类型、`client_id`、`client_secret_hash`、可信域名、回调地址、状态、创建人、最近更新时间。

### 7.2 授权安装表

保存应用 ID、主体类型、主体 ID、授权状态、授权时间、撤销时间、最近使用时间。主体类型可为 `customer`、`backend_user`、`global`。

### 7.3 授权 Scope 表

保存 Installation ID、Scope、访问模式、授权来源、过期时间。

### 7.4 账号关联表

保存本地用户 ID、外部应用 ID、外部主体 ID、外部账号展示名、邮箱指纹、绑定状态、绑定时间、最后登录时间、解绑时间。

### 7.5 Token 表

保存 Access Token、Refresh Token、Token 类型、Installation ID、过期时间、撤销时间、轮换批次、最近使用 IP 和 User-Agent 摘要。

### 7.6 审计日志表

保存授权、换取 Token、刷新 Token、撤销授权、账号绑定、二次验证、失败原因和操作者信息。

## 8. 后端接口建议

### 8.1 授权页接口

- `GET /identity/authorize`：展示授权页。
- `POST /identity/authorize/confirm`：用户确认授权，返回授权码跳转。
- `POST /identity/authorize/cancel`：用户取消授权。

### 8.2 Token 接口

- `POST /identity/token`：授权码换 Access Token 和 Refresh Token。
- `POST /identity/token/refresh`：刷新 Token。
- `POST /identity/token/revoke`：撤销 Token。

### 8.3 用户资料接口

- `GET /identity/userinfo`：返回已授权范围内的用户资料。
- `GET /identity/bindings`：返回当前用户账号绑定列表。
- `POST /identity/bindings/revoke`：解绑外部账号。

### 8.4 管理接口

- 可信应用列表、创建、编辑、禁用。
- Scope 目录维护。
- 授权审计查询。
- 风险授权撤销。

## 9. 前端与 UI 设计

### 9.1 授权确认页

页面重点是信任与可理解，不做复杂营销式页面。

必须展示：

- 当前登录账号头像、昵称、邮箱遮罩。
- 请求授权的应用名称、图标、可信域名。
- 请求权限列表，按低、中、高风险分组。
- 回调域名提示。
- 继续授权、取消两个明确按钮。
- 高风险权限的二次验证入口。

交互要求：

- 默认不勾选高风险权限，除非业务必须。
- 错误提示靠近问题区域，不只在顶部显示。
- Token、密钥、邮箱完整值不在前端暴露。
- 移动端优先保持“应用身份”和“授权按钮”在首屏可见。

### 9.2 个人中心安全页

建议合并为“安全与授权”信息架构：

- 两步验证：开启状态、恢复码、最近验证时间。
- 已授权应用：应用、权限、最近使用、撤销。
- 已绑定账号：论坛、商城、Skill、BBS 等账号绑定状态。
- 登录设备：设备、地区、时间、退出。
- API Token：开发者 Token 和过期状态。

### 9.3 后台管理页

后台使用表格 + 详情抽屉结构：

- 应用列表支持状态、类型、域名、创建人筛选。
- 应用详情展示回调地址、Scope、密钥轮换入口。
- 审计日志支持按用户、应用、动作、结果筛选。

## 10. 安全策略

- 生产环境强制 HTTPS。
- `redirect_uri` 必须精确匹配，不允许模糊匹配。
- 授权请求必须带 `state`，用于防 CSRF 和回跳校验。
- 公共客户端支持 PKCE，避免只依赖 `client_secret`。
- `client_secret` 只保存哈希值，创建时仅显示一次。
- Access Token 短期有效；Refresh Token 每次使用后轮换。
- 撤销授权必须立即使 Refresh Token 失效。
- 账号合并必须用户显式确认，不允许凭邮箱自动合并。
- 敏感 Scope 授权、W 币抵扣、插件上传必须支持两步验证。

## 11. 里程碑

### MVP

- 可信应用注册和后台管理。
- 标准授权页。
- 授权码换 Token。
- `userinfo`。
- 账号自动创建和绑定。
- 用户授权管理页。

### V1

- Refresh Token 轮换。
- 高风险 Scope 二次验证。
- 授权审计日志。
- 密钥轮换。
- 授权撤销通知下游应用。

### V2

- 统一登出。
- 设备管理。
- 风险登录提醒。
- 跨站账号冲突处理向导。
- 应用级风控策略。

## 12. 验收标准

- 可信应用可以完成授权登录、Token 换取和用户资料读取。
- 未登录用户会先进入登录流程，登录后继续授权，不丢失 `state`。
- 已开启两步验证的用户在高风险授权时必须完成验证。
- 撤销授权后，下游应用无法继续刷新 Token。
- 账号关联冲突不会静默合并账号。
- 后台可查看授权日志和失败原因。
- 授权页、个人中心、安全页在移动端和桌面端均可正常操作。

## 13. 关键风险

- 账号自动创建如果缺少冲突处理，可能造成错误合并。
- W 币、插件上传、商城抵扣属于高风险授权，不能只依赖前端登录态。
- 多域名互信如果配置过宽，会扩大攻击面。
- Token 刷新和撤销如果没有统一审计，后续排查困难。

## 14. 推荐实施顺序

1. 梳理现有 `Weline_Multipass` 和 `Weline_Api` 的授权模型，确认复用点。
2. 先落可信应用、Scope、授权页和 Token 换取。
3. 再落账号关联和用户中心授权管理。
4. 最后接入 AppStore、Skill、BBS、社区等消费方。
5. 高风险能力上线前补齐两步验证、审计、撤销和异常处理。
