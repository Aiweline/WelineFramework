# 模块代码与数据库一致性回滚

## 责任边界

- `Weline_Database` 是版本游标、Schema checkpoint、迁移、备份、补偿和回滚编排的唯一所有者。
- `Weline_ModuleManager` 必须依赖 `Weline_Database`，但只注入 `ModuleRollbackManagerInterface`，负责后台展示、ACL 和调用。它不保存另一套 Model 版本，也不实现回滚逻辑。
- `Weline_Framework`、`Weline_Database`、`Weline_ModuleManager` 是运行基础设施，不允许从 ModuleManager 后台自回滚；包含它们的级联变更应转交 `Weline_Deploy` 整站版本回滚。

## 稳定状态

一个模块只有在下列状态同时成立时才能创建回滚计划：

1. 代码制品 `etc/module.php` 版本等于模块注册版本。
2. `module_version.current_version` 等于已安装迁移及 Schema checkpoint 的实际游标。
3. ModuleManager 的 `weline_module.code_version/database_version/schema_status` 是上述状态的同步查询投影。
4. 目标代码的只读 SchemaDiff 验证为零。SQLite 不持久化表注释，因此注释不算 SQLite 物理漂移；唯一索引由索引层比较，不重复当作列变更。

任一项不成立时，`createPlan()` 必须返回 blocker，先审计/修复漂移，不能通过“只改版本号”绕过。

## 公开接口

`Weline\Database\Api\ModuleRollbackManagerInterface` 发布五个能力：

- `getModuleState(moduleName)`：读取代码、注册、Database 游标和 blocker。
- `listTargets(moduleName)`：列出可用精确制品及可回滚性。
- `createPlan(moduleName, targetVersion)`：生成有过期时间和哈希的只读级联预检计划。
- `start(planId, planHash, operator)`：重新验证计划后启动独立 worker。
- `getOperation(operationId)`：读取阶段、逐模块明细、备份、冲突和补偿结果。

操作状态为 `planned -> queued -> running -> succeeded`。失败后进入 `compensating -> failed_recovered`；补偿也失败时为 `manual_recovery_required`，保持维护状态并返回备份 ID 与恢复命令。

## 迁移与 Model 版本

- 迁移文件保持平铺在模块的迁移目录；运行时用迁移对象 `getVersion()` 分组，用完整路径和 ObjectManager 实例化。
- SchemaDiff 记录模块语义版本、batch、sequence、操作类型、Model、正/反向 DDL、前后指纹、checksum 和 operation ID。每个模块版本建立不可伪造的 Schema checkpoint。
- 旧时间戳记录视为 `legacy_unverified`；只有制品、checksum、反向链和备份能力都可验证时才允许自动回滚。
- MySQL/MariaDB DDL 可隐式提交，不宣称 DDL 事务原子性。结构操作依靠持久化日志、反向 DDL 和正向补偿；普通数据更新仍用数据库事务。

## 脚本迁移的数据保全契约

需要自动回滚的脚本迁移必须显式实现
`Weline\Framework\Database\Migration\RollbackBackupStrategyInterface`：

- `none`：作者明确确认不会丢数据，但仍必须提供 `reason`。
- `column`：回滚立即备份指定列的非空值，支持单主键和复合主键。
- `table`：回滚立即备份指定表的完整结构和数据。
- 如果回滚依赖升级前备份，设置 `requires_forward_backup=true` 并声明 `forward_backup_types`。

旧脚本仍可升级，但缺少该契约时自动回滚预检必须阻断。绕过协调编排直接调用脚本回滚也必须阻断。

重新升级后的恢复策略是：只在主键匹配且目标值为空时恢复；值不同则写冲突记录，绝不覆盖现有值。

## 制品与执行顺序

1. 从 `ModuleArtifactProviderInterface` 列出版本，优先 AppStore 精确版本包，本地不可变升级快照兜底。
2. 固定根模块目标版本，为反向依赖求解“不高于当前版本的最高兼容组合”。
3. 执行前一次性暂存并验证所有制品的模块名、版本、依赖、哈希和路径安全。
4. worker 重验计划，获取全局锁、进入维护状态、备份代码/数据，按反向依赖顺序回滚数据库。
5. 原子切换全部代码目录，一次性刷新注册、路由和命令，reload WLS。
6. 验证 Schema、代码/注册/Database 版本和运行状态后，最后提交版本游标。

## 后台接入

ModuleManager QueryProvider 名为 `moduleRollback`，业务请求只使用
`Weline.Api.resource()`，操作为 `listTargets/createPlan/start/getOperation`。
执行前管理员必须输入根模块名和目标版本；QueryProvider 会核对计划、登录用户、独立 ACL 和操作员标识。

## 适配器验证

`DatabaseAdapterRollbackMatrixTest` 必须在四类引擎真实运行，不允许只在 SQLite 上通过：

| 适配器 | 当前本地验证基线 |
|---|---|
| SQLite | PHP PDO SQLite |
| PostgreSQL | 16.14 |
| MySQL | 8.4.10 |
| MariaDB | 11.4.12 |

矩阵覆盖建表、增/改/删列、保留数据、增/删索引、增/删外键、完整结构导出、删表/重建和反向 DDL。CI 定义位于 `.github/workflows/database-rollback-matrix.yml`。

## 验证命令

```bash
php bin/w phpunit:run --module=Weline_Database --no-coverage

WELINE_DB_MATRIX_DRIVER=sqlite vendor/bin/phpunit --bootstrap app/bootstrap.php \
  app/code/Weline/Database/test/DatabaseAdapterRollbackMatrixTest.php
```

PostgreSQL/MySQL/MariaDB 执行同一测试文件，通过
`WELINE_DB_MATRIX_DRIVER/HOST/PORT/DATABASE/USERNAME/PASSWORD` 指向专用临时测试库。
