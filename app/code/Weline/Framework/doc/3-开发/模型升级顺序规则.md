# 模型升级顺序规则

## 概述

在系统升级过程中，模型的安装和升级顺序非常重要。如果模型之间存在依赖关系，错误的升级顺序可能导致数据库表创建失败或其他错误。

为了确保模型按照正确的顺序进行升级，框架实现了基于**目录层级深度**和**类名长度**的自动排序机制。

## 排序规则

模型升级顺序按照以下优先级规则进行排序（优先级从高到低）：

### 1. 目录层级深度（第一优先级）

**规则：目录层级越浅（深度越小）的模型越先升级**

目录层级深度通过类命名空间中的反斜杠（`\`）数量来计算。

**示例：**

```
Weline\Eav\Model\EavEntity                    // 层级深度：3（Model）
Weline\Eav\Model\EavAttribute                 // 层级深度：3（Model）
Weline\Eav\Model\EavAttribute\Type            // 层级深度：4（Model\EavAttribute）
Weline\Eav\Model\EavAttribute\Type\Value      // 层级深度：5（Model\EavAttribute\Type）
```

**升级顺序：**
1. `Weline\Eav\Model\EavEntity`（层级 3）
2. `Weline\Eav\Model\EavAttribute`（层级 3）
3. `Weline\Eav\Model\EavAttribute\Type`（层级 4）
4. `Weline\Eav\Model\EavAttribute\Type\Value`（层级 5）

### 2. 类名长度（第二优先级）

**规则：在相同目录层级下，类名越短的模型越先升级**

当两个模型的目录层级深度相同时，按照类名的字符长度进行排序。

**示例：**

```
Weline\Eav\Model\EavEntity        // 层级：3，长度：32
Weline\Eav\Model\EavAttribute      // 层级：3，长度：35
```

**升级顺序：**
1. `Weline\Eav\Model\EavEntity`（层级 3，长度 32）
2. `Weline\Eav\Model\EavAttribute`（层级 3，长度 35）

### 3. 字母顺序（第三优先级）

**规则：在相同层级和长度下，按字母顺序排序**

当两个模型的目录层级深度和类名长度都相同时，按照字母顺序进行排序，确保排序的稳定性。

**示例：**

```
Weline\Eav\Model\EavAttribute\Type\Value      // 层级：5，长度：45
Weline\Eav\Model\EavAttribute\Type\Option    // 层级：5，长度：45
```

**升级顺序：**
1. `Weline\Eav\Model\EavAttribute\Type\Option`（按字母顺序，O 在 V 之前）
2. `Weline\Eav\Model\EavAttribute\Type\Value`

## 为什么需要这个规则？

### 问题场景

在 EAV（Entity-Attribute-Value）模型中，存在以下依赖关系：

1. `EavEntity` - 实体模型（基础表）
2. `EavAttribute` - 属性模型（依赖实体）
3. `EavAttribute\Type` - 属性类型模型（依赖属性）
4. `EavAttribute\Type\Value` - 属性值模型（依赖属性和类型）

如果 `Value` 模型在 `EavAttribute` 和 `EavAttribute\Type` 之前升级，可能会因为依赖的表不存在而失败。

### 解决方案

通过按照目录层级深度和类名长度排序，确保：

- **基础模型先升级**：层级较浅的模型（如 `EavEntity`）会先于层级较深的模型（如 `EavAttribute\Type\Value`）升级
- **依赖关系正确**：被依赖的模型会在依赖它的模型之前升级
- **稳定性**：排序规则明确，不会因为文件系统顺序或其他因素导致不一致

## 实际应用示例

假设有以下模型结构：

```
Weline\Eav\Model\
├── EavEntity.php                    // 层级 3
├── EavAttribute.php                 // 层级 3
└── EavAttribute\
    ├── Type.php                     // 层级 4
    ├── Option.php                   // 层级 4
    └── Type\
        └── Value.php                // 层级 5
```

**升级顺序：**

1. `Weline\Eav\Model\EavEntity`（层级 3，长度 32）
2. `Weline\Eav\Model\EavAttribute`（层级 3，长度 35）
3. `Weline\Eav\Model\EavAttribute\Option`（层级 4，长度 42）
4. `Weline\Eav\Model\EavAttribute\Type`（层级 4，长度 41）
5. `Weline\Eav\Model\EavAttribute\Type\Value`（层级 5，长度 45）

## 开发者注意事项

### 1. 模型目录结构规划

在创建模型时，应该遵循以下原则：

- **基础模型放在 `Model` 目录下**：如 `Model\EavEntity.php`
- **依赖模型放在子目录中**：如 `Model\EavAttribute\Type\Value.php`
- **目录层级反映依赖关系**：被依赖的模型应该在更浅的目录层级

### 2. 避免循环依赖

虽然排序规则可以处理大部分依赖关系，但**无法处理循环依赖**。如果模型之间存在循环依赖，需要重新设计模型结构。

### 3. 测试升级顺序

在开发新模型时，建议：

1. 检查模型的依赖关系
2. 确保模型目录结构符合依赖关系
3. 在测试环境中验证升级顺序是否正确

### 4. 特殊情况处理

如果模型之间存在特殊的依赖关系，而目录结构无法反映这种关系，可以考虑：

- 调整目录结构以反映依赖关系
- 在模型的 `install()` 或 `upgrade()` 方法中处理依赖检查
- 使用数据库迁移脚本明确处理依赖关系

## 实现细节

排序逻辑在 `Weline\Framework\Database\Model\ModelManager::sortModelsByPriority()` 方法中实现：

```php
private function sortModelsByPriority(array $model_classes): array
{
    usort($model_classes, function ($a, $b) {
        // 第一优先级：目录层级深度
        $depthA = substr_count($a, '\\');
        $depthB = substr_count($b, '\\');
        if ($depthA !== $depthB) {
            return $depthA <=> $depthB;
        }
        
        // 第二优先级：类名长度
        $lengthA = strlen($a);
        $lengthB = strlen($b);
        if ($lengthA !== $lengthB) {
            return $lengthA <=> $lengthB;
        }
        
        // 第三优先级：字母顺序
        return strcmp($a, $b);
    });
    
    return $model_classes;
}
```

## 总结

模型升级顺序规则确保了：

1. ✅ **基础模型优先**：层级较浅的模型先升级
2. ✅ **依赖关系正确**：被依赖的模型在依赖它的模型之前升级
3. ✅ **排序稳定**：排序规则明确，结果可预测
4. ✅ **易于理解**：规则简单直观，开发者容易遵循

遵循这些规则可以避免模型升级过程中的依赖问题，确保系统升级的顺利进行。
