# SEO模块队列化架构说明

## 架构改进

SEO模块已从直接事件触发改为队列化处理架构，大幅提升性能，避免阻塞主流程。

## 核心改进

### 1. 性能优化

**之前的问题**：
- Store保存时直接触发SEO处理
- 包含Feed收集、关键词提取、数据库写入等耗时操作
- 阻塞主流程，影响用户体验

**现在的方案**：
- Store保存时仅将任务入队（轻量级操作）
- 实际处理由Cron任务异步执行
- 主流程快速返回，不阻塞

### 2. 任务队列表

**表名**: `weline_seo_task`

**核心字段**：
- `task_type`: 任务类型（feed_generate, push_urls, keyword_extract）
- `subject_type`: 主体类型（store, website等）
- `subject_id`: 主体ID
- `payload`: 任务数据（JSON格式）
- `status`: 任务状态（pending, processing, done, error）
- `priority`: 优先级（1低，5普通，10高）
- `attempts`: 尝试次数
- `max_attempts`: 最大尝试次数
- `scheduled_at`: 计划执行时间

**特性**：
- 支持任务优先级
- 自动重试机制（失败后延迟重试）
- 任务状态跟踪
- 错误信息记录

### 3. 轻量级Observer

**文件**: `Observer/StoreSaveAfter.php`

**改进**：
- 不再直接处理SEO逻辑
- 仅将任务信息写入队列表
- 检查重复任务，避免重复入队
- 快速返回，不阻塞主流程

### 4. 任务处理器

**文件**: `Service/TaskProcessor.php`

**功能**：
- 统一处理各种SEO任务
- Feed生成处理
- URL推送处理
- 关键词提取处理
- 错误处理和重试

### 5. Cron任务

#### FeedGenerator（Feed生成任务）

**执行频率**: 每10分钟
**功能**: 
- 消费 `feed_generate` 类型的任务
- 批量处理（每次100个）
- 生成Feed并更新SEO主体和关键词

#### UrlPusher（URL推送任务）

**执行频率**: 每天凌晨2点
**功能**:
- 收集需要推送的URL
- 按平台分组创建推送任务
- 批量推送（每批100个URL）
- 支持多平台推送

## 数据流

```
Store保存
    ↓
Observer（轻量级）
    ↓
SeoTask表（入队）
    ↓
Cron定时消费
    ↓
┌─────────────────┴─────────────────┐
↓                                   ↓
FeedGenerator                  UrlPusher
(每10分钟)                      (每天2点)
    ↓                                   ↓
TaskProcessor                  TaskProcessor
    ↓                                   ↓
SeoSubject/Keyword              SearchEngine
(更新数据)                      (推送URL)
```

## 任务类型

### 1. feed_generate（Feed生成）

**触发时机**: Store/Website保存后
**处理内容**:
- 收集Feed数据
- 创建/更新SEO主体
- 提取关键词
- 保存关键词

### 2. push_urls（URL推送）

**触发时机**: 定时任务（每天凌晨2点）
**处理内容**:
- 收集需要推送的URL
- 按平台分组
- 批量推送到搜索引擎

### 3. keyword_extract（关键词提取）

**触发时机**: 手动触发或特定场景
**处理内容**:
- 从文本中提取关键词
- 保存关键词到数据库

## 性能优势

1. **主流程不阻塞**: Store保存操作快速返回
2. **批量处理**: Cron任务批量处理，提升效率
3. **异步执行**: SEO处理在后台异步执行
4. **可扩展**: 支持任务优先级和重试机制
5. **可监控**: 任务状态可追踪，便于监控和调试

## 配置建议

### Feed生成频率

- **默认**: 每10分钟
- **高负载**: 可调整为每5分钟
- **低负载**: 可调整为每30分钟

### URL推送频率

- **默认**: 每天凌晨2点
- **高频需求**: 可调整为每天多次（如每天2点、14点）
- **低频需求**: 可调整为每周一次

### 批量大小

- **Feed生成**: 每次100个任务
- **URL推送**: 每批100个URL

## 监控和维护

### 查看待处理任务

```sql
SELECT * FROM weline_seo_task 
WHERE status = 'pending' 
ORDER BY priority DESC, created_at ASC;
```

### 查看失败任务

```sql
SELECT * FROM weline_seo_task 
WHERE status = 'error' 
ORDER BY updated_at DESC;
```

### 手动重试失败任务

可以通过后台或命令行工具将失败任务重置为pending状态，等待下次Cron执行。

## 注意事项

1. **任务去重**: Observer会检查是否已存在相同任务，避免重复入队
2. **重试机制**: 任务失败后会自动重试，最多尝试3次
3. **延迟重试**: 失败后延迟重试（尝试次数 * 5分钟）
4. **超时处理**: Cron任务有超时限制，防止长时间阻塞

## 后续优化建议

1. **任务优先级**: 可根据业务需求调整任务优先级
2. **批量优化**: 可根据服务器性能调整批量大小
3. **监控告警**: 可添加任务积压告警机制
4. **性能统计**: 可添加任务处理性能统计
