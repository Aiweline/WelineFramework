# SEO模块事件系统设计文档

## 概述

SEO模块采用符合国际标准的事件驱动架构，通过事件解耦模块间依赖，提升系统的可扩展性和可维护性。

## 设计原则

### 1. 事件命名规范

**格式**: `模块名::事件类型::事件名称`

**示例**:
- `Weline_Seo::domain::subject_created` - 领域事件
- `Weline_Seo::integration::task_enqueued` - 集成事件
- `Weline_Seo::application::trend_sync_completed` - 应用事件

### 2. 事件类型分类

#### Domain Events (领域事件)
- **定义**: 业务领域内的事件，表示业务状态的变化
- **特点**: 与业务逻辑紧密相关，通常在领域模型内部触发
- **示例**: `subject_created`, `keywords_extracted`, `suggestion_generated`

#### Integration Events (集成事件)
- **定义**: 跨模块/系统的事件，用于模块间通信
- **特点**: 用于解耦模块依赖，实现模块间协作
- **示例**: `feed_collect`, `task_enqueued`, `task_completed`

#### Application Events (应用事件)
- **定义**: 应用层事件，通常与系统操作相关
- **特点**: 用于应用层面的通知和协调
- **示例**: `trend_sync_completed`, `url_push_completed`

### 3. 事件数据契约

每个事件都有明确的数据契约（Data Contract），确保：
- **类型安全**: 字段类型明确定义
- **必填验证**: 必填字段自动验证
- **向后兼容**: 通过版本管理支持向后兼容

**契约定义示例**:
```php
'data_contract' => [
    'subject_id' => [
        'type' => 'integer',
        'required' => true,
        'description' => 'SEO主体ID',
    ],
    'keywords' => [
        'type' => 'array',
        'required' => true,
        'description' => '关键词列表',
    ],
]
```

### 4. 事件版本管理

- **格式**: 语义化版本（Semantic Versioning）
- **示例**: `1.0.0`, `1.1.0`, `2.0.0`
- **用途**: 支持事件演化和向后兼容

## 事件列表

### Domain Events

#### 1. Weline_Seo::domain::subject_created

**触发时机**: SEO主体创建时

**数据契约**:
- `subject_id` (integer, required): SEO主体ID
- `subject_type` (string, required): 主体类型
- `subject_entity_id` (integer, required): 主体实体ID
- `url` (string, optional): URL地址
- `title` (string, optional): 标题
- `description` (string, optional): 描述

**使用示例**:
```php
$eventDispatcher->dispatchSubjectCreated(
    $subjectId,
    'store',
    $storeId,
    ['url' => 'https://example.com', 'title' => '店铺标题']
);
```

#### 2. Weline_Seo::domain::keywords_extracted

**触发时机**: 关键词提取完成时

**数据契约**:
- `subject_id` (integer, required): SEO主体ID
- `keywords` (array, required): 关键词列表
- `source` (string, required): 关键词来源

**使用示例**:
```php
$eventDispatcher->dispatchKeywordsExtracted(
    $subjectId,
    ['关键词1', '关键词2'],
    'extracted'
);
```

### Integration Events

#### 1. Weline_Seo::integration::task_enqueued

**触发时机**: SEO任务入队时

**数据契约**:
- `task_id` (integer, required): 任务ID
- `task_type` (string, required): 任务类型
- `subject_type` (string, required): 主体类型
- `subject_id` (integer, required): 主体ID

**使用示例**:
```php
$eventDispatcher->dispatchTaskEnqueued(
    $taskId,
    'feed_generate',
    'store',
    $storeId
);
```

## 事件分发器

### EventDispatcher 服务

提供标准化的事件分发接口：

```php
use Weline\Seo\Service\EventDispatcher;

// 获取事件分发器
$eventDispatcher = ObjectManager::getInstance(EventDispatcher::class);

// 分发事件
$eventDispatcher->dispatch($event);
```

### 事件对象

所有事件都继承自 `AbstractSeoEvent`，提供：
- 数据验证
- 类型检查
- 事件元数据（时间戳、事件ID等）

## 事件监听

### 在 event.xml 中注册观察者

```xml

    <observer name="Your_Module::subject_created_handler"
```

## 事件解耦优势

### 1. 模块解耦

- **之前**: 模块间直接调用，强耦合
- **现在**: 通过事件通信，松耦合

### 2. 可扩展性

- 新模块只需监听事件，无需修改现有代码
- 支持多个观察者同时处理同一事件

### 3. 可测试性

- 事件对象可独立测试
- 观察者可独立测试
- 易于Mock和Stub

### 4. 可维护性

- 事件契约明确，易于理解
- 事件版本管理，支持演化
- 事件文档完整，易于维护

## 最佳实践

### 1. 事件命名

- 使用动词过去式：`created`, `updated`, `deleted`
- 明确事件类型：`domain`, `integration`, `application`
- 保持命名一致性

### 2. 事件数据

- 只传递必要的数据
- 避免传递大型对象
- 使用数据契约确保类型安全

### 3. 事件版本

- 向后兼容的变更：增加可选字段（minor版本）
- 破坏性变更：修改必填字段（major版本）
- 文档更新：及时更新事件文档

### 4. 性能考虑

- 事件处理应该是轻量级的
- 耗时操作应放入队列
- 避免在事件中执行阻塞操作

## 总结

SEO模块的事件系统设计遵循国际标准，通过事件解耦实现了：

1. **模块解耦**: 模块间通过事件通信，降低耦合度
2. **类型安全**: 事件数据契约确保类型安全
3. **可扩展性**: 新功能通过监听事件实现，无需修改现有代码
4. **可维护性**: 事件文档和版本管理提升可维护性
5. **可测试性**: 事件对象和观察者可独立测试

这套事件系统为SEO模块提供了强大的扩展能力，同时保持了代码的清晰和可维护性。
