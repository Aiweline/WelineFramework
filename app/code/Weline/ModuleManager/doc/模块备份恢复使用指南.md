# 模块备份与恢复使用指南

## 适用场景

- 升级或卸载模块前，希望保留数据库数据，必要时回滚。
- 误删模块或模块数据后，希望从卸载时的备份中恢复。
- 在开发环境中频繁重装模块，同时保留历史数据。

## 关键概念

- **卸载备份**：在卸载模块时，将模块相关表重命名为 `_backup_时间戳`，并记录到备份记录表。
- **备份记录表**：`Weline\ModuleManager\Model\Module\Backup` 对应的表，用于追踪每次备份批次。
- **模块表注册**：`Weline\ModuleManager\Model\Module\Table` 记录模块与表名之间的映射关系。

## 快速开始

### 1. 确认模块表已注册

模块的 Model 在安装时会自动写入 `Module\Table` 表：

- 字段 `module_name`：模块名，如 `Weline_Demo`
- 字段 `name`：表名，如 `we_demo_table`

只要模块的 Model 正常安装，备份服务即可识别其表。

### 2. 卸载模块并自动备份

```bash
php bin/w module:remove Weline_Demo
```

流程：

1. 通过 `ModuleBackupService::backupModuleTables()` 备份所有注册表：
   - 原表：`we_demo_table`
   - 备份表：`we_demo_table_backup_YYYYMMDD_HHMMSS`
2. 写入备份记录到 `Module\Backup` 表。
3. 触发 `UninstallService::uninstall()` 做模块文件备份与事件通知。

如果数据库备份失败，卸载会被取消，并在控制台输出原因。

### 3. 从备份中恢复模块数据

```bash
# 恢复最新一次备份
php bin/w module:restore -m Weline_Demo

# 恢复指定批次（时间戳）
php bin/w module:restore -m Weline_Demo --backup=20250127_143000
```

行为：

- 根据模块名（及时间戳）查找备份记录。
- 对每张表执行：
  - `DROP TABLE IF EXISTS 原表名`
  - `ALTER TABLE 备份表 RENAME TO 原表名`
- 将备份记录状态标记为 `restored`。

### 4. 重装模块并尝试恢复数据

```bash
# 普通重装（不恢复卸载备份）
php bin/w module:reinstall -m Weline_Demo

# 重装后，如果存在卸载备份记录，则尝试从备份恢复数据
php bin/w module:reinstall -m Weline_Demo --restore
```

流程（`--restore`）：

1. 按原有逻辑：
   - 备份、复制并删除当前表。
   - 清理模块注册与依赖信息。
   - 重新执行 `Setup/Install` 和 `Model install`。
2. 安装完成后：
   - 调用 `ModuleBackupService::restoreModuleTables()`。
   - 如存在卸载备份：覆盖当前表，实现“带数据重装”。

> 提示：`--restore` 依赖于之前的卸载备份记录，建议在生产中谨慎使用，并配合完整测试。

## 命令总结

- `module:remove`：卸载模块 + 自动数据库表备份 + 模块文件备份。
- `module:restore`：从数据库备份中恢复模块数据表。
- `module:reinstall`：
  - 纯重装：重新安装模块（适合开发环境清理）。
  - `--restore`：重装并尝试从卸载备份恢复数据。

## 最佳实践

1. **生产环境卸载/重装**：
   - 始终确保有离线备份（如 mysqldump）。
   - 使用 `module:remove` 进行卸载，以保留可快速恢复的备份表。
   - 必要时用 `module:restore` 回滚。

2. **开发环境重装**：
   - 可频繁使用 `module:reinstall` 清理和重建模块。
   - 若需要重用旧数据，结合 `--restore` 使用。

3. **备份空间管理**：
   - 目前备份表以 `_backup_时间戳` 形式保存在数据库中。
   - 后续可结合单独的清理命令或手动 SQL 清理历史备份表。

## 故障排查

- **找不到备份记录**：
  - 检查是否使用 `module:remove` 卸载过模块。
  - 确认 `Module\Backup` 表中是否有对应模块记录。

- **恢复失败：某些表不存在**：
  - 卸载时该表可能已被手动删除或未注册到 `Module\Table`。
  - 仅会跳过该表，不影响其他表恢复。

- **结构不兼容问题**：
  - 若在升级中变更了表结构，再用旧备份覆盖新结构可能导致不兼容。
  - 建议升级前完整备份，并在测试环境验证回滚流程。
