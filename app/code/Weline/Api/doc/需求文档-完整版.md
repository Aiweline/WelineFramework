## 1. 引言

### 1.1 编写目的

本文档旨在详细描述 Weline API 模块的功能需求、性能需求、接口需求、设计约束等，为系统设计、开发、测试和验收提供依据。

**目标读者**:
- 系统架构师
- 开发人员
- 测试人员
- 项目管理人员
- 最终用户

### 1.2 项目背景

Weline Framework 是一个基于 PHP 的企业级应用开发框架。随着系统规模的扩大和第三方集成的需求增加，需要建立一个统一的 API 管理模块，提供：

- 统一的 API 用户和权限管理
- 自动化的 API 文档生成和浏览
- 安全的 API 认证和授权机制
- 支持在线 API 调试的沙盒环境

**认证策略**：
- **内部使用**：已登录用户（后端或前端）可以直接使用Session认证访问API，无需额外的Token认证
- **第三方访问**：向第三方提供API时，需要建立单独的API用户，使用Token认证流程

## 2. 总体描述

### 2.1 产品视角

Weline API 模块是 Weline Framework 的一个核心模块，与其他模块的关系如下：

```
┌─────────────────────────────────────────┐
│         Weline Framework                │
├─────────────────────────────────────────┤
│  ┌──────────┐  ┌──────────┐  ┌──────┐  │
│  │  Backend │  │   Acl    │  │ Admin│  │
│  └────┬─────┘  └────┬─────┘  └──┬───┘  │
│       │             │            │      │
│       └──────┬──────┴────────────┘      │
│              │                          │
│         ┌────▼─────┐                    │
│         │   Api    │                    │
│         └────┬─────┘                    │
│              │                          │
│  ┌───────────▼──────────────┐          │
│  │ DeveloperWorkspace       │          │
│  └──────────────────────────┘          │
└─────────────────────────────────────────┘
```

**模块依赖关系**:
- **依赖模块**: `Weline_Framework`（必需）、`Weline_Backend`（必需）、`Weline_Acl`（必需）、`Weline_Admin`（必需）、`Weline_DeveloperWorkspace`（可选）
- **被依赖模块**: 无（本模块不强制其他模块依赖）

### 2.2 产品功能

Weline API 模块提供以下核心功能：

1. **API 用户管理**
   - API 用户的创建、编辑、删除
   - API 用户状态管理（启用/禁用）
   - API 用户角色和权限分配

2. **API 认证和授权**
   - 基于Session和令牌的双重身份验证
   - Session认证：已登录用户（后端或前端）可以直接使用Session认证
   - Token认证：第三方访问时使用Token认证
   - 令牌交换机制（访问令牌、刷新令牌、通行令牌）
   - 区分后台 API 和前台 API 的认证

3. **API 文档生成和浏览**
   - 自动从控制器代码生成 API 文档
   - 参照 Swagger 格式的文档展示
   - 支持在线浏览和搜索

4. **API 在线调试**
   - 在文档界面直接测试 API
   - 沙盒数据库隔离，不影响正式数据
   - 请求和响应信息展示

### 2.3 用户特征

**主要用户类型**:

1. **系统管理员**
   - 技术背景：高级
   - 职责：管理 API 用户、配置权限、监控系统
   - 使用频率：日常

2. **API 开发者**
   - 技术背景：中级到高级
   - 职责：开发 API、查看文档、调试接口
   - 使用频率：频繁

3. **第三方集成开发者**
   - 技术背景：中级
   - 职责：集成 API、查看文档、测试接口
   - 使用频率：项目期间频繁

### 2.4 运行环境

**服务器环境**:
- PHP 版本：8.2 或更高
- Web 服务器：Apache 2.4+ / Nginx 1.18+
- 数据库：MySQL 5.7+ / MariaDB 10.3+ / SQLite 3.8+
- 操作系统：Linux / Windows Server / macOS

**客户端环境**:
- 浏览器：Chrome 90+, Firefox 88+, Safari 14+, Edge 90+
- 屏幕分辨率：最小 1280x720
- JavaScript：必须启用

### 2.5 设计和实现约束

1. **技术约束**
   - 必须遵循 Weline Framework 的架构规范
   - 必须兼容 PHP 8.2+ 的语法和特性
   - 必须使用框架提供的数据库抽象层
   - 必须遵循 PSR-4 自动加载规范

2. **标准约束**
   - RESTful API 设计规范
   - OAuth 2.0 令牌机制
   - Swagger/OpenAPI 3.0 文档规范
   - HTTP/HTTPS 协议标准

3. **兼容性约束**
   - 向后兼容现有 Backend 模块的 API 认证功能
   - 兼容现有 ACL 权限系统
   - 兼容现有管理后台界面布局

4. **安全约束**
   - API 密钥必须加密存储
   - 令牌传输必须使用 HTTPS（生产环境）
   - 实现令牌撤销机制
   - 防止令牌泄露和重放攻击

### 2.6 假设和依赖

**假设**:
1. 系统已正确安装和配置 Weline Framework
2. 数据库连接正常
3. 用户具有基本的 Web 使用经验
4. 管理员具有系统管理权限

**依赖**:
1. Weline Framework 核心模块正常运行
2. Backend 模块提供用户模型
3. Acl 模块提供权限管理功能
4. Admin 模块提供管理界面框架

## 5. 系统特性详细说明

### 5.1 FR-001: API 用户管理

#### 5.1.1 功能描述

提供 API 用户的完整生命周期管理，包括创建、编辑、删除、启用/禁用等操作。

#### 5.1.2 输入

**创建 API 用户**:
- 用户名（必填，字符串，3-50 字符）
- 邮箱（必填，字符串，邮箱格式）
- 密码（必填，字符串，最少 6 字符）
- 访问令牌有效期（可选，整数，秒，默认 604800 = 7 天，范围：1-30天）
- 刷新令牌有效期（可选，整数，秒，默认 2592000 = 30 天，范围：7-90天）
- 角色列表（可选，数组，角色 ID 列表）
- IP白名单配置（可选）
- 用户代理限制配置（可选）

**编辑 API 用户**:
- 用户 ID（必填，整数）
- 其他字段同创建（可选）

**删除 API 用户**:
- 用户 ID（必填，整数）

**启用/禁用 API 用户**:
- 用户 ID（必填，整数）
- 状态（必填，布尔值，true=启用，false=禁用）

#### 5.1.3 处理

1. **创建用户流程**:
   ```
   1. 验证输入数据
   2. 检查用户名和邮箱是否已存在
   3. 加密密码
   4. 生成 API Key 和 API Secret（创建时立即生成）
   5. 创建独立的 ApiUser 记录（不依赖 BackendUser）
   6. 如果指定了角色，分配角色
   7. 返回创建结果（API Secret 仅返回一次）
   ```

2. **生成 API Key/Secret**:
   - **生成时机**：在创建 API 用户时立即生成
   - **唯一性保证**：每次创建都生成新的 Key/Secret，不重复使用
   - **API Key**: 32 位随机字符串（Base64 编码）
   - **API Secret**: 64 位随机字符串（Base64 编码），使用 `password_hash()` 加密存储
   - **安全考虑**：API Secret 仅创建时返回一次，之后无法查看

3. **删除用户**:
   - 软删除：设置 `is_deleted = 1`
   - 同时删除所有相关令牌

#### 5.1.4 输出

**成功响应**:
```json
{
  "code": 0,
  "msg": "操作成功",
  "data": {
    "user_id": 1,
    "username": "api_user_1",
    "api_key": "xxx",
    "api_secret": "xxx"  // 仅创建时返回一次
  }
}
```

**错误响应**:
```json
{
  "code": 400,
  "msg": "用户名已存在",
  "data": ""
}
```

#### 5.1.5 异常处理

| 异常情况 | 处理方式 |
|----------|----------|
| 用户名已存在 | 返回错误，不允许创建 |
| 邮箱已存在 | 返回错误，不允许创建 |
| 密码强度不足 | 返回错误提示 |
| 数据库连接失败 | 记录日志，返回系统错误 |
| 权限不足 | 返回 403 错误 |

### 5.2 FR-002: API 用户角色管理

#### 5.2.1 功能描述

为 API 用户分配角色，角色包含多个权限资源。使用 jstree 组件可视化选择权限。

#### 5.2.2 输入

**分配角色**:
- 用户 ID（必填，整数）
- 角色 ID（必填，整数）

**分配权限**:
- 角色 ID（必填，整数）
- 权限资源列表（必填，数组，格式：`["module::resource::action", ...]`）

#### 5.2.3 处理

1. **权限树构建**:
   - **参考实现**：参考后端用户权限配置的实现方式（`Weline\Admin\Controller\Backend\User`）
   - **数据来源**：从 `w_acl` 表获取所有权限资源
   - **权限结构**：每个操作都有唯一的权限名称（格式：`module::resource::action`）
   - **权限没有固定的层级结构**（不是模块->资源->操作的固定结构）
   - **树形构建**：根据权限的 `parent_source` 字段构建树形结构（支持多级父级）
   - **层级处理**：支持多级父级，使用递归算法构建
   - **性能优化**：使用缓存，避免频繁查询数据库
   - **标记已分配**：标记已分配的权限

2. **权限保存**:
   - 清除角色原有权限
   - 保存新选择的权限
   - 更新权限缓存

3. **权限选择规则**:
   - **选择父级**：自动选择所有子级权限
   - **取消父级**：自动取消所有子级权限
   - **选择子级**：
     - 如果所有子级都被选中，自动选中父级
     - 如果部分子级被选中，父级显示为"部分选中"状态（jstree支持）
     - 如果所有子级都被取消，自动取消父级
   - jstree 原生支持多级父级关系的处理

#### 5.2.4 输出

**权限树数据格式**:
```json
{
  "id": "Weline_Api::user::list",
  "text": "用户列表",
  "parent": "Weline_Api::user",
  "state": {
    "selected": true
  }
}
```

### 5.3 FR-003: API 认证

#### 5.3.1 功能描述

提供基于令牌的 API 身份验证机制，支持多种令牌获取方式。

**特殊认证机制**：
- **后端API**：用户登录后端后，可以直接使用后端的Session认证通过，不需要走单独的API用户。这样可以让接口在后端运用时无需复杂的登录和认证措施，只需要校验后端用户的Session和权限即可。
- **前端API**：用户登录前端后，可以直接使用前端的Session认证通过需要登录的接口，不需要走单独的API用户。这样可以让前端页面调用API时无需额外认证，用户体验更好。

**第三方访问**：
- 向第三方提供API时，才需要建立单独的API用户走标准认证流程（Token认证）
- 公开前端的第三方访问也是一样的，在不走Session的时候需要走标准的前端接口认证

#### 5.3.1.1 认证优先级

**后端API认证优先级**：

1. **后端Session认证**（优先）
   - 检查用户是否已登录后端（`BackendSession`）
   - 如果已登录，直接使用后端Session认证
   - 权限验证使用后端用户的权限（`BackendUser`的权限）
   - **优点**：后端管理界面调用API时无需额外认证，用户体验更好
   - **安全限制**：不检查IP白名单和用户代理限制（因为已通过后端登录验证）

2. **API Token认证**（备选，用于第三方访问）
   - 如果后端Session未登录，检查API Token
   - 使用 `BackendApiSession` 进行Token认证
   - 权限验证使用API用户的权限（`ApiUser`的权限）
   - **使用场景**：向第三方系统提供API时使用
   - **安全限制**：支持IP白名单和用户代理限制

**前端API认证优先级**：

1. **前端Session认证**（优先）
   - 检查用户是否已登录前端（`FrontendSession`）
   - 如果已登录，直接使用前端Session认证
   - 权限验证使用前端用户的权限（`FrontendUser`的权限）
   - **优点**：前端页面调用API时无需额外认证，用户体验更好
   - **安全限制**：不检查IP白名单和用户代理限制（因为已通过前端登录验证）

2. **API Token认证**（备选，用于第三方访问）
   - 如果前端Session未登录，检查API Token
   - 使用 `FrontendApiSession` 进行Token认证
   - 权限验证使用API用户的权限（`FrontendApiUser`的权限）
   - **使用场景**：向第三方系统提供前端API时使用
   - **安全限制**：支持IP白名单和用户代理限制（可选）

**认证流程**：
```
后端API认证流程：
1. 检查是否为后端API请求（isApiBackend()）
2. 优先检查后端Session是否已登录
   - 如果已登录：使用BackendSession，权限使用BackendUser的权限
   - 如果未登录：继续检查API Token
3. 检查API Token（Authorization头或X-API-Token头）
   - 如果Token有效：使用BackendApiSession，权限使用ApiUser的权限
   - 如果Token无效：返回401错误

前端API认证流程：
1. 检查是否为前端API请求（isApiFrontend()）
2. 优先检查前端Session是否已登录
   - 如果已登录：使用FrontendSession，权限使用FrontendUser的权限
   - 如果未登录：继续检查API Token
3. 检查API Token（Authorization头或X-API-Token头）
   - 如果Token有效：使用FrontendApiSession，权限使用FrontendApiUser的权限
   - 如果Token无效：返回401错误
```

#### 5.3.2 输入

**登录获取令牌**:
- 用户名（必填，字符串）
- 密码（必填，字符串）
- 过期时间（可选，整数，秒）

**令牌刷新**:
- 访问令牌（必填，字符串）
- 通行令牌（必填，字符串，API Key）
- 过期时间（可选，整数，秒）

**令牌撤销**:
- 令牌（必填，字符串）

#### 5.3.3 处理

1. **登录验证**:
   ```
   1. 验证用户名和密码
   2. 检查用户状态（是否启用）
   3. 检查登录失败次数（防止暴力破解）
   4. 生成访问令牌和刷新令牌
   5. 保存令牌到数据库
   6. 更新用户登录信息
   7. 返回令牌
   ```

2. **令牌生成**:
   - 访问令牌：64 位随机字符串（Hex 编码）
   - 刷新令牌：64 位随机字符串（Hex 编码）
   - 令牌有效期：从用户配置或请求参数获取

3. **令牌验证**:
   ```
   1. 优先检查后端Session是否已登录
      - 如果已登录：直接通过，使用BackendUser的权限
      - 如果未登录：继续检查API Token
   2. 从请求头或参数获取令牌
   3. 查询数据库验证令牌
   4. 检查令牌是否过期
   5. 检查用户状态
   6. 返回验证结果
   ```

4. **后端Session认证**:
   ```
   1. 检查BackendSession是否已登录
   2. 如果已登录，获取BackendUser
   3. 检查BackendUser状态（是否启用）
   4. 使用BackendUser的权限进行权限验证
   5. 认证通过，允许访问
   ```

#### 5.3.4 输出

**登录成功**:
```json
{
  "code": 0,
  "msg": "登录成功",
  "data": {
    "access_token": "xxx",
    "refresh_token": "xxx",
    "expire_time": 1234567890,
    "user": {
      "id": 1,
      "username": "api_user_1"
    }
  }
}
```

### 5.4 FR-004: 令牌交换机制

#### 5.4.1 功能描述

实现安全的令牌交换机制，支持使用 API Key/Secret 交换访问令牌，以及使用访问令牌和通行令牌刷新令牌。

#### 5.4.2 令牌类型

1. **访问令牌（Access Token）**
   - 用途：API 请求认证
   - 有效期：可配置（默认 7 天）
   - 存储：数据库 `BackendUserToken` 表

2. **刷新令牌（Refresh Token）**
   - 用途：刷新访问令牌
   - 有效期：通常比访问令牌长（默认 30 天）
   - 存储：数据库 `BackendUserToken` 表

3. **通行令牌（Pass Token / API Key）**
   - 用途：令牌交换的长期凭证
   - 有效期：长期有效（可手动撤销）
   - 存储：数据库 `ApiUser` 表的 `api_key` 字段

#### 5.4.3 交换流程

**首次获取令牌**:
```
1. 客户端使用 API Key 和 API Secret 请求
2. 服务器验证 API Key/Secret
3. 生成访问令牌和刷新令牌
4. 返回令牌
```

**刷新令牌**:
```
1. 客户端使用访问令牌和通行令牌（API Key）请求（双重验证）
2. 服务器验证访问令牌和通行令牌
3. 使旧令牌立即失效
4. 生成新的访问令牌和刷新令牌
5. 返回新令牌
```

#### 5.4.4 安全机制

- API Secret 加密存储（使用 `password_hash()`）
- 令牌单次使用（刷新后旧令牌立即失效）
- 令牌过期自动清理
- 支持令牌撤销
- **令牌撤销机制**：
  - 撤销后立即失效，无宽限期
  - 撤销的令牌不能恢复，需要重新生成
  - 记录所有令牌撤销操作，便于审计

#### 5.4.5 令牌有效期配置

**访问令牌（Access Token）**:
- 默认有效期：7 天（604800 秒）
- 可配置范围：1 天 - 30 天
- 配置位置：在创建 API 用户时设置，或使用系统默认值

**刷新令牌（Refresh Token）**:
- 默认有效期：30 天（2592000 秒）
- 可配置范围：7 天 - 90 天
- 配置位置：在创建 API 用户时设置，或使用系统默认值

**通行令牌（Pass Token / API Key）**:
- 有效期：长期有效
- 可手动撤销
- 撤销后需要重新生成

### 5.5 FR-005: API 文档生成

#### 5.5.1 功能描述

自动扫描所有 API 控制器，从代码注释和 Acl 注解中提取信息，生成符合 Swagger 格式的 API 文档。

#### 5.5.2 更新机制

**缓存机制**:
- 首次生成后缓存，提高性能
- 缓存时间：开发环境1小时，生产环境24小时

**自动检测**:
- 检测到控制器文件变更时自动刷新缓存
- 通过文件修改时间判断是否需要更新

**手动刷新**:
- 提供手动刷新按钮，强制重新生成
- 提供命令行命令：`bin/w api:doc:generate`
- 提供定时任务（Cron）：定期刷新文档缓存

**命令生成**:
- 命令：`bin/w api:doc:generate [--force]`
- `--force`: 强制重新生成，忽略缓存

**定时刷新**:
- 通过 Cron 定时任务定期刷新文档
- 可配置刷新频率（默认每天凌晨执行）
- 集成到框架的 Cron 系统

#### 5.5.3 数据来源

1. **控制器类信息**:
   - 类名（作为分组名）
   - 类注释（作为分组描述）
   - **模块名（从扫描时的模块信息获取，使用Env::getInstance()->getModuleList()）**
   - **类级别Acl注解**（用于构建API文档层级结构）

2. **方法信息**:
   - 方法名（确定 HTTP 方法和路径）
   - 方法参数（通过反射获取，优先使用类型声明 PHP 8.2+）
   - 方法注释（PHPDoc，补充参数描述和示例）
   - `@example` 注释（包含完整的请求和响应示例，支持Header、Cookie、Body等）
   - **方法级别Acl注解**（权限要求和API识别）

3. **路由信息**:
   - 从路由配置获取完整路径
   - HTTP 方法（从方法名推断：get/post/put/delete）
   - 版本号（从目录结构提取：`Api\Rest\V1` → `v1`）

4. **参数提取规则**:
   - 优先使用 PHP 类型声明（int, string, array 等）
   - 通过 PHPDoc 的 `@param` 标签补充参数描述
   - 对于复杂类型（对象、数组），使用 PHPDoc 说明结构
   - 参数默认值从方法签名中提取

5. **@example 注释解析**:
   - 解析 `@example` 和 `@example-end` 之间的内容
   - 支持标准格式字段：
     - `Method`: HTTP方法（GET/POST/PUT/DELETE/PATCH）
     - `Path`: API路径
     - `Header`: 请求头（多行，格式：`- HeaderName: value`）
     - `Cookie`: Cookie信息（多行，格式：`- cookie_name: value`）
     - `Body`: 请求体（JSON格式，POST/PUT/PATCH时使用）
     - `Response`: 响应示例（JSON格式）
   - 支持简化格式：`Path: METHOD /api/path` 和 `Response: {...}`
   - 解析后的信息用于API文档展示和在线调试功能

6. **Acl层级结构**:
   - **层级结构**（用于API文档组织）：
     ```
     顶级：Api分类（固定）
       └─ 模块名（从扫描时的模块信息获取，如 Weline_Api）
           └─ 接口文档类分组（类级别Acl，使用类的@Document注释中的category或类名）
               └─ 接口方法（方法级别Acl，使用方法的@Document注释中的summary）
     ```
   - **模块名获取方式**：
     - 在扫描API控制器时，使用 `Env::getInstance()->getModuleList()` 获取所有模块信息
     - 根据控制器文件路径确定所属模块
     - 使用模块信息中的 `name` 字段（格式：`Weline_Api`）
     - **不需要从命名空间提取**，因为扫描时已经有模块信息
   - **类级别Acl**：
     - 必须放在控制器类上
     - 用于标识API文档的分组
     - 作为方法级别Acl的父级
     - source_id格式：`{Module}::api::{ClassName}`
     - parent_source指向模块API分类：`{Module}::api`
   - **方法级别Acl**：
     - 必须放在每个API方法上（后台API）
     - 用于标识具体的API接口
     - parent_source指向类级别Acl的source_id
     - source_id格式：`{Module}::api::{ClassName}::{MethodName}`

#### 5.5.4 文档格式

**Swagger/OpenAPI 3.0 格式**:
```yaml
openapi: 3.0.0
info:
  title: Weline API Documentation
  version: 1.0.0
paths:
  /api/v1/user/{id}:
    get:
      summary: 获取用户信息
      parameters:
        - name: id
          in: path
          required: true
          schema:
            type: integer
      responses:
        '200':
          description: 成功
      security:
        - BearerAuth: []
```

### 5.6 FR-006: API 文档浏览和导入

#### 5.6.1 功能描述

提供API文档的浏览界面和导入功能，支持从代码自动生成API文档并导入到文档管理系统。

#### 5.6.2 文档导入方式

**导入位置**：`app\code\Weline\DeveloperWorkspace` 模块

2. **扫描API控制器**：扫描所有实现了 `AbstractRestController` 的控制器
3. **提取API信息**：从代码注释和Acl注解中提取API信息
4. **构建层级结构**：根据Acl层级结构组织API文档
5. **生成文档**：生成符合Swagger格式的API文档
6. **导入文档系统**：将生成的API文档导入到 `DeveloperWorkspace` 文档管理系统

**层级结构**（用于API文档组织）：
```
顶级：Api分类（固定）

      └─ 接口文档类分组（类级别Acl，使用类的@Document注释中的category或类名）
```

**模块名获取方式**：
- 在扫描API控制器时，使用 `Env::getInstance()->getModuleList()` 获取所有模块信息
- 根据控制器文件路径确定所属模块
- 使用模块信息中的 `name` 字段（格式：`Weline_Api`）
- **不需要从命名空间提取**，因为扫描时已经有模块信息

**Acl层级要求**：
- **类级别Acl**：必须放在控制器类上，用于标识API文档的分组
- **方法级别Acl**：必须放在每个API方法上（后台API），用于标识具体的API接口
- **层级关系**：方法级别Acl的parent_source指向类级别Acl的source_id

#### 5.6.3 规范检验

**检验时机**：在API文档导入前进行规范检验

**检验内容**：
- 检查所有API接口是否符合 `app\code\Weline\Framework\Api` 定义的规范
- 验证@Document注释是否完整
- 验证PHPDoc注释是否完整（@param、@return等）
- 验证后台API接口是否有类级别和方法级别的Acl注解
- 验证@example注释格式是否正确

**检验失败处理**：
- 不符合规范的接口会被拦截，无法导入
- 显示详细的错误信息，包括不符合规范的接口列表和具体原因
- 要求修复所有不符合规范的接口后重新导入

### 5.7 FR-007: API 文档浏览

#### 5.7.1 功能描述

提供 Web 界面浏览 API 文档，支持搜索、过滤、分组展示。

#### 5.7.2 权限控制

**完全公开接口（无Acl注解，且不需要登录）**:
- 所有用户都可以查看和访问（无需登录）
- 所有完全公开接口都可见
- **缓存机制**：完全公开接口需要做好缓存
- **安全要求**：公开接口不允许携带Cookie信息
  - 如果请求中包含Cookie，应该拒绝请求或忽略Cookie
  - 这是为了防止用户状态泄露和确保接口的完全公开性
  - 实现方式：在API验证Observer中检查，如果接口标记为完全公开且请求包含Cookie，返回错误或清除Cookie
- **CDN预热**：向 Cdn 模块提供缓存预热链接
  - 实现 `Weline\Cdn\Api\WarmupProviderInterface` 接口
  - 在 `extends/module/Weline_Cdn/ApiDocUrls.php` 中提供完全公开API文档URL列表
  - **重要**：只包含不需要登录的公开接口，需要登录的接口不允许做缓存预热
  - 参考 Cdn 模块的 WarmupProvider 扩展机制

**需要登录的接口（无Acl注解，但需要登录）**:
- 需要登录才能访问
- 所有已登录用户都可以访问（无权限限制）
- **不允许缓存预热**：这些是业务请求，每个用户的请求结果可能不同
- 在文档中标记为"需要登录"

**受保护接口（有Acl注解）**:
- 需要登录才能查看
- 根据用户权限过滤显示
- 只显示用户有权限的API
- 无权限的API不显示在文档中
- **不允许缓存预热**：这些是业务请求，需要权限验证

**权限判断逻辑**:
```
1. 扫描所有API控制器方法
2. 检查方法是否有Acl注解
3. 如果没有Acl注解：
   - 检查是否需要登录（通过Session或Token验证）
   - 如果不需要登录，标记为"完全公开接口"（可缓存预热）
     - 验证请求不包含Cookie（如果包含，拒绝请求）
   - 如果需要登录，标记为"需要登录的接口"（不可缓存预热）
4. 如果有Acl注解，标记为"受保护接口"（不可缓存预热）
5. 根据用户登录状态和权限过滤显示
```

**公开接口Cookie验证**:
```
1. 在API请求处理前，检查接口是否为完全公开接口
2. 如果是完全公开接口，检查请求头中是否包含Cookie
3. 如果包含Cookie：
   - 方案A（推荐）：返回400错误，提示"公开接口不允许携带Cookie"
   - 方案B：清除Cookie后继续处理（不推荐，可能泄露用户状态）
4. 记录违规请求日志（包含IP、User-Agent等信息）
```

**CDN预热规则**:
- ✅ **允许预热**：完全公开接口（无Acl，不需要登录）
  - 需要为每种 `locale` 和 `currency` 参数组合生成预热URL
  - 示例：`/api/v1/products?locale=zh_Hans_CN&currency=CNY`、`/api/v1/products?locale=en_US&currency=USD`
- ❌ **不允许预热**：需要登录的接口（即使无Acl）
- ❌ **不允许预热**：受保护接口（有Acl注解）

**i18n本地化支持**:
- API接口支持通过URL参数 `locale` 和 `currency` 切换语言和货币
- 参数统一在URL上处理（Query参数）
- 支持幂等性，便于缓存
- 详见 **5.12 FR-012: API i18n本地化支持**

#### 5.7.3 功能特性

- 按控制器类分组展示（基于Acl层级结构）
- 支持搜索 API 名称、描述
- 支持按模块过滤
- 支持按 HTTP 方法过滤

### 5.9 FR-009: 沙盒数据库

#### 5.8.1 功能描述

在调试模式下，所有数据库操作使用独立的沙盒数据库，不影响正式数据。

**核心挑战**：沙盒数据库不是线上最新数据，如何处理用户认证和数据一致性问题？

#### 5.9.2 实现机制

1. **调试模式识别**:
   - 请求头：`X-Debug-Mode: 1`
   - 或 URL 参数：`?debug=1`
   - 仅在 API 文档界面发起的请求中自动添加

2. **数据库切换策略**:
   - 在请求层面通过中间件切换数据库连接
   - 使用请求级别的连接池，确保每个请求使用独立的连接
   - 使用 `sandbox_db` 配置
   - 所有模型操作使用沙盒数据库

3. **数据同步机制**（解决数据一致性问题）:
   
   **方案A：混合模式（推荐）**
   - 认证相关操作（登录、令牌验证）使用正式数据库
   - 业务数据操作使用沙盒数据库
   - 优点：保证认证可用，业务数据隔离
   - 实现：在中间件中根据操作类型选择数据库
   
   **方案B：数据镜像**
   - 定期从正式数据库同步基础数据到沙盒数据库（用户表、配置表等）
   - 业务数据在沙盒数据库中独立操作
   - 优点：数据相对完整
   - 缺点：需要维护同步机制
   
   **方案C：只读正式库 + 写沙盒库**
   - 读取操作：优先从正式数据库读取（认证、用户信息等）
   - 写入操作：写入沙盒数据库
   - 优点：保证认证和基础数据可用
   - 实现：在模型层面根据操作类型选择数据库

4. **推荐实现方案**（方案A + 方案C 混合）:
   ```
   1. 认证相关表（w_backend_user, w_backend_user_token, w_api_user）：
      - 始终使用正式数据库（读+写）
      - 确保用户登录和令牌验证正常
   
   2. 业务数据表（其他所有表）：
      - 调试模式：使用沙盒数据库（读+写）
      - 正常模式：使用正式数据库（读+写）
   
   3. 实现方式：
      - 在中间件中检测调试模式
      - 根据表名判断使用哪个数据库
      - 使用请求级别的连接池管理连接
   ```

5. **数据隔离**:
   - 沙盒数据库完全独立
   - 支持清空沙盒数据（不影响正式数据）
   - 支持查看沙盒数据
   - 支持导出沙盒数据

6. **配置说明**:
   - 在 `app/etc/env.php` 中配置 `sandbox_db`
   - 可以配置哪些表始终使用正式数据库（白名单）
   - 可以配置哪些表始终使用沙盒数据库（黑名单）

### 5.10 FR-010: Area 区分

#### 5.10.1 功能描述

区分后台 API（`api_admin`）和前台 API（`api`）的认证机制。

#### 5.10.2 区分规则

**后台 API**:
- URL 路径：`/api_admin/...` 或 `WELINE_AREA=api_admin`
- **认证方式**（优先级顺序）：
  1. **后端Session认证**（优先）
     - 如果用户已登录后端管理界面，直接使用后端Session认证
     - 使用 `BackendSession` 进行认证
     - 权限验证使用 `BackendUser` 的权限
     - **优点**：后端管理界面调用API时无需额外认证，用户体验更好
     - **安全限制**：不检查IP白名单和用户代理限制（因为已通过后端登录验证）
  2. **API Token认证**（备选）
     - 如果后端Session未登录，使用API Token认证
     - 使用 `BackendApiSession` 进行认证
     - 验证独立的 API 用户（ApiUser，不依赖 BackendUser）
     - 权限验证使用 `ApiUser` 的权限
     - **优点**：支持第三方系统集成，独立的API用户管理
     - **安全限制**：支持IP白名单和用户代理（User-Agent）限制
- **权限控制**：基于 ACL 权限系统，需要分配角色和权限

**前台 API**:
- URL 路径：`/api/...` 或 `WELINE_AREA=api`
- **认证方式**（优先级顺序）：
  1. **前端Session认证**（优先）
     - 如果用户已登录前端，直接使用前端Session认证
     - 使用 `FrontendSession` 进行认证
     - 权限验证使用 `FrontendUser` 的权限
     - **优点**：前端页面调用API时无需额外认证，用户体验更好
     - **安全限制**：不检查IP白名单和用户代理限制（因为已通过前端登录验证）
  2. **API Token认证**（备选，用于第三方访问）
     - 如果前端Session未登录，使用API Token认证
     - 使用 `FrontendApiSession` 进行认证（扩展现有实现，功能对等）
     - 验证独立的 API 用户（`FrontendApiUser`，不依赖 FrontendUser）
     - 权限验证使用 `FrontendApiUser` 的权限
     - **使用场景**：向第三方系统提供前端API时使用
     - **优点**：支持第三方系统集成，独立的API用户管理
     - **安全限制**：支持IP白名单和用户代理限制（可选）
- **权限控制**：
  - 前端接口没有权限控制，只要用户登录即可访问
  - 使用Session认证时，使用FrontendUser的权限
  - 使用Token认证时，使用FrontendApiUser的权限
- **用户管理**：
  - 复用现有的前台用户管理
  - 在前台用户管理界面添加"API访问"功能
  - 用户在个人中心可以生成和管理自己的 API Key（用于第三方访问）
  - 使用 Sticker 功能向前端个人中心注入 API Key 管理功能
  - 提供文档地址链接

#### 5.10.3 实现方式

在 `ApiControllerInitBefore` Observer 中：
```php
if ($request->isApiBackend()) {
    // 后端API认证
    // 1. 优先检查后端Session认证

        // 权限验证使用BackendUser的权限
        // 不需要检查IP白名单和用户代理限制（后端Session认证）
    } else {
        // 2. 后端Session未登录，使用API Token认证（第三方访问）
        $apiSession = ObjectManager::getInstance(BackendApiSession::class);
        if (!$apiSession->isLogin()) {
            return 401; // 未登录

        $user = $apiSession->getApiUser();

            return 403; // 用户已被禁用
        }
        // 检查IP白名单和用户代理限制（API Token认证需要）
        $this->checkSecurityRestrictions($request, $user);
    }
} elseif ($request->isApiFrontend()) {
    // 前端API认证
    // 1. 优先检查前端Session认证

        // 权限验证使用FrontendUser的权限
        // 不需要检查IP白名单和用户代理限制（前端Session认证）
    } else {
        // 2. 前端Session未登录，使用API Token认证（第三方访问）
        $apiSession = ObjectManager::getInstance(FrontendApiSession::class);
        if (!$apiSession->isLogin()) {
            return 401; // 未登录
        }
        // 检查用户状态
        $user = $apiSession->getApiUser();
        if (!$user->getIsEnabled()) {
            return 403; // 用户已被禁用
        }
        // 检查IP白名单和用户代理限制（可选，前端API Token认证）
        // $this->checkSecurityRestrictions($request, $user);
    }
}
```

### 5.11 FR-011: 后端API安全限制

#### 5.11.1 功能描述

为后端API提供IP白名单和用户代理（User-Agent）限制功能，增强API安全性。

#### 5.11.1.1 公开接口Cookie限制

**功能说明**：
- 完全公开接口（无Acl注解，不需要登录）不允许携带Cookie信息
- 如果请求中包含Cookie，应该拒绝请求
- 这是为了防止用户状态泄露和确保接口的完全公开性

**验证逻辑**：
```
1. 在API请求处理前，检查接口是否为完全公开接口
2. 如果是完全公开接口，检查请求头中是否包含Cookie
3. 如果包含Cookie，返回400错误：
   {
     "code": 400,
     "msg": "公开接口不允许携带Cookie",
     "data": null
   }
4. 记录违规请求日志（包含IP、User-Agent、请求路径等信息）
```

**实现位置**：
- 在 `ApiControllerInitBefore` Observer 中实现
- 在IP白名单和用户代理验证之前执行
- 只对完全公开接口进行验证

**错误响应**：
```json
{
  "code": 400,
  "msg": "公开接口不允许携带Cookie",
  "data": null
}
```

#### 5.11.2 IP白名单功能

**功能说明**：
- 每个API用户可以配置允许访问的IP地址列表
- 支持单个IP、IP段（CIDR）、IP范围
- 如果未配置IP白名单，则不限制（允许所有IP）
- 如果配置了IP白名单，只允许白名单中的IP访问

**配置方式**：
- 在API用户管理界面配置
- 支持多个IP地址，每行一个
- 支持格式：
  - 单个IP：`192.168.1.100`
  - IP段（CIDR）：`192.168.1.0/24`
  - IP范围：`192.168.1.1-192.168.1.100`

**验证逻辑**：
```
1. 获取API用户的IP白名单配置
2. 如果白名单为空，允许访问
3. 如果白名单不为空，检查请求IP是否在白名单中
4. 如果不在白名单中，返回403错误
```

#### 5.11.3 用户代理限制功能

**功能说明**：
- 每个API用户可以配置允许的用户代理（User-Agent）列表
- 支持精确匹配和正则表达式匹配
- 如果未配置用户代理限制，则不限制（允许所有User-Agent）
- 如果配置了用户代理限制，只允许匹配的User-Agent访问

**配置方式**：
- 在API用户管理界面配置
- 支持多个User-Agent，每行一个
- 支持格式：
  - 精确匹配：`MyApp/1.0`
  - 正则表达式：`/^MyApp\/\d+\.\d+$/`
  - 通配符：`MyApp/*`

**验证逻辑**：
```
1. 获取API用户的User-Agent限制配置
2. 如果限制为空，允许访问
3. 如果限制不为空，检查请求User-Agent是否匹配
4. 如果不匹配，返回403错误
```

#### 5.11.4 数据库设计

**扩展 `w_api_user` 表**：
- `allowed_ips`: TEXT - 允许的IP地址列表（JSON格式或换行分隔）
- `allowed_user_agents`: TEXT - 允许的用户代理列表（JSON格式或换行分隔）
- `ip_whitelist_enabled`: TINYINT(1) - 是否启用IP白名单（默认0=不启用）
- `user_agent_restriction_enabled`: TINYINT(1) - 是否启用用户代理限制（默认0=不启用）

#### 5.11.5 验证流程

**Acl权限验证位置**：`app\code\Weline\Acl\Observer\RouteBefore.php`

**验证流程**：

1. **在 `RouteBefore` Observer 中进行Acl权限验证**：
   - 该Observer监听路由处理前的事件
   - 支持后台和后台API请求的Acl验证
   - 支持前端API请求的Acl验证（预留接口）

2. **用户和角色获取（支持多种认证方式）**：
   - **优先从事件中获取**：支持通过事件传递用户、角色和权限列表（用于第三方认证）
     - `$event->getData('user')`：用户对象
     - `$event->getData('role')`：角色对象
     - `$event->getData('access_sources')`：权限列表
   - **Session认证**（传统方式）：
     - 后台请求：使用 `BackendSession`
     - 后台API请求：优先使用 `BackendApiSession`，其次 `BackendSession`
     - 前端API请求：使用 `FrontendSession` 或 `FrontendApiSession`

3. **权限验证逻辑**：
   - 检查用户是否存在
   - 检查用户状态（是否启用）
   - 检查用户是否有角色
   - 检查用户是否有权限访问当前路由
   - 对于API请求，返回JSON格式的错误响应
   - 对于后台请求，返回HTML错误页面或重定向

**在 `ApiControllerInitBefore` Observer 中（认证验证）**：
```php
// 1. 检查是否为完全公开接口
if ($this->isPublicApi($request)) {
    // 1.1 检查是否包含Cookie（公开接口不允许携带Cookie）
    if ($request->getHeader('Cookie')) {
        $this->returnError(400, __('公开接口不允许携带Cookie'));
        return;
    }
    // 公开接口不需要进一步验证，直接返回
    return;
}

// 2. 认证验证（优先级：后端Session > API Token）
$backendSession = ObjectManager::getInstance(BackendSession::class);
$apiSession = ObjectManager::getInstance(BackendApiSession::class);
$user = null;
$isBackendSessionAuth = false;

// 2.1 优先检查后端Session认证
if ($backendSession->isLogin()) {
    $user = $backendSession->getLoginUser();
    $isBackendSessionAuth = true;
} elseif ($apiSession->isLogin()) {
    // 2.2 后端Session未登录，使用API Token认证
    $user = $apiSession->getApiUser();
    $isBackendSessionAuth = false;
} else {
    // 2.3 都未登录，返回401
    $this->returnError(401, __('请先登录'));
    return;
}

// 3. 检查用户状态
if (!$user || !$user->getIsEnabled()) {
    $this->returnError(403, __('用户已被禁用'));
    return;
}

// 4. 如果是API Token认证，检查IP白名单和用户代理限制
if (!$isBackendSessionAuth) {
    $apiUser = ApiUser::load($user->getId());
    if ($apiUser->getId()) {
        // 4.1 检查IP白名单
        if ($apiUser->isIpWhitelistEnabled()) {
            $allowedIps = $apiUser->getAllowedIps();
            $clientIp = $request->clientIP();
            if (!$this->isIpAllowed($clientIp, $allowedIps)) {
                $this->returnError(403, __('IP地址不在白名单中'));
                return;
            }
        }
        
        // 4.2 检查用户代理限制
        if ($apiUser->isUserAgentRestrictionEnabled()) {
            $allowedUserAgents = $apiUser->getAllowedUserAgents();
            $userAgent = $request->getHeader('User-Agent');
            if (!$this->isUserAgentAllowed($userAgent, $allowedUserAgents)) {
                $this->returnError(403, __('用户代理不匹配'));
                return;
            }
        }
    }
}

// 5. 权限验证（使用对应的用户权限）
// 如果是后端Session认证，使用BackendUser的权限
// 如果是API Token认证，使用ApiUser的权限
// 权限验证在Acl模块中处理
```

### 5.12 FR-012: API i18n本地化支持

#### 5.12.1 功能描述

API接口支持i18n本地化，通过URL参数（`locale`和`currency`）切换语言和货币，确保接口的幂等性，便于缓存。

#### 5.12.2 功能要求

1. **双重支持方式**：
   - **方式1：Path级别**：支持在URL路径中通过前缀切换（如 `/zh_Hans_CN/CNY/api/v1/user`）
   - **方式2：参数级别**：支持在URL参数中通过Query参数切换（如 `/api/v1/user?locale=zh_Hans_CN&currency=CNY`）
   - **混合使用**：两种方式可以同时使用，path级别优先级更高

2. **参数说明**：
   - `locale` 参数：用于切换语言（i18n本地化代码，如 `zh_Hans_CN`、`en_US`）
   - `currency` 参数：用于切换货币（货币代码，如 `CNY`、`USD`）

3. **参数处理优先级**：
   - **优先级1（最高）**：URL路径前缀（如 `/zh_Hans_CN/CNY/api/v1/user`）
   - **优先级2**：URL参数（Query参数，如 `?locale=zh_Hans_CN&currency=CNY`）
   - **优先级3**：Cookie或默认值（如果前两者都不存在）
   - **规则**：如果path级别存在，优先使用path级别的值；如果path级别不存在，则使用URL参数的值；如果两者都不存在，则使用Cookie或默认值

4. **接口URL支持**：
   - 接口也是URL的一种，完全支持path级别和参数级别的切换
   - 所有API接口（包括 `api` 和 `api_admin`）都支持这两种方式
   - 确保 `getApiUrl()`, `getBackendApiUrl()`, `getFrontendApiUrl()` 等方法也支持这两种方式

5. **幂等性要求**：
   - `locale` 和 `currency` 参数必须支持幂等性
   - 相同的 `locale` 和 `currency` 参数组合，返回相同的结果
   - 便于CDN缓存和API响应缓存

6. **缓存支持**：
   - 不同的 `locale` 和 `currency` 参数组合，视为不同的缓存键
   - 缓存键格式：`{api_path}?locale={locale}&currency={currency}`
   - 完全公开接口（无Acl，不需要登录）支持按locale和currency分别缓存

7. **API响应**：
   - API响应内容根据 `locale` 参数切换语言
   - API响应中的货币相关数据根据 `currency` 参数切换货币
   - 错误消息、提示信息等根据 `locale` 参数返回对应语言

#### 5.12.3 实现方式

**处理位置**：`app\code\Weline\Framework\Http\Url` 的 `parser()` 方法

**当前状态**：
- ✅ **Path级别支持**：已实现，`Url::parser()` 方法中已有完整的path级别locale和currency检测逻辑（第658-732行）
- ⚠️ **参数级别支持**：需要增强，添加从URL参数（Query参数）中提取 `locale` 和 `currency` 的逻辑

**处理流程**：

1. **URL解析（增强 `Url::parser()` 方法）**：
   - **步骤1**：检查URL路径前缀是否包含语言和货币代码（已实现）
   - **步骤2**：如果path级别不存在，检查URL参数（Query参数）中是否包含 `locale` 和 `currency`（需要实现）
   - **步骤3**：如果path级别和URL参数都不存在，使用Cookie或默认值（已实现）

2. **参数提取逻辑**：
   ```php
   // 在 Url::parser() 方法中，在path级别检测之后，添加参数级别检测
   // 检查URL参数（Query参数），仅在path级别不存在时使用
   $query = self::parse_url($url, 'query') ?: '';
   parse_str($query, $queryParams);
   
   // 如果path级别不存在locale，且URL参数中存在locale，则使用URL参数
   if (empty($data['language']) && !empty($queryParams['locale'])) {
       $data['language'] = $queryParams['locale'];
       self::$parserServer['WELINE_USER_LANG'] = $queryParams['locale'];
   }
   
   // 如果path级别不存在currency，且URL参数中存在currency，则使用URL参数
   if (empty($data['currency']) && !empty($queryParams['currency'])) {
       $data['currency'] = $queryParams['currency'];
       self::$parserServer['WELINE_USER_CURRENCY'] = $queryParams['currency'];
   }
   ```

3. **参数设置**：
   - 将最终确定的 `locale` 参数设置到 `$_SERVER['WELINE_USER_LANG']`
   - 将最终确定的 `currency` 参数设置到 `$_SERVER['WELINE_USER_CURRENCY']`
   - 确保后续处理可以使用这些参数

4. **API响应处理**：
   - 在API响应生成时，根据 `locale` 参数切换语言
   - 在API响应生成时，根据 `currency` 参数切换货币
   - 确保错误消息、提示信息等使用正确的语言

   - 支持从Session中获取用户和角色（传统认证）

2. **`validateFrontendApiAccess()` 方法**：
   - 验证前端API请求的访问权限（预留接口）

#### 13.3.3 CDN预热Provider实现

```php
// app/code/Weline/Api/extends/module/Weline_Cdn/ApiDocUrls.php

namespace Weline\Api\Extends\Weline_Cdn;

use Weline\Cdn\Api\WarmupProviderInterface;
use Weline\Api\Service\ApiDocService;
use Weline\Framework\Manager\ObjectManager;

class ApiDocUrls implements WarmupProviderInterface
{
    public static function execute(): array
    {
        // 获取所有完全公开的API接口
        $apiDocService = ObjectManager::getInstance(ApiDocService::class);
        $publicApis = $apiDocService->getPublicApis(); // 只返回完全公开的接口
        
        $urls = [];
        
        // 支持的locale和currency列表
        $locales = ['zh_Hans_CN', 'en_US', 'ja_JP']; // 根据实际配置获取
        $currencies = ['CNY', 'USD', 'JPY']; // 根据实际配置获取
        
        foreach ($publicApis as $api) {
            $baseUrl = $api['path'];
            
            // 为每种locale和currency组合生成预热URL
            foreach ($locales as $locale) {
                foreach ($currencies as $currency) {
                    $urls[] = $baseUrl . '?locale=' . $locale . '&currency=' . $currency;
                }
            }
        }
        
        return $urls;
    }
}
```

**实现说明**:
- `getPublicApis()` 方法需要过滤掉需要登录的接口
- 判断标准：
  - 无Acl注解
  - 不需要Session或Token验证
  - 不依赖用户登录状态
- 需要登录的接口（即使无Acl）不应该加入预热队列
- **i18n支持**：为每种 `locale` 和 `currency` 参数组合生成预热URL，确保不同语言和货币的接口都能被预热
