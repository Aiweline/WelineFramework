# 沙盒数据库实现方案

## 一、问题描述

### 1.1 框架现有机制

**框架已支持沙盒模式**（`app/code/Weline/Framework/App.php`）：
- 通过 URL 参数 `?sandbox={sandbox_key}` 或 Cookie `w_sandbox=1` 启用
- 定义 `SANDBOX` 常量标识沙盒模式
- `Env::getDbConfig()` 方法在 `SANDBOX || DEBUG` 时返回 `sandbox_db` 配置

**当前问题**：
- 框架的沙盒模式是**全局切换**，所有表都使用沙盒数据库
- 这导致认证相关操作（登录、令牌验证）无法正常工作

### 1.2 核心挑战

在 API 调试模式下，需要实现**表级别的数据库选择**：

1. **数据一致性问题**：
   - 沙盒数据库不是线上最新数据
   - 用户在正式环境建立的账户，登录也在正式环境
   - 如何在沙盒数据库中验证用户身份？

2. **混合数据库策略**：
   - 认证相关表（用户、令牌）必须使用正式数据库
   - 业务数据表使用沙盒数据库
   - 需要根据表名动态选择数据库连接

3. **切换机制问题**：
   - 如何确保调试模式下业务表使用沙盒数据库？
   - 如何确保认证表始终使用正式数据库？
   - 如何避免影响其他请求？

## 二、解决方案

### 2.1 基于框架现有机制的扩展

**核心思想**：
1. 利用框架现有的 `SANDBOX` 常量机制
2. 在 `Env::getDbConfig()` 或模型连接层面，根据表名选择数据库
3. 认证相关表始终使用正式数据库，业务表在沙盒模式下使用沙盒数据库

### 2.2 混合数据库策略（推荐）

**核心思想**：根据数据表类型，选择使用正式数据库还是沙盒数据库。

#### 2.2.1 表分类

**A类：认证相关表（始终使用正式数据库）**
- `w_backend_user` - 后台用户表
- `w_backend_user_token` - 用户令牌表
- `w_api_user` - API用户扩展表
- `w_frontend_user` - 前台用户表（如需要）
- `w_frontend_user_token` - 前台用户令牌表（如需要）

**原因**：
- 用户账户在正式环境创建和管理
- 令牌验证需要实时性
- 保证调试时能够正常登录和认证

**B类：业务数据表（调试模式使用沙盒数据库）**
- 所有其他业务表
- 如：订单表、商品表、配置表等

**原因**：
- 业务数据可以独立测试
- 不影响正式数据
- 可以随意修改和删除

#### 2.2.2 实现机制

```php
// 1. 扩展 Env 类，支持表级别的数据库选择
// 在 Weline\Api\App\Env 中扩展（或修改框架的 Env 类）

namespace Weline\Api\App;

use Weline\Framework\App\Env as FrameworkEnv;

class Env extends FrameworkEnv
{
    // 认证相关表白名单（始终使用正式数据库）
    private static array $authTables = [
        'w_backend_user',
        'w_backend_user_token',
        'w_api_user',
        'w_frontend_user',
        'w_frontend_user_token',
    ];
    
    /**
     * 获取数据库配置（根据表名选择）
     * 
     * @param string|null $tableName 表名，如果提供则根据表名选择数据库
     * @return array
     */
    public function getDbConfig(?string $tableName = null): array
    {
        // 如果是认证相关表，始终使用正式数据库
        if ($tableName && in_array($tableName, self::$authTables)) {
            return parent::getDbConfig(); // 返回正式数据库配置
        }
        
        // 如果定义了 SANDBOX 常量且为 true，使用沙盒数据库
        if (defined('SANDBOX') && SANDBOX) {
            $sandbox_db = $this->config['sandbox_db'] ?? [];
            if ($sandbox_db) {
                return $sandbox_db;
            }
        }
        
        // 默认使用正式数据库
        return parent::getDbConfig();
    }
}

// 2. 修改模型基类，传递表名给 getDbConfig
// 在 Weline\Framework\Database\AbstractModel 中修改

class AbstractModel
{
    public function getConnection()
    {
        // 获取表名
        $tableName = $this->getTable();
        
        // 获取数据库配置（根据表名选择）
        $env = ObjectManager::getInstance(Env::class);
        $dbConfig = $env->getDbConfig($tableName);
        
        // 创建或获取数据库连接
        $configProvider = new ConfigProvider($dbConfig['master']);
        return DbManager::getInstance()->create('sandbox_' . $tableName, $configProvider);
    }
}
```

**或者更简单的方案**：在 `ConfigProvider` 层面处理

```php
// 在 Weline\Framework\Database\DbManager\ConfigProvider 中修改

class ConfigProvider implements ConfigProviderInterface
{
    private ?string $tableName = null;
    
    public function setTableName(?string $tableName): self
    {
        $this->tableName = $tableName;
        return $this;
    }
    
    protected function getConfig(): mixed
    {
        // 认证相关表白名单
        $authTables = [
            'w_backend_user',
            'w_backend_user_token',
            'w_api_user',
            'w_frontend_user',
            'w_frontend_user_token',
        ];
        
        // 如果是认证相关表，始终使用正式数据库
        if ($this->tableName && in_array($this->tableName, $authTables)) {
            // 强制使用正式数据库配置
            $db_conf = Env::getInstance()->getConfig('db');
            if (empty($db_conf) || !isset($db_conf['master'])) {
                throw new Exception('数据库尚未配置');
            }
            return $db_conf;
        }
        
        // 如果定义了 SANDBOX 且为 true，使用沙盒数据库
        if (defined('SANDBOX') && SANDBOX) {
            $sandbox_db = Env::getInstance()->getConfig('sandbox_db') ?? [];
            if ($sandbox_db) {
                return $sandbox_db;
            }
        }
        
        // 默认逻辑（原有逻辑）
        $db_conf = Env::getInstance()->getDbConfig();
        if (empty($db_conf) || !isset($db_conf['master'])) {
            if ('cli' === PHP_SAPI) {
                throw new Exception('数据库尚未配置，请安装系统后操作:bin/w system:install');
            }
            throw new Exception('数据库尚未配置，请安装系统后操作:bin/w system:install');
        }
        return $db_conf;
    }
}
```

### 2.3 请求级别连接池

**问题**：如何确保每个请求使用独立的连接，不影响其他请求？

**解决方案**：使用请求级别的连接池

```php
class RequestConnectionPool
{
    private static array $connections = [];
    
    public static function getConnection(string $key, callable $factory): ConnectionFactory
    {
        // 使用请求ID作为key，确保每个请求独立
        $requestId = self::getRequestId();
        $poolKey = $requestId . '_' . $key;
        
        if (!isset(self::$connections[$poolKey])) {
            self::$connections[$poolKey] = $factory();
        }
        
        return self::$connections[$poolKey];
    }
    
    // 请求结束后清理连接
    public static function cleanup(string $requestId): void
    {
        foreach (self::$connections as $key => $conn) {
            if (strpos($key, $requestId) === 0) {
                unset(self::$connections[$key]);
            }
        }
    }
    
    private static function getRequestId(): string
    {
        return $_SERVER['REQUEST_ID'] ?? uniqid('req_', true);
    }
}
```

### 2.4 启用沙盒模式

**框架已支持**：通过 URL 参数或 Cookie 启用沙盒模式

```php
// 方式1：URL参数（需要配置 sandbox_key）
// http://example.com/api/v1/user/list?sandbox={sandbox_key}

// 方式2：Cookie（已设置 w_sandbox=1 后）
// Cookie: w_sandbox=1

// 方式3：在API文档界面自动启用
// 在API文档界面发起的请求，自动添加 ?sandbox={sandbox_key} 参数
```

**API文档界面自动启用**：

```javascript
// 在API文档界面发起请求时，自动添加沙盒参数
const sandboxKey = 'your_sandbox_key'; // 从配置获取
const url = new URL('/api/v1/user/list', window.location.origin);
url.searchParams.set('sandbox', sandboxKey);

fetch(url, {
    headers: {
        'Authorization': 'Bearer ' + token
    }
})
```

## 四、数据同步（可选）

### 4.1 基础数据同步

如果需要更完整的数据，可以实现基础数据同步：

```php
class SandboxDataSync
{
    // 同步基础数据到沙盒数据库
    public function syncBaseData(): void
    {
        $tables = [
            'w_system_config',  // 系统配置
            'w_module',         // 模块信息
            // 其他只读的基础数据表
        ];
        
        foreach ($tables as $table) {
            $this->syncTable($table);
        }
    }
    
    private function syncTable(string $tableName): void
    {
        // 从正式数据库读取
        $prodConn = $this->getProductionConnection();
        $data = $prodConn->query("SELECT * FROM {$tableName}")->fetchAll();
        
        // 写入沙盒数据库
        $sandboxConn = $this->getSandboxConnection();
        $sandboxConn->query("DELETE FROM {$tableName}");
        foreach ($data as $row) {
            $sandboxConn->insert($tableName, $row);
        }
    }
}
```

### 4.2 同步策略

- **手动同步**：管理员手动触发同步
- **定时同步**：每天凌晨同步一次基础数据
- **按需同步**：首次访问时自动同步

## 五、优势分析

### 5.1 方案优势

1. **保证认证可用**：
   - 用户可以在沙盒环境中正常登录
   - 令牌验证正常工作
   - 权限检查正常

2. **业务数据隔离**：
   - 业务数据完全独立
   - 可以随意测试和修改
   - 不影响正式数据

3. **实现简单**：
   - 不需要复杂的数据同步机制
   - 不需要维护两套完整的数据
   - 配置灵活

4. **性能良好**：
   - 认证操作使用正式数据库，性能不受影响
   - 业务操作使用沙盒数据库，不影响正式数据库性能

### 5.2 注意事项

1. **数据一致性**：
   - 沙盒数据库中的业务数据可能与正式环境不一致
   - 测试时需要考虑数据差异的影响

2. **权限测试**：
   - 权限数据在正式数据库，可以正常测试权限功能
   - 但需要注意权限变更的实时性

3. **配置管理**：
   - 系统配置如果在正式数据库，沙盒环境使用的是正式配置
   - 如果需要测试不同配置，需要单独处理

## 六、实施步骤

1. **第一步**：扩展 `Env::getDbConfig()` 或 `ConfigProvider::getConfig()`，支持表名参数
2. **第二步**：修改模型基类 `AbstractModel::getConnection()`，传递表名
3. **第三步**：配置认证表白名单（在代码或配置文件中）
4. **第四步**：在API文档界面实现自动添加沙盒参数
5. **第五步**：测试认证表使用正式数据库，业务表使用沙盒数据库
6. **第六步**：（可选）实现基础数据同步机制
