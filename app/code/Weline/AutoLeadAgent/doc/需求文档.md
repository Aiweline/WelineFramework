## 核心需求

### 1. WASM + JS 混合架构

#### 1.1 架构设计原则

**核心原则**：逻辑层完全使用 WASM，I/O 层（与浏览器交互）必须通过 JS 桥接。

**架构分工**：

| 模块 | 建议技术 | 原因 |
|------|---------|------|
| 智能体大脑 (Reasoning) | WASM | 处理状态机、规划任务步骤、决定调用哪个 MCP Tool |
| MCP 协议封装 | WASM | 将 JSON-RPC 消息与智能体状态进行映射和编解码 |
| 浏览器 API 访问 | JS (Glue Code) | WASM 无法直接调用 chrome.tabs 或 chrome.runtime，必须通过 JS 代理 |
| DOM 操作/数据抓取 | JS (Content Script) | JS 拥有对 DOM 的原生访问权，WASM 需通过 JS 指令获取数据 |
| 模型推理 | JS (Chrome AI / WebLLM) | 模型运行在浏览器内核层，WASM 通过 JS 接口发送 Prompt 并接收结果 |

#### 1.2 为什么使用 WASM 处理 MCP 逻辑

**优势**：

1. **计算性能**：智能体需要频繁处理 DOM 树分析、向量计算或复杂的 MCP 消息验证，WASM（通过 Rust 或 C++ 编译）比原生 JS 快得多
2. **跨语言生态**：可以复用现有的 Rust 版 MCP SDK 或 Python 编写的决策算法，通过 WASM 迁移到浏览器是最优路径
3. **代码保护**：智能体的核心提示词工程（Prompt Engineering）或私有逻辑，编译成 WASM 后比 JS 更难被逆向工程

#### 1.4 JS 桥接层实现

**Extension Service Worker**：

```javascript
// 在浏览器扩展中加载 WASM，并作为 MCP 客户端的控制器
import init, { AgentBrain } from './agent_wasm.js';

async function runAgent() {
    await init();
    const brain = new AgentBrain();
    
    // 1. 获取浏览器数据 (JS)
    const pageData = await chrome.tabs.captureVisibleTab(); 
    const domContent = await chrome.scripting.executeScript({
        target: { tabId: tab.id },
        func: () => document.body.innerText
    });
    
    // 2. 调用大脑决策 (WASM)
    const mcpInstruction = brain.decide_next_action(domContent);
    
    // 3. 执行 MCP 动作 (JS 调用浏览器 API)
    const instruction = JSON.parse(mcpInstruction);
    await executeMcpTool(instruction);
    
    // 4. 获取执行结果，反馈给 WASM
    const result = await getToolResult();
    brain.update_state(result);
}
```

#### 1.5 关键限制与解决方案

**限制 1：WASM 内存限制**

- **问题**：如果端侧模型（如 Gemini Nano）很大，WASM 无法直接管理其内存
- **解决方案**：模型运行在浏览器内核层，WASM 通过 JS 的 `window.ai` 接口发送 Prompt 并接收结果

**限制 2：扩展安全策略 (CSP)**

- **问题**：Chrome 扩展对 WASM 的执行有严格的 CSP 限制
- **解决方案**：在 `manifest.json` 中明确允许 `wasm-unsafe-eval`
  ```json
  {
    "content_security_policy": {
      "extension_pages": "script-src 'self' 'wasm-unsafe-eval'; object-src 'self'"
    }
  }
  ```

**限制 3：双向通信开销**

- **问题**：如果 WASM 与 JS 频繁交换大量 DOM 数据，性能会下降
- **解决方案**：使用 `SharedArrayBuffer` 在 JS 和 WASM 之间共享内存

### 2. 端侧 AI 模型集成

#### 1.1 模型选择策略

**优先级方案**：

1. **Chrome Built-in AI (Gemini Nano) - 推荐首选**
   - 2025年 Chrome 已全面内置 Gemini Nano
   - 通过 `window.ai.createTextSession()` 即可调用
   - 不占额外内存，无需下载模型文件
   - 延迟极低，秒开响应
   - 完全本地运行，隐私安全
   - **使用方式**：
     ```javascript
     const session = await window.ai.createTextSession();
     const response = await session.prompt(prompt);
     ```
   - **适用场景**：优先使用，如果浏览器不支持则降级到方案B

2. **WebLLM / @xenova/transformers - 降级方案**
   - **Qwen2.5-1.5B-Instruct** (Qwen/Qwen2.5-1.5B-Instruct): 1.5B 参数，**原生多语言支持**（中文、英文、日文、韩文等）
   - **Phi-3 Mini** (microsoft/Phi-3-mini-4k-instruct): 3.8B 参数，多语言支持良好
   - **TinyLlama-1.1B** (TinyLlama/TinyLlama-1.1B-Chat-v1.0): 1.1B 参数，快速推理
   - **使用方式**：通过 @xenova/transformers 加载，缓存到 IndexedDB

**模型选择策略**：

1. **优先检测 Chrome Built-in AI**：如果可用，直接使用（最快、最省资源）
2. **降级到 WebLLM**：如果 Chrome AI 不可用，使用 @xenova/transformers 加载模型
3. **确保模型支持**：
   - instruction following 和 JSON 输出
   - 多语言输入和输出
   - 能够理解任务指令并生成结构化决策

#### 1.3 模型使用场景（完全模型驱动）

所有推理功能必须通过模型调用实现，**完全移除基于规则的逻辑**：

1. **客户画像分析**: 模型分析店铺描述，提取客户特征（**移除基于规则的 extractProfileFeatures**）
2. **语义匹配**: 模型匹配网页内容与客户画像的相似度（**移除基于规则的 matchWebContentWithProfile**）
3. **关键词生成**: 模型从画像生成搜索关键词（**移除基于规则的 analyzeProfileForKeywords**）
4. **决策评分**: 模型综合多个维度给出匹配分数（**移除基于规则的 calculateMatchScore**）
5. **ReAct 决策**: 模型分析当前状态，决定下一步操作（搜索、提取、分析、完成）
6. **多语言处理**: 模型自动识别输入语言，使用对应语言进行推理和输出
7. **场景映射**: 模型从画像反推客户场景（**移除硬编码的场景映射规则**）
8. **搜索策略生成**: 模型生成搜索查询和策略（**移除基于规则的查询生成**）

### 3. Browser MCP 集成

#### 2.1 MCP 协议实现

**架构设计**：

- 浏览器扩展作为 MCP Client，负责解析模型指令并转换为浏览器 API 调用
- 通过 `chrome.runtime.sendMessage` 或 `postMessage` 与扩展通信
- 使用 MCP (Model Context Protocol) 标准协议
- 参考 [Browser MCP 文档](https://docs.browsermcp.io/) 实现协议细节

**MCP 工具集定义**：

在扩展的 `background.js` 或新建 `mcp-tools.js` 中定义工具集：

```javascript
const mcpTools = {
  "browser_navigate": async (args) => {
    // 导航到指定URL
  },
  "browser_snapshot": async (args) => {
    // 获取页面快照（DOM预处理：过滤script和style标签）
  },
  "browser_extract": async (args) => {
    // 提取客户信息（邮箱、电话、社媒等）
  },
  "browser_click": async (args) => {
    // 点击页面元素
  },
  "browser_type": async (args) => {
    // 输入文本
  },
  "browser_wait_for": async (args) => {
    // 等待元素出现
  },
  // 其他工具...
};
```

**支持的 MCP 工具**（根据 Browser MCP 文档）：

- `browser_navigate`: 导航到 URL
- `browser_click`: 点击页面元素
- `browser_type`: 输入文本
- `browser_fill_form`: 填写表单
- `browser_snapshot`: 获取页面快照（用于模型分析）
- `browser_extract`: 提取页面内容
- `browser_wait_for`: 等待元素出现
- `browser_select_option`: 选择下拉选项
- `browser_drag`: 拖放操作

### 4. ReAct Agent 实现

#### 3.1 ReAct 循环

**新文件**: `app/code/Weline/AutoLeadAgent/view/statics/js/react-agent.js`

**核心功能**：

- `think(currentState, profile, availableTools)`: 
  - 模型分析当前状态，生成下一步操作决策
  - 接收当前状态、客户画像和可用工具列表
  - 调用 `prompts.generateDecisionPrompt()` 生成包含工具描述的提示词
  - 调用模型生成决策（支持 Chrome AI 或 WebLLM）

- `act(decision)`: 
  - 根据模型决策调用 MCP 工具执行浏览器操作
  - 解析决策中的工具名称和参数
  - 通过 `mcpClient.callTool()` 执行工具调用
  - 返回工具执行结果

- `observe(toolResult)`: 
  - 获取 MCP 工具返回的结果
  - 更新当前状态
  - 准备反馈给模型的观察数据

- `reactLoop(taskId, profile)`: 
  - ReAct 主循环，持续执行 Think-Act-Observe 直到任务完成
  - 维护循环状态（当前步骤、已执行操作、结果等）
  - 设置最大循环次数，避免无限循环

**模型决策格式**：

```json
{
  "action": "mcp_tool_call",
  "tool": "browser_navigate|browser_click|browser_extract|browser_snapshot|...",
  "arguments": {
    "url": "...",
    "selector": "...",
    "text": "..."
  },
  "reason": "决策原因（支持多语言）",
  "language": "zh|en|ja|ko|...",
  "confidence": 0.0-1.0
}
```

#### 3.2 寻客逻辑保持不变

寻客的核心逻辑保持不变（搜索、提取、匹配、分页），只是实现方式从直接调用浏览器扩展改为通过 MCP 工具调用：

- **搜索引擎搜索**:
  - 模型决策：使用 `browser_navigate` 导航到搜索引擎
  - 模型决策：使用 `browser_type` 在搜索框输入关键词
  - 模型决策：使用 `browser_click` 点击搜索按钮
  - 模型决策：使用 `browser_wait_for` 等待搜索结果加载

- **提取搜索结果**:
  - 模型决策：使用 `browser_snapshot` 获取页面快照
  - 模型分析：模型分析快照内容，识别搜索结果项
  - 模型决策：使用 `browser_extract` 提取搜索结果 URL 和标题

- **访问目标网站**:
  - 模型决策：使用 `browser_navigate` 访问提取的 URL
  - 模型决策：使用 `browser_wait_for` 等待页面加载

- **提取客户信息**:
  - 模型决策：使用 `browser_snapshot` 获取页面内容
  - 模型分析：模型分析页面内容，提取邮箱、电话、社媒等信息
  - 模型决策：使用 `browser_extract` 提取结构化数据

- **匹配画像**:
  - 模型分析：模型分析提取的内容与客户画像的匹配度
  - 模型决策：决定是否保存该客户或继续搜索

### 5. 浏览器兼容性要求

#### 4.1 Chrome 浏览器必需

**要求**：

- 需要 Chrome Extension API 支持（`chrome.runtime`, `chrome.tabs`, `chrome.scripting`）
- 需要 Chrome Built-in AI 支持（`window.ai` API，可选但推荐）
- 非 Chrome 浏览器（Edge、Firefox、Safari、Opera 等）不支持

**检测逻辑**：

```javascript
function isChrome() {
  const userAgent = navigator.userAgent;
  // Chrome 浏览器：包含 'Chrome' 但不包含 'Edg'（Edge）、'OPR'（Opera）、'Safari'（Safari）
  return userAgent.includes('Chrome') && 
         !userAgent.includes('Edg') && 
         !userAgent.includes('OPR') && 
         !(userAgent.includes('Safari') && !userAgent.includes('Chrome'));
}
```

  2. 如果 Google 翻译失败（返回空字符串或抛出异常），降级使用模型翻译
  3. 调用 `generateTranslation()` 进行模型翻译
  4. 如果都失败，返回原始文本（不抛出异常，确保流程继续）

**缓存机制**：

- 翻译结果需要缓存，避免重复翻译相同内容
- Google 翻译的网络检测结果缓存 5 分钟

### 13. 注意事项

1. **浏览器兼容性检查**: 任务必须在 Chrome 浏览器内运行，在任务列表页面和任务启动时都要检测浏览器类型，非 Chrome 浏览器需显示弹窗提示

2. **模型安装检查**: 任务启动前必须检查模型是否已安装，未安装时阻止启动并提示用户

3. **Browser MCP 检查**: 任务启动前必须检查 Browser MCP 扩展是否已安装，未安装时阻止启动并提示用户安装

4. **完全移除规则逻辑**: 所有基于规则的推理方法必须完全删除，确保只使用模型

5. **模型文件管理**: 模型文件较大，需要合理管理 IndexedDB 存储空间

6. **加载性能**: 首次加载模型需要时间，需要显示加载进度

7. **计算资源**: 模型推理需要一定计算资源，考虑使用 Web Workers

8. **错误处理**: 需要处理模型加载失败的情况，提供降级方案

9. **ReAct 循环控制**: 需要设置最大循环次数，避免无限循环

10. **决策格式**: 模型返回的决策必须是有效的 JSON 格式，需要验证和错误处理

11. **翻译功能降级机制**: 翻译功能优先使用 Google 翻译 API（复用现有代码），如果 Google 翻译不可用（网络问题、API 限制、服务不可达），自动降级使用模型翻译

12. **MCP 工具调用**: 模型决策必须包含有效的 MCP 工具名称和参数，需要验证工具调用的有效性，处理工具调用失败的情况，工具调用结果需要反馈给模型进行下一步决策

13. **寻客逻辑保持**: 寻客的核心逻辑保持不变（搜索、提取、匹配、分页），只是实现方式从直接调用浏览器扩展改为通过 MCP 工具调用

14. **多语言支持**: 模型必须支持多语言输入和输出，提供多语言提示词模板（中文、英文、日文、韩文等），自动检测输入语言并选择对应语言处理，所有用户可见的输出支持多语言

15. **模型输出验证**: 模型返回的 JSON 必须经过严格验证，确保格式正确

16. **MCP 协议实现**: 需要正确实现 MCP (Model Context Protocol) 协议，浏览器扩展作为 MCP Client，负责解析模型指令并转换为浏览器 API 调用，通过 `chrome.runtime.sendMessage` 或 `postMessage` 与扩展通信，处理 MCP 连接断开和重连，参考 [Browser MCP 文档](https://docs.browsermcp.io/) 实现协议细节

17. **WASM + JS 混合架构**: 
    - 逻辑层（智能体大脑、MCP 协议处理）使用 WASM 实现，以获得更好的性能和代码保护
    - I/O 层（浏览器 API 访问、DOM 操作）必须通过 JS 桥接实现
    - WASM 通过 JS 接口与模型交互（`window.ai` 或 WebLLM）
    - 使用 `SharedArrayBuffer` 优化 WASM 与 JS 之间的数据交换
    - 在 `manifest.json` 中明确允许 `wasm-unsafe-eval` 以支持 WASM 执行
