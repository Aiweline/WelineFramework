## 1. 目标

把分销模块从“代码链路存在”推进到“运营可用、用户好理解、浏览器可验收”的分销服务。

最终体验应满足：

- 分销用户能进入分销中心，清楚看到自己的推荐码、推广链接、佣金状态和推广效果。
- 运营能在后台创建、筛选、查看、编辑和安全处置分销账户。
- 商品分享、点击归因、订单佣金、支付/取消/退款状态变化能形成可解释闭环。
- 前后台页面在桌面和移动端都不溢出、不遮挡、不出现未翻译英文或误导性字段。
- 每一轮验收必须有测试命令和浏览器截图证据。

## 2. 本轮验收结论

第一轮子智能体只读验收结论：不通过。

### 2.1 后端闭环结论

当前已有前台中心、商品分享、点击归因、订单佣金、后台管理等主体代码，但仍是半闭环：

- `/register?ref={code}` 链接生成后未被 Customer 注册流程消费，注册归因不成立。
- 佣金 `pending/approved/paid/cancelled/reversed` 语义和汇总指标不够清晰。
- 后台 `Delete::get()` 会调用 `post()`，GET 可触发删除。
- 硬删分销账户可能留下分享、触点、归因、佣金孤儿数据。
- 部分模型加载路径仍需统一实例生命周期。

### 2.2 前端体验结论

当前 UI 不是合格的分销运营工作台：

- 前台中心偏营销页，缺少强主操作和推广效果复盘。
- 推荐注册链接、商品推广链接的用途没有清晰区分。
- 佣金率后台输入显示 `0.10` 但带 `%` 后缀，存在运营误配风险。
- 中文默认体验有英文源文案和翻译缺口。
- 后台详情页复制功能依赖全局内联脚本，健壮性不足。

### 2.3 运行态验收结论

当前不能完成浏览器验收：

- 当前浏览器后端分销 URL 返回纯 `404`。
- 现有 `127.0.0.1:9502` 进程疑似裸 `php -S`，不是可验证的健康 WLS 实例。
- PostgreSQL `127.0.0.1:5432` 不通，阻塞 `setup:upgrade --route`、`runtime-info.php` 和标准 WLS 启动。
- 生成路由文件中能看到 Affiliate 路由，因此当前 404 更像运行环境/dispatcher/实例状态问题。

## 3. 验收红线

以下任一项存在时，不允许验收通过：

- `/affiliate`、`/affiliate/redirect` 或后台分销入口返回纯 `404`。
- 标准 WLS 实例无法启动，且没有等价运行态证据。
- GET 请求可以删除或修改分销数据。
- 佣金率输入、存储、展示语义不一致。
- 默认中文界面出现明显英文按钮、标题或业务提示。

## 5. 首轮开发派发策略

首轮只派发一个小范围 P0 任务：`AFF-P0-04`。

原因：

- 它是确定的安全缺陷。
- 写入范围小，不依赖当前不可用的 PostgreSQL/WLS。
- 可用单元测试先锁住 GET 不变更数据的底线。

验收方式：

- 代码复核 `Delete::get()` 不再调用 `post()`。
- 单元测试证明 GET 非破坏性、POST 有效 id 才执行删除路径。
- 运行 `php vendor\bin\phpunit --no-coverage app\code\WeShop\Affiliate\Test\Unit\Controller\Backend\Affiliate\DeleteTest.php`。
- 如测试依赖框架基类难以实例化，至少用反射/静态测试覆盖 GET 方法不委托 POST，并记录剩余风险。
