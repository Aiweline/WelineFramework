# PR 需求：PageBuilder AI 智能建站强契约与整站视觉验收

> 状态：PR 需求基准
> 范围：`app/code/GuoLaiRen/PageBuilder`
> 目标：让 PageBuilder 下的 AI 智能建站在强契约和底座提示词约束下，稳定生成审美过关、内容过关、真实可用的网站 block，并最终完成整站 block 生成。

## 1. 背景

当前要处理的问题不是单次生成结果不好看，也不是通过手动调某个方案得到一个好看的演示站点。真正目标是提升 PageBuilder AI 智能建站的底座能力：强契约、底座提示词、生成链路、渲染链路和质量反馈必须一起收敛，让 AI 在不同业务输入下都能产出精美、完整、可用的网站 block。

因此，本 PR 不能以“代码写完”“接口能跑”“方案能生成”“局部 block 偶然好看”作为完成标准。完成必须以真实浏览器截图验收为准。

## 2. 核心需求

1. AI 建站必须支持新建会话，并在会话内生成整站方案。
2. 方案生成后，必须支持指定某一个 block 单独生成。
3. 单个 block 生成结果必须能在真实页面中渲染，并通过 Codex in-app Browser 插件截图检查。
4. 至少选取 3 个不同类型的 block 进行指定生成与浏览器截图验证。
5. 3 个 block 都必须在审美、内容、布局、视觉层次、响应式表现和真实渲染上过关。
6. 单 block 质量过关后，必须继续尝试整站 block 生成。
7. 整站生成后，必须查看很多页面和多个 block 的真实浏览器截图，不允许只看一个页面、一个接口结果或一段源码。
8. 只有整站 block 生成可用，并且浏览器截图检查下来整体审美、内容、页面结构和多页面体验都过关，才算完成。

## 3. 强约束

### 3.1 禁止硬编码修结果

禁止为了通过某一次验证而硬编码特定行业、特定 block、特定文案、特定颜色、特定图片或特定方案结果。修复必须落在通用能力上，包括但不限于：

1. 强契约字段与校验规则。
2. 底座提示词和分层提示词。
3. block 生成上下文。
4. 视觉质量门禁。
5. 内容质量门禁。
6. 渲染数据结构。
7. 失败反馈与 AI 自修复链路。

### 3.2 禁止手调方案凑结果

禁止通过手动修改已生成方案、手动替换 block 内容、手动挑选特殊方案来得到好看的截图。方案必须由 AI 会话流程生成，指定 block 也必须基于该方案生成。

### 3.3 禁止用非浏览器验证替代视觉验收

对于 PageBuilder AI 建站这类浏览器可见功能，不能把单元测试、路由测试、源码检查、命令行 HTTP 检查或接口返回成功当作视觉验收。最终验收必须依赖 Codex in-app Browser 插件打开真实页面并截图检查。

如果 Browser 验证被运行时、WLS、端口、登录、证书或沙箱问题阻塞，PR 必须明确记录具体 blocker，不能声称前端效果已验证。

### 3.4 禁止未完成停下

没有达到“完成定义”前不能停止在半成品状态。不能以“基础功能已实现”“局部能力已打通”“后续继续优化”“当前先到这里”作为交付结论。

### 3.5 允许小步架构优化但禁止大改

调整过程中可以从框架层设计更合理的结构来提升 AI 建站质量，例如抽象通用契约、统一质量门禁输入、补齐失败恢复链路或整理可复用提示词片段。但架构调整必须小步、局部、可解释、可验证，禁止为了追求一次效果而进行大面积重构、跨模块重写或引入不可控行为。任何结构优化都必须服务于通用生成质量，不能变成硬编码结果、手动凑 block 内容或绕过真实 Browser 验收的替代路径。

## 4. 完成定义

本 PR 的完成定义是：

1. 可以新建 AI 建站会话。
2. 可以生成整站方案。
3. 可以从方案中指定单个 block 生成。
4. 至少 3 个不同 block 经过真实浏览器页面截图验证，审美和内容都过关。
5. 可以继续执行整站 block 生成。
6. 整站生成后的多个页面必须用 Browser 打开并截图检查。
7. 首页、核心转化页、内容/产品/服务页、关于或联系页、政策或支持类页面，以及共享 header/footer 在多页面中的表现都必须检查；如果方案页数少于这些类型，则检查方案中所有页面。
8. 多页面截图中不能出现明显的模板感、占位内容、内容空洞、视觉层次不足、响应式破版、语言混乱、图片缺失、按钮不可见、头尾部不一致或 block 之间风格断裂。
9. 整站 block 生成可用，截图看过并确认整体过关后，才算完成。

## 5. 验收流程

1. 新建 AI 建站会话。
2. 输入一个真实业务需求，生成整站方案。
3. 从方案里选择第一个 block，执行指定 block 生成。
4. 用 Browser 打开真实页面，截图检查。
5. 如果不过关，定位底层原因并修复强契约、底座提示词、生成逻辑、质量门禁或渲染链路。
6. 重复生成与截图验证，直到该 block 过关。
7. 对第二个、第三个不同类型 block 重复上述流程。
8. 3 个 block 过关后，执行整站 block 生成。
9. 用 Browser 查看整站多个页面和多个 block 的截图效果。
10. 如果整站不过关，继续定位底层能力问题并修复，再重新执行生成与截图验收。

## 6. 视觉与内容质量标准

### 6.1 审美标准

1. 页面必须有明确视觉层次：主标题、辅助文案、行动区、证明信息、视觉资产和留白关系清晰。
2. block 之间要有节奏变化，不能所有区块都像同一种卡片堆叠。
3. 配色必须服务行业和品牌气质，不能默认落入单一紫蓝、灰蓝、米色、棕橙等模板化色系。
4. CTA、导航、表单、卡片、图文关系要真实可用，不能只是装饰。
5. 视觉资产要能支撑内容，不允许只有空背景、抽象渐变或无意义占位。
6. 凡是看起来像按钮、Tab、pill、badge、步骤、轮播点、状态点、输入框、进度条、chip、指标或控件的元素，必须有可理解的文字、图标或可访问标签；禁止出现无标签圆点、空 pill、空横条、孤立轮播点、无图标步骤点等让用户误以为可操作但无法理解的结构。
7. Hero/Banner 默认应承担强首屏号召力；在有 hero cover 契约时，图片应作为全屏宽幅或 viewport-width 的主视觉背景层，而不是右侧小图、窄图岛或被卡片化的缩略图。
8. 强契约和底座提示词必须包含负向提示词覆盖：占位/线框/骨架感、模板重复、无意义装饰、假控件、虚假事实/指标/联系方式/安全/支付承诺、低对比、层级混乱、CTA 不明确、图片缺失/伪造、响应式重叠/裁切/横向滚动，都应在返回结果前被拒绝并重写。

### 6.2 内容标准

1. 内容必须贴合用户输入的业务，不允许泛泛而谈。
2. 每个 block 都要承担清晰的信息职责。
3. 标题、卖点、证明、CTA 和辅助说明之间必须能形成转化逻辑。
4. 不允许出现占位词、解释性 prompt 痕迹、AI 自我说明、乱码或中英混杂。
5. 多页面之间的信息不能互相矛盾。

### 6.3 响应式与真实渲染标准

1. 桌面端和移动端都不能出现文字溢出、按钮挤压、元素重叠或横向滚动。
2. 图片、背景、图标、视频或其他视觉资产必须实际渲染。
3. 共享 header/footer 在不同页面中要稳定一致。
4. 页面首屏必须有强识别度，不能只有导航或空白背景。
5. 真实浏览器中的效果优先于静态结构判断。

## 7. PR 交付要求

1. PR 描述必须说明本次改动触及的强契约、提示词、生成链路、渲染链路或质量门禁。
2. PR 描述必须列出 3 个单 block 指定生成的验证结果。
3. PR 描述必须列出整站生成验证覆盖了哪些页面。
4. PR 描述必须附上或说明 Browser 截图验证结论。
5. 如果某些页面无法验证，必须说明具体阻塞原因。
6. 在真实流程未通过前，不应新增或固化回归测试；视觉流程稳定后，再补测试锁定行为。
