# 智能体组件生成：真实组件学习与一次性生成计划

## 一、目标

- 智能体**先学会**真实组件如何创建，再按同一套规则生成。
- 提示词写全规则与流程，**尽可能一次性**写好组件，减少多轮试错与「我明白了再修正」。

---

## 二、真实组件如何创建（必学）

### 2.1 文件与目录规范

- 组件位于 `style/{template}/components/{region}/{code}.phtml`，如 `footer/links.phtml`。
- 组件代码（code）只能**小写字母、数字、连字符**，如 `footer-links`、`hero-slider`。
- 每个模板有 `component.json`，定义组件的 name、file、region、category、name_en、description 等。

### 2.2 组件文件结构（真实 phtml）

1. **文档块**：`@fields_start` / `@fields_end` 定义可配置字段，供后台表单展示。
   - 格式：`group:组名 => 组标题`，`key => 标签:类型:默认值|选项`。
   - 类型：text、textarea、number、color、select、image 等。
   - 配置读取时使用 `$component_config['group.key']` 或主题提供的 `$getConfig('group.key', '默认值')`。

2. **PHP 变量准备**：
   - 从 `$component_config` 或 `$getConfig()` 读取配置，赋给普通变量（如 `$brandName`、`$col1Title`）。
   - 链接等需解析时，写小型解析逻辑（如 `$parseLinks`）再得到数组。
   - 所有输出用 `htmlspecialchars()`，禁止裸插用户/配置内容。

3. **HTML + CSS**：
   - 根节点带唯一 id（如 `id="<?= $instanceId ?>"`），CSS 选择器均以该 id 开头做隔离。
   - 类名 BEM + 模板前缀，避免跨组件冲突。
   - 样式可内联 `<style>` 或外链，但选择器必须带组件 id。

4. **禁止项**：
   - 禁止 HTML/CSS/JS/PHP 注释（生产规约）。
   - 禁止内联样式写死颜色，使用主题变量或配置。
   - 禁止在 php_variables 中写 `<?php`/`?>`、if/foreach/function/class、echo/print。

### 2.3 与 AI 框架的对应关系

- **真实组件**：一个 phtml = 自己定义变量 + 自己写 HTML/CSS/JS。
- **AI 组件**：系统先注入**框架 PHP 块**（定义好 `$brandLogo`、`$col1Title`、`$getConfig` 等），再拼接你提供的 `html_content`、`css_content`、`css_responsive`、`js_content`；`php_variables` 仅用于**组件独有的**、框架未提供的变量。
- 因此：**凡框架已注入的变量，禁止在 php_variables 中重复声明**；只在 html_content 中直接使用即可。

## 三、各区域框架已提供内容（必知）

### 3.1 占位符与最终文件如何生成

- 系统保存时：**框架 PHP 块**（含 `{{PHP_VARIABLES}}` 替换为你的 php_variables） + **你的 html_content** + `<style>`（css_content + css_responsive） + `<script>`（js_content）。
- 框架 PHP 块内已定义：`$page`、`$getConfig`、`$componentId` 以及各区域变量（见下）。

### 3.2 Footer

- **已有变量**：`$brandLogo`, `$brandName`, `$brandDesc`, `$col1Title`, `$col1Items`, `$col2Title`, `$col2Items`, `$showSocial`, `$socialLinks`, `$copyrightText`, `$yearDisplay`, `$bgColor`, `$textColor`, `$titleColor`, `$linkColor`, `$linkHoverColor`, `$accentColor`，以及循环中的 `$item`, `$platform`, `$url`。
- 你的 **html_content**：整块 footer 主体 HTML（可直接用上述变量），不需要再声明。

### 3.3 Header

- **已有变量**：`$showLogo`, `$logoImage`, `$logoText`, `$showNav`, `$navItems`, `$showCta`, `$ctaText`, `$ctaUrl`，以及样式相关变量。
- 你的 **html_content**：整块 header 主体 HTML。

### 3.4 Content

- **已有变量**：`$title`, `$subtitle`, `$description`, `$containerWidth`, `$paddingTop`, `$paddingBottom`, `$textAlign`, `$bgStyle`, `$maxWidth`, `$cssPrefix` 等。
- 你的 **html_content**：放在内容区主体内的 HTML。

## 四、智能体重写方案（分步骤）

### 4.1 提示词结构（按顺序）

1. **角色与目标**：专业前端/全栈，一次生成符合规约的 PageBuilder 组件。
2. **真实组件创建方式**：简述文件结构、@fields、变量从配置读取、htmlspecialchars、选择器以 id 隔离。
3. **你的输出如何被使用**：框架 PHP 块 + 你的 html_content + css + js；框架已注入变量不得在 php_variables 中声明。
4. **框架已注入变量**：按当前 region 列出（从 FrameworkBuilder 动态注入）。
5. **必须遵守的规则**：命名、CSS（#componentId、BEM、主题变量）、HTML（语义化、htmlspecialchars、禁止注释）、PHP（仅简单赋值、无控制结构）、JS（用 component、不用 $componentId）、extra_fields 格式。
6. **一次性生成流程**：
   - 第一步：必调 `get_component_framework(category)` 获取当前区域框架与变量列表。
   - 可选：`list_components` 看同区域已有组件风格。
   - 第二步：直接输出**完整** JSON（name、description、html_content、css_content、css_responsive、js_content、php_variables、extra_fields），不再分步「先写 HTML 再写 CSS」。
   - 第三步：仅当 `validate_code` 报错时，根据错误修正后再次输出 JSON；通过则立即输出最终 JSON，不再调用其他工具。
7. **最终输出格式**：整条消息有且仅有一个合法 JSON 对象，禁止 \`\`\`json、禁止前后说明。
8. **工具说明**：list_components、preview_reference_component、get_component_framework、validate_code、get_page_layout 及使用时机。

### 4.2 实施步骤

| 步骤 | 内容 |
|------|------|
| 1 | 在 doc 下新增本文档（计划与真实组件学习要点）。 |
| 2 | 重写 PageBuilderAgent::getSystemPrompt()：按 4.1 顺序写全「真实组件创建 → 你的输出如何被使用 → 框架变量 → 规则 → 一次性流程 → 输出格式 → 工具」。 |
| 3 | 保持 ValidateCodeTool、GetComponentFrameworkTool、FrameworkBuilder 的框架变量列表与提示词一致。 |
| 4 | 可选：在 GetComponentFrameworkTool 返回中增加「建议一次性生成流程」简短说明，供模型参考。 |

## 五、验收预期

- 智能体在**第一轮或第二轮**即输出完整、合规的 JSON，validate_code 一次通过。
- 用户侧不再出现「我明白了…让我修正…」的多轮空转；仅在校验失败时做**一次**针对性修正。
- 生成的 phtml 渲染无 Undefined variable，样式隔离、主题变量、htmlspecialchars 均符合真实组件规约。
