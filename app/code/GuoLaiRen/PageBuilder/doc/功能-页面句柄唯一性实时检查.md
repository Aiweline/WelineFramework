# 功能：页面句柄唯一性实时检查

## 📋 功能概述

在页面创建/编辑表单中，为"页面句柄"（handle）字段添加实时唯一性检查功能。用户输入 handle 后，系统自动检查是否重复，并在界面上显示验证结果，防止提交重复的 handle。

## ✨ 核心特性

### 1. **实时检查（Debounce）**
- 用户输入时延迟 500ms 后自动检查
- 避免频繁的 API 请求
- 使用 debounce 技术优化性能

### 2. **格式自动修正**
- 自动转换为小写字母
- 自动过滤非法字符（只允许 `a-z`、`0-9`、`-`）
- 实时格式化，用户体验友好

### 3. **视觉反馈**
- ⏳ **检查中**：显示旋转图标和灰色状态
- ✅ **可用**：显示绿色对勾和成功消息
- ❌ **不可用**：显示红色叉号和错误消息，标红输入框

### 4. **智能编辑模式**
- 编辑模式下，保持原 handle 不变时自动标记为可用
- 无需发送请求，直接验证通过
- 修改 handle 时才会触发实时检查

### 5. **提交阻止**
- 检测到重复 handle 时，阻止表单提交
- 显示 alert 提示用户
- 自动聚焦到 handle 字段

## 🎯 用户场景

### 场景 1：创建新页面
1. 用户输入 handle：`about-us`
2. 系统延迟 500ms 后检查
3. 如果可用：显示绿色对勾 ✅ "页面句柄可用"
4. 如果重复：显示红色叉号 ❌ "页面句柄已被使用"
5. 提交前再次验证，重复时阻止提交

### 场景 2：编辑现有页面（不修改 handle）
1. 页面加载，handle 字段值为 `about-us`
2. 用户修改其他字段，不改 handle
3. 系统识别 handle 未改变
4. **直接标记为可用，无需请求** ✅
5. 允许提交

### 场景 3：编辑现有页面（修改 handle）
1. 页面加载，原 handle 为 `about-us`
2. 用户修改为 `contact-us`
3. 系统检测到 handle 改变，触发检查
4. 如果 `contact-us` 被其他页面使用：标红 ❌
5. 如果可用：显示绿色对勾 ✅

### 场景 4：格式自动修正
1. 用户输入：`About Us!`
2. 自动转换为：`about-us`（移除空格和特殊字符）
3. 然后触发唯一性检查

## 🔧 技术实现

### 后端接口

**路由**：`*/backend/page/checkHandle`

**方法**：`getCheckHandle()`

**参数**：
- `handle` - 要检查的句柄（必需）
- `page_id` - 当前编辑的页面ID（编辑模式，可选）

**返回格式**：
```json
{
    "success": true,
    "available": true,
    "message": "页面句柄可用"
}
```

或

```json
{
    "success": true,
    "available": false,
    "message": "页面句柄已被使用，请使用其他句柄",
    "existing_page": {
        "id": 123,
        "name": "关于我们",
        "title": "About Us"
    }
}
```

### 前端实现

#### HTML 结构

```html
<div class="position-relative">
    <input type="text" 
           id="handle" 
           name="handle" 
           data-original-value="原始值">
    <div id="handle-validation-icon" class="position-absolute top-50 end-0">
        <span class="handle-checking">⏳</span>
        <span class="handle-available">✅</span>
        <span class="handle-unavailable">❌</span>
    </div>
</div>
<div id="handle-validation-message"></div>
```

#### JavaScript 逻辑

1. **Debounce 函数**：延迟执行检查
2. **格式化监听**：实时转小写和过滤非法字符
3. **唯一性检查**：
   - 检查是否为空 → 隐藏验证
   - 编辑模式 + handle 未改变 → 直接标记可用
   - 否则 → 发送 AJAX 请求检查
4. **提交拦截**：验证不通过时阻止提交

### 验证规则

#### 格式验证（前端 + 后端）
- 只允许小写字母 `a-z`
- 只允许数字 `0-9`
- 只允许连字符 `-`
- 正则：`/^[a-z0-9\-]+$/`

#### 唯一性验证（后端）
- 查询数据库：`WHERE handle = ?`
- 排除当前编辑的页面：`AND id != ?`

## 📊 状态流转

```
用户输入 → 格式化 → Debounce(500ms) → 检查逻辑
                                           ↓
                        ┌──────────────────┴──────────────────┐
                        ↓                                      ↓
                    空值？                              是编辑模式且未改变？
                    Yes ↓                                  Yes ↓
                  隐藏验证                              直接标记可用 ✅
                    No ↓                                  No ↓
                发送 AJAX 请求 → 后端检查 → 返回结果
                                              ↓
                        ┌─────────────────────┴─────────────────────┐
                        ↓                                            ↓
                   available=true                              available=false
                        ↓                                            ↓
                  显示可用 ✅                                  显示不可用 ❌
                    绿色边框                                      红色边框
                    成功消息                                      错误消息
                                                              + 占用页面信息
```

## 🐛 常见问题

### Q1：编辑页面时，保持原 handle 提示"已被使用"
**原因**：后端没有正确排除当前页面
**解决**：检查 `page_id` 参数是否正确传递

### Q2：输入大写字母被转换为小写
**说明**：这是预期行为，handle 只允许小写字母
**提示**：在 placeholder 中说明格式要求

### Q3：提交时没有阻止重复 handle
**原因**：`isHandleValid` 状态未正确更新
**解决**：确保 `lastCheckedHandle === currentHandle` 条件匹配

### Q4：检查请求发送过于频繁
**原因**：debounce 延迟设置过短
**调整**：修改延迟时间（当前 500ms）

## 🎓 开发经验总结

### 1. 双重验证策略
- 前端：实时检查，提升用户体验
- 后端：最终验证，保证数据安全

### 2. 编辑模式优化
- 识别未改变的值，减少不必要的请求
- 提升性能，降低服务器压力

### 3. 防抖（Debounce）的重要性
- 避免用户每输入一个字符就发送请求
- 500ms 是体验和性能的平衡点

### 4. 状态同步
- `lastCheckedHandle`：记录最后检查的值
- `isHandleValid`：记录验证结果
- 提交前对比状态，确保一致性
